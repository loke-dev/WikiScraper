class="mw-editsection-bracket">[</span><a href="/w/index.php?title=PostScript&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>The concepts of the PostScript language were seeded in 1976 when <a href="/wiki/John_Warnock" title="John Warnock">John Warnock</a> was working at <a href="/wiki/Evans_%26_Sutherland" title="Evans &amp; Sutherland">Evans &amp; Sutherland</a>, a <a href="/wiki/Computer_graphics" title="Computer graphics">computer graphics</a> company. At that time John Warnock was developing an interpreter for a large three-dimensional graphics database of <a href="/wiki/New_York_City" title="New York City">New York</a> harbor. Warnock conceived the Design System language to process the graphics.</p>/n<p>Concurrently, researchers at <a href="/wiki/Xerox_PARC" class="mw-redirect" title="Xerox PARC">Xerox PARC</a> had developed the first <a href="/wiki/Laser_printer" class="mw-redirect" title="Laser printer">laser printer</a> and had recognized the need for a standard means of defining page images. In 1975-76 <a href="/wiki/Bob_Sproull" title="Bob Sproull">Bob Sproull</a> and <a href="/wiki/William_Newman_(computer_scientist)" title="William Newman (computer scientist)">William Newman</a> developed the Press format, which was eventually used in the <a href="/wiki/Xerox_Star" title="Xerox Star">Xerox Star</a> system to drive laser printers. But Press, a data format rather than a language, lacked flexibility, and PARC mounted the <a href="/wiki/Interpress" title="Interpress">Interpress</a> effort to create a successor.</p>/n<p>In 1978 Warnock left Evans &amp; Sutherland and joined Xerox PARC to work with <a href="/wiki/Martin_Newell_(computer_graphics)" class="mw-redirect" title="Martin Newell (computer graphics)">Martin Newell</a>. They rewrote Design System to create the interpretive language, J &amp; M or JaM<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">[1]</a></sup> (for "John and Martin") which was used for <a href="/wiki/Very-large-scale_integration" title="Very-large-scale integration">VLSI</a> design and the investigation of type and graphics printing. This work later evolved and expanded into the <a href="/wiki/Interpress" title="Interpress">Interpress</a> language.</p>/n<p>Warnock left with <a href="/wiki/Chuck_Geschke" class="mw-redirect" title="Chuck Geschke">Chuck Geschke</a> and founded <a href="/wiki/Adobe_Systems" title="Adobe Systems">Adobe Systems</a> in December 1982. They, together with Doug Brotz, Ed Taft and Bill Paxton created a simpler language, similar to Interpress, called PostScript, which went on the market in 1984. At about this time they were visited by <a href="/wiki/Steve_Jobs" title="Steve Jobs">Steve Jobs</a>, who urged them to adapt PostScript to be used as the language for driving laser printers.</p>/n<p>In March 1985, the <a href="/wiki/Apple_Computer" class="mw-redirect" title="Apple Computer">Apple</a> <a href="/wiki/LaserWriter" title="LaserWriter">LaserWriter</a> was the first printer to ship with PostScript, sparking the <a href="/wiki/Desktop_publishing" title="Desktop publishing">desktop publishing</a> (DTP) revolution in the mid-1980s. The combination of technical merits and widespread availability made PostScript a language of choice for graphical output for printing applications. For a time an <a href="/wiki/Interpreter_(computer_software)" class="mw-redirect" title="Interpreter (computer software)">interpreter</a> (sometimes referred to as a <a href="/wiki/Raster_Image_Processor" class="mw-redirect" title="Raster Image Processor">RIP</a> for Raster Image Processor) for the PostScript language was a common component of laser printers, into the 1990s.</p>/n<p>However, the cost of implementation was high; computers output raw PS code that would be interpreted by the printer into a raster image at the printer's natural resolution. This required high performance <a href="/wiki/Microprocessor" title="Microprocessor">microprocessors</a> and ample <a href="/wiki/Computer_memory" title="Computer memory">memory</a>. The LaserWriter used a 12&#160;MHz <a href="/wiki/Motorola_68000" title="Motorola 68000">Motorola 68000</a>, making it faster than any of the Macintosh computers to which it attached. When the laser printer engines themselves cost over a thousand dollars the added cost of PS was marginal. But as printer mechanisms fell in price, the cost of implementing PS became too great a fraction of overall printer cost; in addition, with desktop computers becoming more powerful, it no longer made sense to offload the rasterisation work onto the resource-constrained printer. By 2001, few lower-end printer models came with support for PostScript, largely due to growing competition from much cheaper non-PostScript ink jet printers, and new software-based methods to render PostScript images on the computer, making them suitable for any printer; <a href="/wiki/Portable_Document_Format" title="Portable Document Format">PDF</a>, a descendant of PostScript, provides one such method, and has largely replaced PostScript as <i><a href="/wiki/De_facto" title="De facto">de facto</a></i> standard for electronic document distribution.</p>/n<p>On high-end printers, PostScript processors remain common, and their use can dramatically reduce the CPU work involved in printing documents, transferring the work of rendering PostScript images from the computer to the printer.</p>/n<h3><span class="mw-headline" id="PostScript_Level_1">PostScript Level 1</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=PostScript&amp;action=edit&amp;section=2" title="Edit section: PostScript Level 1">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>The first version of the PostScript language was released to the market in 1984. The term "Level 1" was added when Level 2 was introduced.</p>/n<h3><span class="mw-headline" id="PostScript_Level_2">PostScript Level 2</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=PostScript&amp;action=edit&amp;section=3" title="Edit section: PostScript Level 2">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>PostScript Level 2 was introduced in 1991, and included several improvements: improved speed and reliability, support for in-RIP separations, <a href="/wiki/Image_compression" title="Image compression">image decompression</a> (for example, <a href="/wiki/JPEG" title="JPEG">JPEG</a> images could be rendered by a PostScript program), support for composite <a href="/wiki/Fonts" class="mw-redirect" title="Fonts">fonts</a>, and the form mechanism for caching reusable content.</p>/n<h3><span class="mw-headline" id="PostScript_3">PostScript 3</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=PostScript&amp;action=edit&amp;section=4" title="Edit section: PostScript 3">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>PostScript 3 (Adobe dropped the "level" terminology in favor of simple versioning) came at the end of 1997, and along with many new dictionary-based versions of older operators, introduced better color handling and new filters (which allow in-program compression/decompression, program chunking, and advanced error-handling).</p>/n<p>PostScript 3 was significant in terms of replacing the existing proprietary color electronic prepress systems, then widely used for magazine production, through the introduction of smooth shading operations with up to 4096 shades of grey (rather than the 256 available in PostScript Level 2), as well as DeviceN, a <a href="/wiki/Color_space" title="Color space">color space</a> that allowed the addition of additional ink colors (called <a href="/wiki/Spot_color" title="Spot color">spot colors</a>) into composite color pages.</p>/n<h2><span class="mw-headline" id="Use_in_printing">Use in printing</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=PostScript&amp;action=edit&amp;section=5" title="Edit section: Use in printing">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<h3><span class="mw-headline" id="Before_PostScript">Before PostScript</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=PostScript&amp;action=edit&amp;section=6" title="Edit section: Before PostScript">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>Prior to the introduction of PostScript, printers were designed to print character output given the text—typically in <a href="/wiki/ASCII" title="ASCII">ASCII</a>—as input. There were a number of technologies for this task, but most shared the property that the <a href="/wiki/Glyph" title="Glyph">glyphs</a> were physically difficult to change, as they were stamped onto <a href="/wiki/Typewriter" title="Typewriter">typewriter</a> keys, bands of metal, or optical plates.</p>/n<p>This changed to some degree with the increasing popularity of <a href="/wiki/Dot_matrix_printer" class="mw-redirect" title="Dot matrix printer">dot matrix printers</a>. The characters on these systems were drawn as a series of dots, as defined by a <a href="/wiki/Typeface" title="Typeface">font</a> table inside the printer. As they grew in sophistication, dot matrix printers started including several built-in fonts from which the user could select, and some models allowed users to upload their own custom glyphs into the printer.</p>/n<p>Dot matrix printers also introduced the ability to print <a href="/wiki/Raster_graphics" title="Raster graphics">raster graphics</a>. The graphics were interpreted by the computer and sent as a series of dots to the printer using a series of <a href="/wiki/Escape_sequence" title="Escape sequence">escape sequences</a>. These <a href="/wiki/Printer_control_language" class="mw-redirect" title="Printer control language">printer control languages</a> varied from printer to printer, requiring program authors to create numerous <a href="/wiki/Device_driver" title="Device driver">drivers</a>.</p>/n<p>Vector graphics printing was left to special-purpose devices, called <a href="/wiki/Plotter" title="Plotter">plotters</a>. Almost all plotters did share a common command language, <a href="/wiki/HPGL" class="mw-redirect" title="HPGL">HPGL</a>, but were of limited use for anything other than printing graphics. In addition, they tended to be expensive and slow, and thus rare.</p>/n<h3><span class="mw-headline" id="PostScript_printing">PostScript printing</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=PostScript&amp;action=edit&amp;section=7" title="Edit section: PostScript printing">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>Laser printers combine the best features of both printers and plotters. Like plotters, laser printers offer high quality line art, and like dot-matrix printers, they are able to generate pages of text and raster graphics. Unlike either printers or plotters, however, a laser printer makes it possible to position high-quality graphics and text on the same page. PostScript made it possible to fully exploit these characteristics, by offering a single control language that could be used on any brand of printer.</p>/n<p>PostScript went beyond the typical printer control language and was a complete programming language of its own. Many applications can transform a document into a PostScript program whose execution will result in the original document. This program can be sent to an <a href="/wiki/Interpreter_(computer_software)" class="mw-redirect" title="Interpreter (computer software)">interpreter</a> in a printer, which results in a printed document, or to one inside another application, which will display the document on-screen. Since the document-program is the same regardless of its destination, it is called <i>device-independent</i>.</p>/n<p>PostScript is noteworthy for implementing on-the fly <a href="/wiki/Rasterization" class="mw-redirect" title="Rasterization">rasterization</a>; everything, even text, is specified in terms of straight lines and cubic <a href="/wiki/B%C3%A9zier_curve" title="Bézier curve">Bézier curves</a> (previously found only in <a href="/wiki/Computer-aided_design" title="Computer-aided design">CAD</a> applications), which allows arbitrary scaling, rotating and other transformations. When the PostScript program is interpreted, the interpreter converts these instructions into the dots needed to form the output. For this reason PostScript interpreters are occasionally called PostScript <a href="/wiki/Raster_Image_Processor" class="mw-redirect" title="Raster Image Processor">raster image processors</a>, or RIPs.</p>/n<h3><span class="mw-headline" id="Font_handling">Font handling</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=PostScript&amp;action=edit&amp;section=8" title="Edit section: Font handling">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/PostScript_fonts" title="PostScript fonts">PostScript fonts</a></div>/n<p>Almost as complex as PostScript itself is its handling of <a href="/wiki/Typeface" title="Typeface">fonts</a>. The font system uses the PS graphics primitives to draw glyphs as curves, which can then be rendered at any <a href="/wiki/Image_resolution" title="Image resolution">resolution</a>. A number of <a href="/wiki/Typographic" class="mw-redirect" title="Typographic">typographic</a> issues had to be considered with this approach.</p>/n<p>One issue is that fonts do not actually scale linearly at small sizes; features of the glyphs will become proportionally too large or small and they start to look wrong. PostScript avoided this problem with the inclusion of <a href="/wiki/Font_hinting" title="Font hinting">font hinting</a>, in which additional information is provided in horizontal or vertical bands to help identify the features in each letter that are important for the rasterizer to maintain. The result was significantly better-looking fonts even at low resolution; it had formerly been believed that hand-tuned bitmap fonts were required for this task.</p>/n<p>At the time, the technology for including these hints in fonts was carefully guarded, and the hinted fonts were compressed and encrypted into what Adobe called a <i><a href="/wiki/Type_1_Font" class="mw-redirect" title="Type 1 Font">Type 1 Font</a></i> (also known as <i>PostScript Type 1 Font</i>, <i>PS1</i>, <i>T1</i> or <i>Adobe Type 1</i>). Type 1 was effectively a simplification of the PS system to store outline information only, as opposed to being a complete language (PDF is similar in this regard). Adobe would then sell licenses to the Type 1 technology to those wanting to add hints to their own fonts. Those who did not license the technology were left with the <i><a href="/wiki/Type_3_Font" class="mw-redirect" title="Type 3 Font">Type 3 Font</a></i> (also known as <i>PostScript Type 3 Font</i>, <i>PS3</i> or <i>T3</i>). Type 3 fonts allowed for all the sophistication of the PostScript language, but without the standardized approach to hinting.</p>/n<p>The <a href="/wiki/Type_2_font" class="mw-redirect" title="Type 2 font">Type 2 font</a> format was designed to be used with <a href="/wiki/Compact_Font_Format" class="mw-redirect" title="Compact Font Format">Compact Font Format</a> (CFF) charstrings, and was implemented to reduce the overall font file size. The <a href="/wiki/CFF/Type_2_font" class="mw-redirect" title="CFF/Type 2 font">CFF/Type2</a> format later became the basis for handling PostScript outlines in <a href="/wiki/OpenType" title="OpenType">OpenType</a> fonts.</p>/n<p>The <a href="/wiki/CID-keyed_font" class="mw-redirect" title="CID-keyed font">CID-keyed font</a> format was also designed, to solve the problems in the <a href="/w/index.php?title=OCF/Type_0_font&amp;action=edit&amp;redlink=1" class="new" title="OCF/Type 0 font (page does not exist)">OCF/Type 0 fonts</a>, for addressing the complex Asian-language (<a href="/wiki/CJK" class="mw-redirect" title="CJK">CJK</a>) encoding and very large character set issues. The CID-keyed font format can be used with the Type 1 font format for standard CID-keyed fonts, or Type 2 for CID-keyed OpenType fonts.</p>/n<p>To compete with Adobe's system, Apple designed their own system, <a href="/wiki/TrueType" title="TrueType">TrueType</a>, around 1991. Immediately following the announcement of TrueType, Adobe published the specification for the Type 1 font format. Retail tools such as Altsys <a href="/wiki/Fontographer" title="Fontographer">Fontographer</a> (acquired by <a href="/wiki/Macromedia" title="Macromedia">Macromedia</a> in January 1995, owned by <a href="/wiki/FontLab" title="FontLab">FontLab</a> since May 2005) added the ability to create Type 1 fonts. Since then, many free Type 1 fonts have been released; for instance, the fonts used with the <a href="/wiki/TeX" title="TeX">TeX</a> typesetting system are available in this format.</p>/n<p>In the early 1990s there were several other systems for storing outline-based fonts, developed by <a href="/wiki/Bitstream_Inc." title="Bitstream Inc.">Bitstream</a> and <a href="/wiki/METAFONT" class="mw-redirect" title="METAFONT">METAFONT</a> for instance, but none included a general-purpose printing solution and they were therefore not widely used.</p>/n<p>In the late 1990s, Adobe joined Microsoft in developing <a href="/wiki/OpenType" title="OpenType">OpenType</a>, essentially a functional superset of the Type 1 and TrueType formats. When printed to a PostScript output device, the unneeded parts of the OpenType font are omitted, and what is sent to the device by the driver is the same as it would be for a TrueType or Type 1 font, depending on which kind of outlines were present in the OpenType font.</p>/n<h3><span class="mw-headline" id="Other_implementations">Other implementations</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=PostScript&amp;action=edit&amp;section=9" title="Edit section: Other implementations">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>In the 1980s, Adobe drew most of its revenue from the licensing fees for their implementation of PostScript for printers, known as a <a href="/wiki/Raster_image_processor" title="Raster image processor">raster image processor</a> or <i>RIP</i>. As a number of new <a href="/wiki/RISC" class="mw-redirect" title="RISC">RISC</a>-based platforms became available in the mid-1980s, some found Adobe's support of the new machines to be lacking.</p>/n<p>This and issues of cost led to third-party implementations of PostScript becoming common, particularly in low-cost printers (where the licensing fee was the sticking point) or in high-end typesetting equipment (where the quest for speed demanded support for new platforms faster than Adobe could provide). At one point, Microsoft licensed to Apple a PostScript-compatible interpreter it had bought called <a href="/wiki/TrueImage" title="TrueImage">TrueImage</a>, and Apple licensing to Microsoft its new font format, <a href="/wiki/TrueType" title="TrueType">TrueType</a>. Apple ended up reaching an accord with Adobe and licensed genuine PostScript for its printers, but TrueType became the standard <a href="/wiki/Outline_font" class="mw-redirect" title="Outline font">outline font</a> technology for both Windows and the Macintosh.</p>/n<p>Today, third-party PostScript-compatible interpreters are widely used in printers and multifunction peripherals (MFPs). For example, <a href="/wiki/CSR_plc" class="mw-redirect" title="CSR plc">CSR plc</a>'s IPS PS3<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">[2]</a></sup> interpreter, formerly known as PhoenixPage, is standard in many printers and MFPs, including those developed by <a href="/wiki/Hewlett-Packard" title="Hewlett-Packard">Hewlett-Packard</a> and sold under the <a href="/wiki/LaserJet" class="mw-redirect" title="LaserJet">LaserJet</a> and Color LaserJet lines. Other third-party PostScript solutions used by print and MFP manufacturers include Jaws<sup id="cite_ref-3" class="reference"><a href="#cite_note-3">[3]</a></sup> and the <a href="/wiki/Harlequin_RIP" title="Harlequin RIP">Harlequin RIP</a>, both by <a href="/wiki/Global_Graphics" title="Global Graphics">Global Graphics</a>. A <a href="/wiki/Free_software" title="Free software">free software</a> version, with several other applications, is <a href="/wiki/Ghostscript" title="Ghostscript">Ghostscript</a>. Several compatible interpreters are listed on the Undocumented Printing Wiki.<sup id="cite_ref-4" class="reference"><a href="#cite_note-4">[4]</a></sup></p>/n<p>Some basic, inexpensive laser printers do not support PostScript, instead coming with drivers that simply rasterize the platform's native graphics formats rather than converting them to PostScript first.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (July 2015)">citation needed</span></a></i>]</sup> When PostScript support is needed for such a printer, Ghostscript can be used. There are also a number of commercial PostScript interpreters, such as <a href="/wiki/TeleType_Co." title="TeleType Co.">TeleType Co.</a>'s T-Script.</p>/n<h2><span class="mw-headline" id="Use_as_a_display_system">Use as a display system</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=PostScript&amp;action=edit&amp;section=10" title="Edit section: Use as a display system">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<div role="note" class="hatnote navigation-not-searchable">Main articles: <a href="/wiki/Display_PostScript" title="Display PostScript">Display PostScript</a> and <a href="/wiki/NeWS" title="NeWS">NeWS</a></div>/n<p>PostScript became commercially successful due to the introduction of the <a href="/wiki/Graphical_user_interface" title="Graphical user interface">graphical user interface</a>, allowing designers to directly lay out pages for eventual output on laser printers. However, the GUI's own graphics systems were generally much less sophisticated than PostScript; Apple's <a href="/wiki/QuickDraw" title="QuickDraw">QuickDraw</a>, for instance, supported only basic lines and arcs, not the complex <a href="/wiki/B-spline" title="B-spline">B-splines</a> and advanced region filling options of PostScript. In order to take full advantage of PostScript printing, applications on the computers had to re-implement those features using the host platform's own graphics system. This led to numerous issues where the on-screen layout would not exactly match the printed output, due to differences in the implementation of these features.</p>/n<p>As computer power grew, it became possible to host the PS system in the computer rather than the printer. This led to the natural evolution of PS from a printing system to one that could also be used as the host's own graphics language. There were numerous advantages to this approach; not only did it help eliminate the possibility of different output on screen and printer, but it also provided a powerful graphics system for the computer, and allowed the printers to be "dumb" at a time when the cost of the laser engines was falling. In a production setting, using PostScript as a display system meant that the host computer could render low-resolution to the screen, higher resolution to the printer, or simply send the PS code to a smart printer for offboard printing.</p>/n<p>However, PostScript was written with printing in mind, and had numerous features that made it unsuitable for direct use in an interactive display system. In particular, PS was based on the idea of collecting up PS commands until the <code>showpage</code> command was seen, at which point all of the commands read up to that point were interpreted and output. In an interactive system this was clearly not appropriate. Nor did PS have any sort of interactivity built in; for example, supporting hit detection for mouse interactivity obviously did not apply when PS was being used on a printer.</p>/n<p>When <a href="/wiki/Steve_Jobs" title="Steve Jobs">Steve Jobs</a> left Apple and started <a href="/wiki/NeXT" title="NeXT">NeXT</a>, he pitched Adobe on the idea of using PS as the display system for his new workstation computers. The result was <a href="/wiki/Display_PostScript" title="Display PostScript">Display PostScript</a>, or DPS. DPS added basic functionality to improve performance by changing many string lookups into 32 bit integers, adding support for direct output with every command, and adding functions to allow the GUI to inspect the diagram. Additionally, a set of "bindings" was provided to allow PS code to be called directly from the <a href="/wiki/C_programming_language" class="mw-redirect" title="C programming language">C programming language</a>. NeXT used these bindings in their <a href="/wiki/NeXTStep" class="mw-redirect" title="NeXTStep">NeXTStep</a> system to provide an <a href="/wiki/Object_oriented" class="mw-redirect" title="Object oriented">object oriented</a> graphics system. Although DPS was written in conjunction with NeXT, Adobe sold it commercially and it was a common feature of most <a href="/wiki/Unix_workstation" class="mw-redirect" title="Unix workstation">Unix workstations</a> in the 1990s.</p>/n<p><a href="/wiki/Sun_Microsystems" title="Sun Microsystems">Sun Microsystems</a> took another approach, creating <a href="/wiki/NeWS" title="NeWS">NeWS</a>. Instead of DPS's concept of allowing PS to interact with C programs, NeWS instead extended PS into a language suitable for running the entire GUI of a computer. Sun added a number of new commands for timers, mouse control, interrupts and other systems needed for interactivity, and added <a href="/wiki/Data_structure" title="Data structure">data structures</a> and language elements to allow it to be completely object oriented internally. A complete GUI, three in fact, were written in NeWS and provided for a time on their workstations. However, the ongoing efforts to standardize the <a href="/wiki/X11" class="mw-redirect" title="X11">X11</a> system led to its introduction and widespread use on Sun systems, and NeWS never became widely used.</p>/n<h2><span class="mw-headline" id="The_language">The language</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=PostScript&amp;action=edit&amp;section=11" title="Edit section: The language">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>PostScript is a <a href="/wiki/Turing-complete" class="mw-redirect" title="Turing-complete">Turing-complete</a> programming language, belonging to the <a href="/wiki/Concatenative_programming_language" title="Concatenative programming language">concatenative</a> group. Typically, PostScript programs are not produced by humans, but by other programs. However, it is possible to write computer programs in PostScript just like any other programming language.<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">[5]</a></sup></p>/n<p>PostScript is an <a href="/wiki/Interpreted_language" title="Interpreted language">interpreted</a>, <a href="/wiki/Stack-oriented_programming_language" title="Stack-oriented programming language">stack-based</a> language similar to <a href="/wiki/Forth_(programming_language)" title="Forth (programming language)">Forth</a> but with strong dynamic <a href="/wiki/Type_system" title="Type system">typing</a>, data structures inspired by those found in <a href="/wiki/Lisp_(programming_language)" title="Lisp (programming language)">Lisp</a>, <a href="/w/index.php?title=Scoped_memory&amp;action=edit&amp;redlink=1" class="new" title="Scoped memory (page does not exist)">scoped memory</a> and, since language level 2, <a href="/wiki/Garbage_collection_(computer_science)" title="Garbage collection (computer science)">garbage collection</a>. The language syntax uses <a href="/wiki/Reverse_Polish_notation" title="Reverse Polish notation">reverse Polish notation</a>, which makes the order of operations unambiguous, but reading a program requires some practice, because one has to keep the layout of the <a href="/wiki/Stack_(data_structure)" class="mw-redirect" title="Stack (data structure)">stack</a> in mind. Most <i>operators</i> (what other languages term <i>functions</i>) take their arguments from the stack, and place their results onto the stack. <i><a href="/wiki/Literal_(computer_science)" class="mw-redirect" title="Literal (computer science)">Literals</a></i> (for example, numbers) have the effect of placing a copy of themselves on the stack. Sophisticated data structures can be built on the <i>array</i> and <i>dictionary</i> types, but cannot be declared to the type system, which sees them all only as arrays and dictionaries, so any further typing discipline to be applied to such user-defined "types" is left to the code that implements them.</p>/n<p>The character "%" is used to introduce comments in PostScript programs. As a general convention, every PostScript program should start with the characters "%!PS" as an <a href="/wiki/Interpreter_directive" title="Interpreter directive">interpreter directive</a> so that all devices will properly interpret it as PostScript.</p>/n<h3><span id="&quot;Hello_world&quot;"></span><span class="mw-headline" id=".22Hello_world.22">"Hello world"</span><span class="mw-editsection-bracket">]</span></span></h3>/n<p>A <a href="/wiki/Hello_World_program" class="mw-redirect" title="Hello World program">Hello World program</a>, the customary way to show a small example of a complete program in a given language, might look like this in PostScript (level 2):</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n <span class="c1">%!PS</span>/n <span
