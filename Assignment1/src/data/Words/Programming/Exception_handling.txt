id="contentSub"></div>/n												<div id="jump-to-nav" class="mw-jump">/n					Jump to:					<a href="#mw-head">navigation</a>, 					<a href="#p-search">search</a>/n				</div>/n				<div id="mw-content-text" lang="en" dir="ltr" class="mw-content-ltr"><div class="mw-parser-output"><p><b>Exception handling</b> is the process of responding to the occurrence, during <a href="/wiki/Computation" title="Computation">computation</a>, of <i>exceptions</i> – anomalous or exceptional conditions requiring special processing – often changing the normal flow of <a href="/wiki/Computer_program" title="Computer program">program</a> <a href="/wiki/Execution_(computing)" title="Execution (computing)">execution</a>. It is provided by specialized <a href="/wiki/Programming_language" title="Programming language">programming language</a> constructs, <a href="/wiki/Computer_hardware" title="Computer hardware">computer hardware</a> mechanisms like <a href="/wiki/Interrupt" title="Interrupt">interrupts</a> or <a href="/wiki/Operating_system" title="Operating system">operating system</a> <a href="/wiki/Inter-process_communication" title="Inter-process communication">IPC</a> facilities like <a href="/wiki/Signal_(IPC)" title="Signal (IPC)">signals</a>.</p>/n<p>In general, an exception breaks the normal flow of execution and executes a pre-registered <i>exception handler</i>. The details of how this is done depends on whether it is a hardware or software exception and how the software exception is implemented. Some exceptions, especially hardware ones, may be handled so gracefully that execution can resume where it was interrupted.</p>/n<p>Alternative approaches to exception handling in software are error checking, which maintains normal program flow with later explicit checks for contingencies reported using special return values or some auxiliary global variable such as C's <a href="/wiki/Errno" class="mw-redirect" title="Errno"><code class="mw-highlight" dir="ltr"><span class="n">errno</span></code></a> or floating point status flags; or input validation to preemptively filter exceptional cases.</p>/n<p>Some programmers write software with <b>error reporting</b> features that collect details that may be helpful in fixing the problem, and display those details on the screen, or store them to a file such as a <a href="/wiki/Core_dump" title="Core dump">core dump</a>, or in some cases an <a href="/wiki/Motorola_Q#Error_reporting" title="Motorola Q">automatic error reporting</a> system such as <a href="/wiki/Windows_Error_Reporting" title="Windows Error Reporting">Windows Error Reporting</a> can automatically <a href="/wiki/Phoning_home" title="Phoning home">phone home</a> and email those details to the programmers.</p>/n<p></p>/n<div id="toc" class="toc">/n<div class="toctitle">/n<h2>Contents</h2>/n</div>/n<ul>/n<li class="toclevel-1 tocsection-1"><a links</span></a></li>/n</ul>/n</div>/n<p></p>/n<h2><span class="mw-headline" id="Exception_handling_in_hardware">Exception handling in hardware</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Exception_handling&amp;action=edit&amp;section=1" title="Edit section: Exception handling in hardware">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Hardware exception mechanisms are processed by the CPU. It is intended to support error detection and redirects the program flow to error handling service routines. The state before the exception is saved on the stack.<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">[1]</a></sup></p>/n<h3><span id="Hardware_exception_handling/traps:_IEEE_754_floating_point"></span><span class="mw-headline" id="Hardware_exception_handling.2Ftraps:_IEEE_754_floating_point">Hardware exception handling/traps: IEEE 754 floating point</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Exception_handling&amp;action=edit&amp;section=2" title="Edit section: Hardware exception handling/traps: IEEE 754 floating point">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>Exception handling in the <a href="/wiki/IEEE_754" title="IEEE 754">IEEE 754</a> <a href="/wiki/Floating_point#Dealing_with_exceptional_cases" class="mw-redirect" title="Floating point">floating point</a> hardware standard refers in general to exceptional conditions and defines an exception as "an event that occurs when an operation on some particular operands has no outcome suitable for every reasonable application. That operation might signal one or more exceptions by invoking the default or, if explicitly requested, a language-defined alternate handling."</p>/n<p>By default, an IEEE 754 exception is resumable and is handled by substituting a predefined value for different exceptions, e.g. infinity for a divide by zero exception, and providing <a href="/wiki/Floating_point#Dealing_with_exceptional_cases" class="mw-redirect" title="Floating point">status flags</a> for later checking of whether the exception occurred (see <a href="/wiki/C99#IEEE_754_floating_point_support" title="C99">C99 programming language</a> for a typical example of handling of IEEE 754 exceptions). An exception-handling style enabled by the use of status flags involves: first computing an expression using a fast, direct implementation; checking whether it failed by testing status flags; and then, if necessary, calling a slower, more numerically robust, implementation.<sup id="cite_ref-Xiaoye_Li_and_James_Demmel_1994_983.E2.80.93992_2-0" class="reference"><a href="#cite_note-Xiaoye_Li_and_James_Demmel_1994_983.E2.80.93992-2">[2]</a></sup></p>/n<p>The IEEE 754 standard uses the term "trapping" to refer to the calling of a user-supplied exception-handling routine on exceptional conditions, and is an optional feature of the standard. The standard recommends several usage scenarios for this, including the implementation of non-default pre-substitution of a value followed by resumption, to concisely handle <a href="/wiki/Removable_singularity" title="Removable singularity">removable singularities</a>.<sup id="cite_ref-Xiaoye_Li_and_James_Demmel_1994_983.E2.80.93992_2-1" class="reference"><a href="#cite_note-Xiaoye_Li_and_James_Demmel_1994_983.E2.80.93992-2">[2]</a></sup><sup id="cite_ref-grail_3-0" class="reference"><a href="#cite_note-grail-3">[3]</a></sup><sup id="cite_ref-4" class="reference"><a href="#cite_note-4">[4]</a></sup></p>/n<p>The default IEEE 754 exception handling behaviour of resumption following pre-substitution of a default value avoids the risks inherent in changing flow of program control on numerical exceptions. For example, in 1996 the <a href="/wiki/Ariane_5_Flight_501" class="mw-redirect" title="Ariane 5 Flight 501">maiden flight of the Ariane 5 (Flight 501)</a> ended in a catastrophic explosion due in part to the <a href="/wiki/Ada_(programming_language)" title="Ada (programming language)">Ada</a> programming language exception handling policy of aborting computation on arithmetic error, which in this case was a 64-bit floating point to 16-bit integer conversion <a href="/wiki/Integer_overflow" title="Integer overflow">overflow</a>.<sup id="cite_ref-grail_3-1" class="reference"><a href="#cite_note-grail-3">[3]</a></sup> In the Ariane Flight 501 case, the programmers protected only four out of seven critical variables against overflow due to concerns about the computational constraints of the on-board computer and relied on what turned out to be incorrect assumptions about the possible range of values for the three unprotected variables because they <a href="/wiki/Code_reuse" title="Code reuse">reused code</a> from the Ariane 4, for which their assumptions were correct.<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">[5]</a></sup> According to <a href="/wiki/William_Kahan" title="William Kahan">William Kahan</a>, the loss of Flight 501 would have been avoided if the IEEE 754 exception-handling policy of default substitution had been used because the overflowing 64-bit to 16-bit conversion that caused the software to abort occurred in a piece of code that turned out to be completely unnecessary on the Ariane 5.<sup id="cite_ref-grail_3-2" class="reference"><a href="#cite_note-grail-3">[3]</a></sup> The official report on the crash (conducted by an inquiry board headed by <a href="/wiki/Jacques-Louis_Lions" title="Jacques-Louis Lions">Jacques-Louis Lions</a>) noted that "An underlying theme in the development of Ariane 5 is the bias towards the <a href="/wiki/Fault-tolerant_computer_system" title="Fault-tolerant computer system">mitigation of random failure</a>. The supplier of the <a href="/wiki/Inertial_navigation_system" title="Inertial navigation system">inertial navigation system</a> (SRI) was only following the specification given to it, which stipulated that in the event of any detected exception the processor was to be stopped. The exception which occurred was not due to random failure but a design error. The exception was detected, but inappropriately handled because the view had been taken that software should be considered correct until it is shown to be at fault. [...] Although the failure was due to a systematic software design error, mechanisms can be introduced to mitigate this type of problem. For example the computers within the SRIs could have continued to provide their best estimates of the required <a href="/wiki/Attitude_control" title="Attitude control">attitude</a> information. There is reason for concern that a software exception should be allowed, or even required, to cause a processor to halt while handling mission-critical equipment. Indeed, the loss of a proper software function is hazardous because the same software runs in both SRI units. In the case of Ariane 501, this resulted in the switch-off of two still healthy critical units of equipment."<sup id="cite_ref-6" class="reference"><a href="#cite_note-6">[6]</a></sup></p>/n<p>From the processing point of view, hardware <a href="/wiki/Interrupt" title="Interrupt">interrupts</a> are similar to resumable exceptions, though they are typically unrelated to the user program's <a href="/wiki/Control_flow" title="Control flow">control flow</a>.</p>/n<h2><span class="mw-headline" id="Exception_handling_facilities_provided_by_the_operating_system">Exception handling facilities provided by the operating system</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Exception_handling&amp;action=edit&amp;section=3" title="Edit section: Exception handling facilities provided by the operating system">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>The <a href="/wiki/Operating_system" title="Operating system">operating system</a> may provide facilities for handling exceptions in programs via <a href="/wiki/Inter-process_communication" title="Inter-process communication">IPC</a>. Typically, interrupts caused by the execution of a process are handled by the interrupt service routines of the operating system, and the operating system may then send a <a href="/wiki/Signal_(IPC)" title="Signal (IPC)">signal</a> to that process, which may have asked the operating system to register a signal handler to be called when the signal is raised, or let the operating system execute a default action (like terminating the program). Typical examples are <a href="/wiki/SIGSEGV" class="mw-redirect" title="SIGSEGV">SIGSEGV</a>, <a href="/wiki/SIGBUS" class="mw-redirect" title="SIGBUS">SIGBUS</a>, <a href="/wiki/SIGILL" class="mw-redirect" title="SIGILL">SIGILL</a> and <a href="/wiki/SIGFPE" class="mw-redirect" title="SIGFPE">SIGFPE</a>.</p>/n<h2><span class="mw-headline" id="Exception_handling_in_software">Exception handling in software</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Exception_handling&amp;action=edit&amp;section=4" title="Edit section: Exception handling in software">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Software exception handling and the support provided by software tools differs somewhat from what is understood under exception in hardware, but similar concepts are involved. In programming language mechanisms for exception handling, the term <i>exception</i> is typically used in a specific sense to denote a data structure storing information about an exceptional condition. One mechanism to transfer control, or <i>raise</i> an exception, is known as a <i>throw</i>. The exception is said to be <i>thrown</i>. Execution is transferred to a "catch".</p>/n<p>From the point of view of the author of a <a href="/wiki/Subroutine" title="Subroutine">routine</a>, raising an exception is a useful way to signal that a routine could not execute normally - for example, when an input argument is invalid (e.g. value is outside of the <a href="/wiki/Domain_of_a_function" title="Domain of a function">domain of a function</a>) or when a resource it relies on is unavailable (like a missing file, a hard disk error, or out-of-memory errors). In systems without exceptions, routines would need to return some special <a href="/wiki/Error_code" class="mw-redirect" title="Error code">error code</a>. However, this is sometimes complicated by the <a href="/wiki/Semipredicate_problem" title="Semipredicate problem">semipredicate problem</a>, in which users of the routine need to write extra code to distinguish normal return values from erroneous ones.</p>/n<p>Programming languages differ substantially in their notion of what is an exception. Contemporary languages can roughly be divided in two groups:<sup id="cite_ref-Kiniry_7-0" class="reference"><a href="#cite_note-Kiniry-7">[7]</a></sup></p>/n<ul>/n<li>Languages where exceptions are designed to be used as flow control structures: Ada, Java, Modula-3, ML, OCaml, Python, and Ruby fall in this category.</li>/n<li>Languages where exceptions are only used to handle abnormal, unpredictable, erroneous situations: C++,<sup id="cite_ref-8" class="reference"><a href="#cite_note-8">[8]</a></sup> C#, Common Lisp, Eiffel, and Modula-2.</li>/n</ul>/n<p>Kiniry also notes that "Language design only partially influences the use of exceptions, and consequently, the manner in which one handles partial and total failures during system execution. The other major influence is examples of use, typically in core libraries and code examples in technical books, magazine articles, and online discussion forums, and in an organization’s code standards."<sup id="cite_ref-Kiniry_7-1" class="reference"><a href="#cite_note-Kiniry-7">[7]</a></sup></p>/n<p>Contemporary applications face many design challenges when considering exception handling strategies. Particularly in modern enterprise level applications, exceptions must often cross process boundaries and machine boundaries. Part of designing a solid exception handling strategy is recognizing when a process has failed to the point where it cannot be economically handled by the software portion of the process.<sup id="cite_ref-9" class="reference"><a href="#cite_note-9">[9]</a></sup></p>/n<p>Exception handling is often not handled correctly in software, especially when there are multiple sources of exceptions; <a href="/wiki/Data_flow_analysis" class="mw-redirect" title="Data flow analysis">data flow analysis</a> of 5 million lines of Java code found over 1300 exception handling defects.<sup id="cite_ref-toplas2008_10-0" class="reference"><a href="#cite_note-toplas2008-10">[10]</a></sup></p>/n<h3><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Exception_handling&amp;action=edit&amp;section=5" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>Software exception handling developed in <a href="/wiki/Lisp_(programming_language)" title="Lisp (programming language)">Lisp</a> in the 1960s and 1970s. This originated in <a href="/wiki/LISP_1.5" class="mw-redirect" title="LISP 1.5">LISP 1.5</a> (1962), where exceptions were <i>caught</i> by the <code>ERRSET</code> keyword, which returned <code>NIL</code> in case of an error, instead of terminating the program or entering the debugger.<sup id="cite_ref-FOOTNOTEGabrielSteele20083_11-0" class="reference"><a href="#cite_note-FOOTNOTEGabrielSteele20083-11">[11]</a></sup> Error <i>raising</i> was introduced in <a href="/wiki/MacLisp" class="mw-redirect" title="MacLisp">MacLisp</a> in the late 1960s via the <code>ERR</code> keyword.<sup id="cite_ref-FOOTNOTEGabrielSteele20083_11-1" class="reference"><a href="#cite_note-FOOTNOTEGabrielSteele20083-11">[11]</a></sup> This was rapidly used not only for error raising, but for non-local control flow, and thus was augmented by two new keywords, <code>CATCH</code> and <code>THROW</code> (MacLisp June 1972), reserving <code>ERRSET</code> and <code>ERR</code> for error handling. The cleanup behavior now generally called "finally" was introduced in <a href="/wiki/NIL_(programming_language)" title="NIL (programming language)">NIL</a> (New Implementation of LISP) in the mid- to late-1970s as <code>UNWIND-PROTECT</code>.<sup id="cite_ref-FOOTNOTEWhite1979194_12-0" class="reference"><a href="#cite_note-FOOTNOTEWhite1979194-12">[12]</a></sup> This was then adopted by <a href="/wiki/Common_Lisp" title="Common Lisp">Common Lisp</a>. Contemporary with this was <code>dynamic-wind</code> in Scheme, which handled exceptions in closures. The first papers on structured exception handling were <a href="#CITEREFGoodenough1975a">Goodenough (1975a)</a> and <a href="#CITEREFGoodenough1975b">Goodenough (1975b)</a>.<sup id="cite_ref-FOOTNOTEStroustrup1994392_13-0" class="reference"><a href="#cite_note-FOOTNOTEStroustrup1994392-13">[13]</a></sup> Exception handling was subsequently widely adopted by many programming languages from the 1980s onward.</p>/n<p>Originally, software exception handling included both resumable exceptions (resumption semantics), like most hardware exceptions, and non-resumable exceptions (termination semantics). However, resumption semantics were considered ineffective in practice in the 1970s and 1980s (see C++ standardization discussion, quoted below)<sup id="cite_ref-FOOTNOTEStroustrup199416.6_Exception_Handling:_Resumption_vs._Termination.2C_pp._390.E2.80.93393_14-0" class="reference"><a href="#cite_note-FOOTNOTEStroustrup199416.6_Exception_Handling:_Resumption_vs._Termination.2C_pp._390.E2.80.93393-14">[14]</a></sup> and are no longer in common use, though provided by programming languages like Common Lisp and Dylan.</p>/n<h3><span class="mw-headline" id="Termination_semantics">Termination semantics</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Exception_handling&amp;action=edit&amp;section=6" title="Edit section: Termination semantics">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>Exception handling mechanisms in contemporary languages are typically non-resumable ("termination semantics") as opposed to hardware exceptions, which are typically resumable. This is based on experience of using both, as there are theoretical and design arguments in favor of either decision; these were extensively debated during C++ standardization discussions 1989–1991, which resulted in a definitive decision for termination semantics.<sup id="cite_ref-FOOTNOTEStroustrup199416.6_Exception_Handling:_Resumption_vs._Termination.2C_pp._390.E2.80.93393_14-1" class="reference"><a href="#cite_note-FOOTNOTEStroustrup199416.6_Exception_Handling:_Resumption_vs._Termination.2C_pp._390.E2.80.93393-14">[14]</a></sup> On the rationale for such a design for the C++ mechanism, <a href="/wiki/Bjarne_Stroustrup" title="Bjarne Stroustrup">Stroustrup</a> notes:</p>/n<blockquote class="templatequote">/n<p>[A]t the Palo Alto [C++ standardization] meeting in November 1991, we heard a brilliant summary of the arguments for termination semantics backed with both personal experience and data from <a href="/wiki/James_G._Mitchell" title="James G. Mitchell">Jim Mitchell</a> (from Sun, formerly from Xerox PARC). Jim had used exception handling in half a dozen languages over a period of 20 years and was an early proponent of resumption semantics as one of the main designers and implementers of Xerox's <a href="/wiki/Mesa/Cedar" class="mw-redirect" title="Mesa/Cedar">Cedar/Mesa</a> system. His message was</p>/n<dl>/n<dd>“termination is preferred over resumption; this is not a matter of opinion but a matter of years of experience. Resumption is seductive, but not valid.”</dd>/n</dl>/n<p>He backed this statement with experience from several operating systems. The key example was Cedar/Mesa: It was written by people who liked and used resumption, but after ten years of use, there was only one use of resumption left in the half million line system – and that was a context inquiry. Because resumption wasn't actually necessary for such a context inquiry, they removed it and found a significant speed increase in that part of the system. In each and every case where resumption had been used it had – over the ten years – become a problem and a more appropriate design had replaced it. Basically, every use of resumption had represented a failure to keep separate levels of abstraction disjoint.<sup id="cite_ref-FOOTNOTEStroustrup1994392_13-1" class="reference"><a href="#cite_note-FOOTNOTEStroustrup1994392-13">[13]</a></sup></p>/n</blockquote>/n<h3><span class="mw-headline" id="Criticism">Criticism</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Exception_handling&amp;action=edit&amp;section=7" title="Edit section: Criticism">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>A contrasting view on the safety of exception handling was given by <a href="/wiki/Tony_Hoare" title="Tony Hoare">C.A.R Hoare</a> in 1980, describing the <a href="/wiki/Ada_(programming_language)" title="Ada (programming language)">Ada programming language</a> as having "...a plethora of features and notational conventions, many of them unnecessary and some of them, like exception handling, even dangerous. [...] Do not allow this language in its present state to be used in applications where reliability is critical[...]. The next rocket to go astray as a result of a programming language error may not be an exploratory space rocket on a harmless trip to Venus: It may be a nuclear warhead exploding over one of our own cities." <sup id="cite_ref-15" class="reference"><a href="#cite_note-15">[15]</a></sup></p>/n<p>Citing multiple prior studies by others (1999–2004) and their own results, Weimer and Necula wrote that a significant problem with exceptions is that they "create hidden control-flow paths that are difficult for programmers to reason about".<sup id="cite_ref-toplas2008_10-1" class="reference"><a href="#cite_note-toplas2008-10">[10]</a></sup><sup class="reference" style="white-space:nowrap;">:8:27</sup></p>/n<p><a href="/wiki/Go_(programming_language)" title="Go (programming language)">Go</a> was initially released with exception handling explicitly omitted, with the developers arguing that it obfuscated <a href="/wiki/Control_flow" title="Control flow">control flow</a>.<sup id="cite_ref-16" class="reference"><a href="#cite_note-16">[16]</a></sup> Later, the exception-like <span style="font-family: monospace, monospace;">panic</span>/<span style="font-family: monospace, monospace;">recover</span> mechanism was added to the language, which the Go authors advise using only for unrecoverable errors that should halt the entire process.<sup id="cite_ref-17" class="reference"><a href="#cite_note-17">[17]</a></sup><sup id="cite_ref-18" class="reference"><a href="#cite_note-18">[18]</a></sup><sup id="cite_ref-19" class="reference"><a href="#cite_note-19">[19]</a></sup><sup id="cite_ref-20" class="reference"><a href="#cite_note-20">[20]</a></sup></p>/n<p>One case of early criticism against Exception handling was dealing with resource leaks or state inconsistence, such as escaping a section locked by a <a href="/wiki/Mutex" class="mw-redirect" title="Mutex">mutex</a>, or one temporarily holding a file open. This have largely been solved by <a href="/wiki/RAII" class="mw-redirect" title="RAII">RAII</a> and the <a href="/wiki/Dispose_pattern" title="Dispose pattern">dispose pattern</a>, which had to be specifically invented to solve the issue, but have proven to be useful in other contexts.</p>/n<h3><span class="mw-headline" id="Exception_support_in_programming_languages">Exception support in programming languages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Exception_handling&amp;action=edit&amp;section=8" title="Edit section: Exception support in programming languages">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<div role="note" class="hatnote navigation-not-searchable">See also: <a href="/wiki/Exception_handling_syntax" title="Exception handling syntax">Exception handling syntax</a></div>/n<p>Many computer languages have built-in support for exceptions and exception handling. This includes <a href="/wiki/ActionScript" title="ActionScript">ActionScript</a>, <a href="/wiki/Ada_programming_language" class="mw-redirect" title="Ada programming language">Ada</a>, <a href="/wiki/BlitzMax" class="mw-redirect" title="BlitzMax">BlitzMax</a>, <a href="/wiki/C%2B%2B" title="C++">C++</a>, <a href="/wiki/C_Sharp_(programming_language)" title="C Sharp (programming language)">C#</a>, <a href="/wiki/COBOL" title="COBOL">COBOL</a>, <a href="/wiki/D_programming_language" class="mw-redirect" title="D programming language">D</a>, <a href="/wiki/ECMAScript" title="ECMAScript">ECMAScript</a>, <a href="/wiki/Eiffel_(programming_language)" title="Eiffel (programming language)">Eiffel</a>, <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>, <a href="/wiki/ML_programming_language" class="mw-redirect" title="ML programming language">ML</a>, <a href="/wiki/Object_Pascal" title="Object Pascal">Object Pascal</a> (e.g. <a href="/wiki/Delphi_programming_language" class="mw-redirect" title="Delphi programming language">Delphi</a>, <a href="/wiki/Free_Pascal" title="Free Pascal">Free Pascal</a>, and the like), <a href="/wiki/PowerBuilder" title="PowerBuilder">PowerBuilder</a>, <a href="/wiki/Objective-C" title="Objective-C">Objective-C</a>, <a href="/wiki/OCaml" title="OCaml">OCaml</a>, <a href="/wiki/PHP" title="PHP">PHP</a> (as of version 5), <a href="/wiki/PL/1" class="mw-redirect" title="PL/1">PL/1</a>, <a href="/wiki/PL/SQL" title="PL/SQL">PL/SQL</a>, <a href="/wiki/Prolog" title="Prolog">Prolog</a>, <a href="/wiki/Python_(programming_language)" title="Python (programming language)">Python</a>, <a href="/wiki/REALbasic" class="mw-redirect" title="REALbasic">REALbasic</a>, <a href="/wiki/Ruby_(programming_language)" title="Ruby (programming language)">Ruby</a>, <a href="/wiki/Scala_(programming_language)" title="Scala (programming language)">Scala</a>, <a href="/wiki/Seed7" title="Seed7">Seed7</a>, <a href="/wiki/Tcl" title="Tcl">Tcl</a>, <a href="/wiki/Visual_Prolog" title="Visual Prolog">Visual Prolog</a> and most <a href="/wiki/.NET_Framework" title=".NET Framework">.NET</a> languages. Exception handling is commonly not resumable in those languages, and when an exception is thrown, the program searches back through the <a href="/wiki/Call_stack" title="Call stack">stack</a> of function calls until an exception handler is found.</p>/n<p>Some languages call for <a href="/wiki/Stack_unwinding" class="mw-redirect" title="Stack unwinding">unwinding</a> the stack as this search progresses. That is, if function <span style="font-family: monospace, monospace;">f</span>, containing a handler <span style="font-family: monospace, monospace;">H</span> for exception <span style="font-family: monospace, monospace;">E</span>, calls function <span style="font-family: monospace, monospace;">g</span>, which in turn calls function <span style="font-family: monospace, monospace;">h</span>, and an exception <span style="font-family: monospace, monospace;">E</span> occurs in <span style="font-family: monospace, monospace;">h</span>, then functions <span style="font-family: monospace, monospace;">h</span> and <span style="font-family: monospace, monospace;">g</span> may be terminated, and <span style="font-family: monospace, monospace;">H</span> in <span style="font-family: monospace, monospace;">f</span> will handle <span style="font-family: monospace, monospace;">E</span>.</p>/n<p>An exception-handling language without this unwinding is <a href="/wiki/Common_Lisp" title="Common Lisp">Common Lisp</a> with its <a href="/wiki/Exception_handling#Condition_systems" title="Exception handling">Condition System</a>. Common Lisp calls the exception handler and does not unwind the stack. This allows the program to continue the computation at exactly the same place where the error occurred (for example when a previously missing file has become available). The stackless implementation of the <i>Mythryl</i> programming language supports constant-time exception handling without stack unwinding.</p>/n<p>Excluding minor syntactic differences, there are only a couple of exception handling styles in use. In the most popular style, an exception is initiated by a special statement (<code>throw</code> or <code>raise</code>) with an exception object (e.g. with Java or Object Pascal) or a value of a special extendable enumerated type (e.g. with Ada). The scope for exception handlers starts with a marker clause (<code>try</code> or the language's block starter such as <code>begin</code>) and ends in the start of the first handler clause (<code>catch</code>, <code>except</code>, <code>rescue</code>). Several handler clauses can follow, and each can specify which exception types it handles and what name it uses for the exception object.</p>/n<p>A few languages also permit a clause (<code>else</code>) that is used in case no exception occurred before the end of the handler's scope was reached.</p>/n<p>More common is a related clause (<code>finally</code> or <code>ensure</code>) that is executed whether an exception occurred or not, typically to release resources acquired within the body of the exception-handling block. Notably, C++ does not provide this construct, since it encourages the <a href="/wiki/Resource_Acquisition_Is_Initialization" class="mw-redirect" title="Resource Acquisition Is Initialization">Resource Acquisition Is Initialization</a> (RAII) technique which frees resources using <a href="/wiki/Destructor_(computer_programming)" title="Destructor (computer programming)">destructors</a>.</p>/n<p>In its whole, exception handling code might look like this (in <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>-like <a href="/wiki/Pseudocode" title="Pseudocode">pseudocode</a>; note that an exception type called <code>EmptyLineException</code> would need to be declared somewhere):</p>/n<div class="mw-highlight mw-content-ltr" class="n">readLine</span><span class="p">();</span>/n/n    <span class="k">if</span> <span class="p">(</span><span class="n">line</span><span class="p">.</span><span class="n">length</span><span class="p">()</span> <span class="p">==</span> <span class="m">0</span><span class="p">)</span> <span class="p">{</span>/n        <span class="k">throw</span> <span class="k">new</span> <span minor variation, some languages use a single handler clause, which deals with the class of the exception internally.</p>/n<p>According to a 2008 paper by Westley Wiemer and <a href="/wiki/George_Necula" title="George Necula">George Necula</a>, the syntax of the <code>try</code>...<code>finally</code> blocks in Java is a contributing factor to software defects. When a method needs to handle the acquisition and release of 3–5 resources, programmers are apparently unwilling to nest enough blocks due to readability concerns, even when this would be a correct solution. It is possible to use a single <code>try</code>...<code>finally</code> block even when dealing with multiple resources, but that requires a correct use of <a href="/wiki/Sentinel_value" title="Sentinel value">sentinel values</a>, which is another common source of bugs for this type of problem.<sup id="cite_ref-toplas2008_10-2" class="reference"><a href="#cite_note-toplas2008-10">[10]</a></sup><sup class="reference" style="white-space:nowrap;">:8:6–8:7</sup> Regarding the semantics of the <code>try</code>...<code>catch</code>...<code>finally</code> construct in general, Wiemer and Necula write that "While try-catch-finally is conceptually simple, it has the most complicated execution description in the language specification [Gosling et al. 1996] and requires four levels of nested “if”s in its official English description. In short, it contains a large number of <a href="/wiki/Corner_cases" class="mw-redirect" title="Corner cases">corner cases</a> that programmers often overlook."<sup id="cite_ref-toplas2008_10-3" class="reference"><a href="#cite_note-toplas2008-10">[10]</a></sup><sup class="reference" style="white-space:nowrap;">:8:13–8:14</sup></p>/n<p>C supports various means of error checking, but generally is not considered to support "exception handling," although <a href="/wiki/Setjmp.h" title="Setjmp.h">the <code>setjmp</code> and <code>longjmp</code> standard library functions</a> can be used to implement exception semantics.</p>/n<p><a href="/wiki/Perl" title="Perl">Perl</a> has optional support for structured exception handling.</p>/n<p><a href="/wiki/Python_(programming_language)" title="Python (programming language)">Python</a>'s support for exception handling is pervasive and consistent. It's difficult to write a robust Python program without using its <code class="mw-highlight" dir="ltr"><span class="k">try</span></code> and <code class="mw-highlight" dir="ltr"><span class="k">except</span></code> keywords.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (November 2016)">citation needed</span></a></i>]</sup></p>/n<h3><span class="mw-headline" id="Exception_handling_implementation">Exception handling implementation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Exception_handling&amp;action=edit&amp;section=9" title="Edit section: Exception handling implementation">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>The implementation of exception handling in programming languages typically involves a fair amount of support from both a code generator and the <a href="/wiki/Runtime_system" title="Runtime system">runtime system</a> accompanying a compiler. (It was the addition of exception handling to C++ that ended the useful lifetime of the original C++ compiler, <a href="/wiki/Cfront" title="Cfront">Cfront</a>.<sup id="cite_ref-21" class="reference"><a href="#cite_note-21">[21]</a></sup>) Two schemes are most common. The first, <i>dynamic registration</i>, generates code that continually updates structures about the program state in terms of exception handling.<sup id="cite_ref-22" class="reference"><a href="#cite_note-22">[22]</a></sup> Typically, this adds a new element to the <a href="/wiki/Call_stack" title="Call stack">stack frame layout</a> that knows what handlers are available for the function or method associated with that frame; if an exception is thrown, a pointer in the layout directs the runtime to the appropriate handler code. This approach is compact in terms of space, but adds execution overhead on frame entry and exit. It was commonly used in many Ada implementations, for example, where complex generation and runtime support was already needed for many other language features. Dynamic registration, being fairly straightforward to define, is amenable to <a href="/wiki/Proof_of_correctness" class="mw-redirect" title="Proof of correctness">proof of correctness</a>.<sup id="cite_ref-23" class="reference"><a href="#cite_note-23">[23]</a></sup></p>/n<p>The second scheme, and the one implemented in many production-quality C++ compilers, is a <i>table-driven</i> approach. This creates static tables at <a href="/wiki/Compile_time" title="Compile time">compile time</a> and <a href="/wiki/Link_time" title="Link time">link time</a> that relate ranges of the <a href="/wiki/Program_counter" title="Program counter">program counter</a> to the program state with respect to exception handling.<sup id="cite_ref-24" class="reference"><a href="#cite_note-24">[24]</a></sup> Then, if an exception is thrown, the runtime system looks up the current instruction location in the tables and determines what handlers are in play and what needs to be done. This approach minimizes executive overhead for the case where an exception is not thrown. This happens at the cost of some space, but this space can be allocated into read-only, special-purpose data sections that are not loaded or relocated until an exception is actually thrown.<sup id="cite_ref-cppeh_25-0" class="reference"><a href="#cite_note-cppeh-25">[25]</a></sup> This second approach is also superior in terms of achieving <a href="/wiki/Thread_safety" title="Thread safety">thread safety</a><sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (September 2012)">citation needed</span></a></i>]</sup>.</p>/n<p>Other definitional and implementation schemes have been proposed as well.<sup id="cite_ref-26" class="reference"><a href="#cite_note-26">[26]</a></sup> For languages that support <a href="/wiki/Metaprogramming" title="Metaprogramming">metaprogramming</a>, approaches that involve no overhead at all have been advanced.<sup id="cite_ref-27" class="reference"><a href="#cite_note-27">[27]</a></sup></p>/n<h3><span class="mw-headline" id="Exception_handling_based_on_design_by_contract">Exception handling based on design by contract</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Exception_handling&amp;action=edit&amp;section=10" title="Edit section: Exception handling based on design by contract">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>A different view of exceptions is based on the principles of <a href="/wiki/Design_by_contract" title="Design by contract">design by contract</a> and is supported in particular by the <a href="/wiki/Eiffel_(programming_language)" title="Eiffel (programming language)">Eiffel language</a>. The idea is to provide a more rigorous basis for exception handling by defining precisely what is "normal" and "abnormal" behavior. Specifically, the approach is based on two concepts:</p>/n<ul>/n<li><b>Failure</b>: the inability of an operation to fulfill its contract. For example, an addition may produce an arithmetic overflow (it does not fulfill its contract of computing a good approximation to the mathematical sum); or a routine may fail to meet its postcondition.</li>/n<li><b>Exception</b>: an abnormal event occurring during the execution of a routine (that routine is the "<i>recipient</i>" of the exception) during its execution. Such an abnormal event results from the <i>failure</i> of an operation called by the routine.</li>/n</ul>/n<p>The "Safe Exception Handling principle" as introduced by Bertrand Meyer in <a href="/wiki/Object-Oriented_Software_Construction" title="Object-Oriented Software Construction">Object-Oriented Software Construction</a> then holds that there are only two meaningful ways a routine can react when an exception occurs:</p>/n<ul>/n<li>Failure, or "organized panic": The routine fixes the object's state by re-establishing the invariant (this is the "organized" part), and then fails (panics), triggering an exception in its caller (so that the abnormal event is not ignored).</li>/n<li>Retry: The routine tries the algorithm again, usually after changing some values so that the next attempt will have a better chance to succeed.</li>/n</ul>/n<p>In particular, simply ignoring an exception is not permitted; a block must either be retried and successfully complete, or propagate the exception to its caller.</p>/n<p>Here is an example expressed in Eiffel syntax. It assumes that a routine <code class="mw-highlight" dir="ltr"><span class="n">send_fast</span></code> is normally the better way to send a message, but it may fail, triggering an exception; if so, the algorithm next uses <code class="mw-highlight" dir="ltr"><span class="n">send_slow</span></code>, which will fail less often. If <code class="mw-highlight" dir="ltr"><span class="n">send_slow</span></code> fails, the routine <code class="mw-highlight" dir="ltr"><span class="n">send</span></code> as a whole should fail, causing the caller to get an exception.</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n<span class="n">send</span> <span class="p">(</span><span class="n">m</span><span class="p">:</span> <span class="nc">MESSAGE</span><span class="p">)</span> <span class="n">is</span>/n  <span class="c1">-- Send m through fast link, if possible, otherwise through slow link.</span>/n<span class="kr">local</span>/n  <span class="n">tried_fast</span><span class="p">,</span> <span class="n">tried_slow</span><span class="p">:</span> <span class="nc">BOOLEAN</span>/n<span class="kr">do</span>/n  <span class="kr">if</span> <span class="n">tried_fast</span> <span class="kr">then</span>/n     <span class="n">tried_slow</span> <span class="o">:=</span> <span class="kc">True</span>/n     <span class="n">send_slow</span> <span class="p">(</span><span class="n">m</span><span class="p">)</span>/n  <span class="kr">else</span>/n     <span class="n">tried_fast</span> <span class="o">:=</span> <span class="kc">True</span>/n     <span class="n">send_fast</span> <span class="p">(</span><span class="n">m</span><span class="p">)</span>/n  <span class="kr">end</span>/n<span class="kr">rescue</span>/n  <span class="kr">if</span> <span class="ow">not</span> <span class="n">tried_slow</span> <span class="kr">then</span>/n     <span class="kr">retry</span>/n  <span class="kr">end</span>/n<span class="kr">end</span>/n</pre></div>/n<p>The boolean local variables are initialized to False at the start. If <code class="mw-highlight" dir="ltr"><span class="n">send_fast</span></code> fails, the body (<code class="mw-highlight" dir="ltr"><span class="kr">do</span></code> clause) will be executed again, causing execution of <code class="mw-highlight" dir="ltr"><span class="n">send_slow</span></code>. If this execution of <code class="mw-highlight" dir="ltr"><span class="n">send_slow</span></code> fails, the <code class="mw-highlight" dir="ltr"><span class="kr">rescue</span></code> clause will execute to the end with no <code class="mw-highlight" dir="ltr"><span class="kr">retry</span></code> (no <code class="mw-highlight" dir="ltr"><span class="kr">else</span></code> clause in the final <code class="mw-highlight" dir="ltr"><span class="kr">if</span></code>), causing the routine execution as a whole to fail.</p>/n<p>This approach has the merit of defining clearly what "normal" and "abnormal" cases are: an abnormal case, causing an exception, is one in which the routine is unable to fulfill its contract. It defines a clear distribution of roles: the <code class="mw-highlight" dir="ltr"><span class="kr">do</span></code> clause (normal body) is in charge of achieving, or attempting to achieve, the routine's contract; the <code class="mw-highlight" dir="ltr"><span class="kr">rescue</span></code> clause is in charge of reestablishing the context and restarting the process, if this has a chance of succeeding, but not of performing any actual computation.</p>/n<p>Although exceptions in Eiffel have a fairly clear philosophy, Kiniry (2006) criticizes their implementation because "Exceptions that are part of the language definition are represented by INTEGER values, developer-defined exceptions by STRING values. [...] Additionally, because they are basic values and not objects, they have no inherent semantics beyond that which is expressed in a helper routine which necessarily cannot be foolproof because of the representation overloading in effect (e.g., one cannot differentiate two integers of the same value)."<sup id="cite_ref-Kiniry_7-2" class="reference"><a href="#cite_note-Kiniry-7">[7]</a></sup></p>/n<h3><span class="mw-headline" id="Uncaught_exceptions">Uncaught exceptions</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Exception_handling&amp;action=edit&amp;section=11" title="Edit section: Uncaught exceptions">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>If an exception is thrown and not caught (operationally, an exception is thrown when there is no applicable handler specified), the uncaught exception is handled by the runtime; the routine that does this is called the <b><span id="uncaught_exception_handler">uncaught exception handler</span></b>.<sup id="cite_ref-cocoa_28-0" class="reference"><a href="#cite_note-cocoa-28">[28]</a></sup><sup id="cite_ref-29" class="reference"><a href="#cite_note-29">[29]</a></sup> The most common default behavior is to terminate the program and print an error message to the console, usually including debug information such as a string representation of the exception and the <a href="/wiki/Stack_trace" title="Stack trace">stack trace</a>.<sup id="cite_ref-cocoa_28-1" class="reference"><a href="#cite_note-cocoa-28">[28]</a></sup><sup id="cite_ref-30" class="reference"><a href="#cite_note-30">[30]</a></sup><sup id="cite_ref-31" class="reference"><a href="#cite_note-31">[31]</a></sup> This is often avoided by having a top-level (application-level) handler (for example in an <a href="/wiki/Event_loop" title="Event loop">event loop</a>) that catches exceptions before they reach the runtime.<sup id="cite_ref-cocoa_28-2" class="reference"><a href="#cite_note-cocoa-28">[28]</a></sup><sup id="cite_ref-pmotw_32-0" class="reference"><a href="#cite_note-pmotw-32">[32]</a></sup></p>/n<p>Note that even though an uncaught exception may result in the <i>program</i> terminating abnormally (the program may not be correct if an exception is not caught, notably by not rolling back partially completed transactions, or not releasing resources), the <i>process</i> terminates normally (assuming the runtime works correctly), as the runtime (which is controlling execution of the program) can ensure orderly shutdown of the process.</p>/n<p>In a multithreaded program, an uncaught exception in a thread may instead result in termination of just that thread, not the entire process (uncaught exceptions in the thread-level handler are caught by the top-level handler). This is particularly important for servers, where for example a <a href="/wiki/Servlet" class="mw-redirect" title="Servlet">servlet</a> (running in its own thread) can be terminated without the server overall being affected.</p>/n<p>This default uncaught exception handler may be overridden, either globally or per-thread, for example to provide alternative logging or end-user reporting of uncaught exceptions, or to restart threads that terminate due to an uncaught exception. For example, in Java this is done for a single thread via <code><a rel="nofollow" class="external text" href="https://docs.oracle.com/javase/8/docs/api/java/lang/Thread.html#setUncaughtExceptionHandler-java.lang.Thread.UncaughtExceptionHandler-">Thread.setUncaughtExceptionHandler</a></code> and globally via <code><a rel="nofollow" class="external text" href="https://docs.oracle.com/javase/8/docs/api/java/lang/Thread.html#setDefaultUncaughtExceptionHandler-">Thread.setDefaultUncaughtExceptionHandler</a></code>; in Python this is done by modifying <code><a rel="nofollow" class="external text" href="https://docs.python.org/3/library/sys.html#sys.excepthook">sys.excepthook</a></code>.</p>/n<h3><span class="mw-headline" id="Static_checking_of_exceptions">Static checking of exceptions</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Exception_handling&amp;action=edit&amp;section=12" title="Edit section: Static checking of exceptions">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<h4><span class="mw-headline" id="Checked_exceptions">Checked exceptions</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Exception_handling&amp;action=edit&amp;section=13" title="Edit section: Checked exceptions">edit</a><span class="mw-editsection-bracket">]</span></span></h4>/n<p>The designers of Java devised<sup id="cite_ref-33" class="reference"><a href="#cite_note-33">[33]</a></sup><sup id="cite_ref-34" class="reference"><a href="#cite_note-34">[34]</a></sup> checked exceptions,<sup id="cite_ref-35" class="reference"><a href="#cite_note-35">[35]</a></sup> which are a special set of exceptions. The checked exceptions that a method may raise are part of the method's <a href="/wiki/Type_signature" title="Type signature">signature</a>. For instance, if a method might throw an <code class="mw-highlight" dir="ltr"><span class="n">IOException</span></code>, it must declare this fact explicitly in its method signature. Failure to do so raises a compile-time error.</p>/n<p>Kiniry (2006) notes however that Java's libraries (as they were in 2006) were often inconsistent in their approach to error reporting, because "Not all erroneous situations in Java are represented by exceptions though. Many methods return special values which indicate failure encoded as constant field of related classes."<sup id="cite_ref-Kiniry_7-3" class="reference"><a href="#cite_note-Kiniry-7">[7]</a></sup></p>/n<p>Checked exceptions are related to exception checkers that exist for the <a href="/wiki/OCaml" title="OCaml">OCaml</a> programming language.<sup id="cite_ref-36" class="reference"><a href="#cite_note-36">[36]</a></sup> The external tool for OCaml is both invisible (i.e. it does not require any syntactic annotations) and optional (i.e. it is possible to compile and run a program without having checked the exceptions, although this is not recommended for production code).</p>/n<p>The <a href="/wiki/CLU_programming_language" class="mw-redirect" title="CLU programming language">CLU programming language</a> had a feature with the interface closer to what Java has introduced later. A function could raise only exceptions listed in its type, but any leaking exceptions from called functions would automatically be turned into the sole runtime exception, <code class="mw-highlight" id="" style="" dir="ltr">failure</code>, instead of resulting in compile-time error. Later, <a href="/wiki/Modula-3" title="Modula-3">Modula-3</a> had a similar feature.<sup id="cite_ref-37" class="reference"><a href="#cite_note-37">[37]</a></sup> These features don't include the compile time checking that is central in the concept of checked exceptions, and hasn't (as of 2006) been incorporated into major programming languages other than Java.<sup id="cite_ref-38" class="reference"><a href="#cite_note-38">[38]</a></sup></p>/n<p>The C++ programming language introduces an optional mechanism for checked exceptions, called <i>exception specifications</i>. By default any function can throw any exception, but this can be limited by a <code class="mw-highlight" dir="ltr"><span class="k">throw</span></code> clause added to the function signature, that specifies which exceptions the function may throw. Exception specifications are not enforced at compile-time. Violations result in the global function <code class="mw-highlight" dir="ltr"><span class="n">std</span><span class="o">::</span><span class="n">unexpected</span></code> being called.<sup id="cite_ref-bjarne-exc_39-0" class="reference"><a href="#cite_note-bjarne-exc-39">[39]</a></sup> An empty exception specification may be given, which indicates that the function will throw no exception. This was not made the default when exception handling was added to the language because it would require too much modification of existing code, would impede interaction with code written in another language, and would tempt programmers into writing too many handlers at the local level.<sup id="cite_ref-bjarne-exc_39-1" class="reference"><a href="#cite_note-bjarne-exc-39">[39]</a></sup> Explicit use of empty exception specifications can, however, allow C++ compilers to perform significant code and stack layout optimizations that generally have to be suppressed when exception handling may take place in a function.<sup id="cite_ref-cppeh_25-1" class="reference"><a href="#cite_note-cppeh-25">[25]</a></sup> Some analysts view the proper use of exception specifications in C++ as difficult to achieve.<sup id="cite_ref-40" class="reference"><a href="#cite_note-40">[40]</a></sup> In the recent<sup class="plainlinks noprint asof-tag ref" style="display:none;"><a rel="nofollow" class="external text" href="http://www.open-std.org/jtc1/sc22/wg21/">[ref]</a></sup> C++ language standard (<a href="/wiki/C%2B%2B11" title="C++11">C++11</a>), the use of exception specifications as specified in the preceding version of the standard (<a href="/wiki/C%2B%2B03" title="C++03">C++03</a>) is <a href="/wiki/Deprecation" title="Deprecation">deprecated</a>.<sup id="cite_ref-41" class="reference"><a href="#cite_note-41">[41]</a></sup></p>/n<p>In contrast to Java, languages like C# do not enforce that exceptions have to be caught. According to Hanspeter Mössenböck, not distinguishing between to-be-called (checked) exceptions and not-to-be-called (unchecked) exceptions makes the written program more convenient, but less robust, as an uncaught exception results in an abort with a <a href="/wiki/Stack_trace" title="Stack trace">stack trace</a>.<sup id="cite_ref-42" class="reference"><a href="#cite_note-42">[42]</a></sup> Kiniry (2006) notes however that Java's JDK (version 1.4.1) throws a large number of unchecked exceptions: one for every 140 lines of code, whereas Eiffel uses them much more sparingly, with one thrown every 4,600 lines of code. Kiniry also writes that "As any Java programmer knows, the volume of <code><a href="/wiki/Try_catch" class="mw-redirect" title="Try catch">try catch</a></code> code in a typical Java application is sometimes larger than the comparable code necessary for explicit formal parameter and return value checking in other languages that do not have checked exceptions. In fact, the general consensus among in-the-trenches Java programmers is that dealing with checked exceptions is nearly as unpleasant a task as writing documentation. Thus, many programmers report that they “resent” checked exceptions. This leads to an abundance of checked-but-ignored exceptions".<sup id="cite_ref-Kiniry_7-4" class="reference"><a href="#cite_note-Kiniry-7">[7]</a></sup> Kiniry also notes that the developers of C# apparently were influenced by this kind of user experiences, with the following quote being attributed to them (via Eric Gunnerson):</p>/n<blockquote class="templatequote">/n<p>"Examination of small programs leads to the conclusion that requiring exception specifications could both enhance developer productivity and enhance code quality, but experience with large software projects suggests a different result – decreased productivity and little or no increase in code quality."<sup id="cite_ref-Kiniry_7-5" class="reference"><a href="#cite_note-Kiniry-7">[7]</a></sup></p>/n</blockquote>/n<p>According to <a href="/wiki/Anders_Hejlsberg" title="Anders Hejlsberg">Anders Hejlsberg</a> there was fairly broad agreement in their design group to not have checked exceptions as a language feature in C#. Hejlsberg explained in an interview that</p>/n<blockquote class="templatequote">/n<p>“The throws clause, at least the way it's implemented in Java, doesn't necessarily force you to handle the exceptions, but if you don't handle them, it forces you to acknowledge precisely which exceptions might pass through. It requires you to either catch declared exceptions or put them in your own throws clause. To work around this requirement, people do ridiculous things. For example, they decorate every method with, "throws Exception." That just completely defeats the feature, and you just made the programmer write more gobbledy gunk. That doesn't help anybody.”<sup id="cite_ref-43" class="reference"><a href="#cite_note-43">[43]</a></sup></p>/n</blockquote>/n<h4><span class="mw-headline" id="Views_on_usage">Views on usage</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Exception_handling&amp;action=edit&amp;section=14" title="Edit section: Views on usage">edit</a><span class="mw-editsection-bracket">]</span></span></h4>/n<table class="plainlinks metadata ambox ambox-style ambox-essay-like" role="presentation">/n<tr>/n<td class="mbox-image">/n<div style="width:52px"><img alt="" src="//upload.wikimedia.org/wikipedia/en/thumb/f/f2/Edit-clear.svg/40px-Edit-clear.svg.png" width="40" height="40" srcset="//upload.wikimedia.org/wikipedia/en/thumb/f/f2/Edit-clear.svg/60px-Edit-clear.svg.png 1.5x, //upload.wikimedia.org/wikipedia/en/thumb/f/f2/Edit-clear.svg/80px-Edit-clear.svg.png 2x" data-file-width="48" data-file-height="48" /></div>/n</td>/n<td class="mbox-text">/n<div class="mbox-text-span">This section <b>is written like a&#160;personal reflection or opinion essay</b> that states a Wikipedia editor's personal feelings about a topic. <span class="hide-when-compact">Please <a class="external text" href="//en.wikipedia.org/w/index.php?title=Exception_handling&amp;action=edit">help improve it</a> by rewriting it in an <a href="/wiki/Wikipedia:Encyclopedic_style" class="mw-redirect" title="Wikipedia:Encyclopedic style">encyclopedic style</a>.</span> <small><i>(July 2014)</i></small> <small class="hide-when-compact"><i>(<a href="/wiki/Help:Maintenance_template_removal" title="Help:Maintenance template removal">Learn how and when to remove this template message</a>)</i></small></div>/n</td>/n</tr>/n</table>/n<p>Checked exceptions can, at <a href="/wiki/Compile_time" title="Compile time">compile time</a>, reduce the incidence of unhandled exceptions surfacing at <a href="/wiki/Run_time_(program_lifecycle_phase)" title="Run time (program lifecycle phase)">runtime</a> in a given application. Unchecked exceptions (such as the <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a> objects <code class="mw-highlight" dir="ltr"><span class="n">RuntimeException</span></code> and <code class="mw-highlight" dir="ltr"><span class="n">Error</span></code>) remain unhandled.</p>/n<p>However, checked exceptions can either require extensive <code class="mw-highlight" dir="ltr"><span class="kd">throws</span></code> declarations, revealing implementation details and reducing <a href="/wiki/Encapsulation_(computer_science)" class="mw-redirect" title="Encapsulation (computer science)">encapsulation</a>, or encourage coding poorly considered <code class="mw-highlight" dir="ltr"><span class="k">try</span><span class="o">/</span><span class="k">catch</span></code> blocks that can hide legitimate exceptions from their appropriate handlers.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (February 2010)">citation needed</span></a></i>]</sup> Consider a growing <a href="/wiki/Codebase" title="Codebase">codebase</a> over time. An interface may be declared to throw exceptions X and Y. In a later version of the code, if one wants to throw exception Z, it would make the new code incompatible with the earlier uses. Furthermore, with the <a href="/wiki/Adapter_pattern" title="Adapter pattern">adapter pattern</a>, in which one body of code declares an interface that is then implemented by a different body of code so that code can be plugged in and called by the first, the adapter code may have a rich set of exceptions to describe problems, but is forced to use the exception types declared in the interface.</p>/n<p>It is possible to reduce the number of declared exceptions either by declaring a <a href="/wiki/Superclass_(computer_science)" class="mw-redirect" title="Superclass (computer science)">superclass</a> of all potentially thrown exceptions, or by defining and declaring exception types that are suitable for the level of abstraction of the called method<sup id="cite_ref-44" class="reference"><a href="#cite_note-44">[44]</a></sup> and mapping lower level exceptions to these types, preferably wrapped using <a href="/wiki/Exception_chaining" title="Exception chaining">exception chaining</a> in order to preserve the root cause. In addition, it's very possible that in the example above of the changing interface that the calling code would need to be modified as well, since in some sense the exceptions a method may throw are part of the method's implicit interface anyway.</p>/n<p>Using a <code class="mw-highlight" dir="ltr"><span class="k">throws</span> <span class="n">Exception</span></code> declaration or <code class="mw-highlight" dir="ltr"><span class="k">catch</span> <span class="p">(</span><span class="n">Exception</span> <span class="n">e</span><span class="p">)</span></code> is usually sufficient for satisfying the checking in Java. While this may have some use, it essentially circumvents the checked exception mechanism, which Oracle discourages.<sup id="cite_ref-45" class="reference"><a href="#cite_note-45">[45]</a></sup></p>/n<p>Unchecked exception types should generally not be handled, except possibly at the outermost levels of scope. These often represent scenarios that do not allow for recovery: <code class="mw-highlight" dir="ltr"><span class="n">RuntimeException</span></code>s frequently reflect programming defects,<sup id="cite_ref-46" class="reference"><a href="#cite_note-46">[46]</a></sup> and <code class="mw-highlight" dir="ltr"><span class="n">Error</span></code>s generally represent unrecoverable JVM failures. Even in a language that supports checked exceptions, there are cases where the use of checked exceptions is not appropriate.<sup id="cite_ref-47" class="reference"><a href="#cite_note-47">[47]</a></sup></p>/n<h3><span class="mw-headline" id="Dynamic_checking_of_exceptions">Dynamic checking of exceptions</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Exception_handling&amp;action=edit&amp;section=15" title="Edit section: Dynamic checking of exceptions">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>The point of exception handling routines is to ensure that the code can handle error conditions. In order to establish that exception handling routines are sufficiently robust, it is necessary to present the code with a wide spectrum of invalid or unexpected inputs, such as can be created via software <a href="/wiki/Fault_injection" title="Fault injection">fault injection</a> and <a href="/wiki/Mutation_testing" title="Mutation testing">mutation testing</a> (that is also sometimes referred to as <a href="/wiki/Fuzz_testing" class="mw-redirect" title="Fuzz testing">fuzz testing</a>). One of the most difficult types of software for which to write exception handling routines is protocol software, since a robust protocol implementation must be prepared to receive input that does not comply with the relevant specification(s).</p>/n<p>In order to ensure that meaningful regression analysis can be conducted throughout a <a href="/wiki/Software_development_process" title="Software development process">software development lifecycle process</a>, any exception handling testing should be highly automated, and the test cases must be generated in a scientific, repeatable fashion. Several commercially available systems exist that perform such testing.</p>/n<p>In runtime engine environments such as <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a> or <a href="/wiki/.NET_Framework" title=".NET Framework">.NET</a>, there exist tools that attach to the runtime engine and every time that an exception of interest occurs, they record debugging information that existed in memory at the time the exception was thrown (<a href="/wiki/Call_stack" title="Call stack">call stack</a> and <a href="/wiki/Heap_(data_structure)" title="Heap (data structure)">heap</a> values). These tools are called <a href="/wiki/Automated_exception_handling" title="Automated exception handling">automated exception handling</a> or error interception tools and provide 'root-cause' information for exceptions.</p>/n<h3><span class="mw-headline" id="Exception_synchronicity">Exception synchronicity</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Exception_handling&amp;action=edit&amp;section=16" title="Edit section: Exception synchronicity">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>Somewhat related with the concept of checked exceptions is <i>exception synchronicity</i>. Synchronous exceptions happen at a specific program statement whereas <b>asynchronous exceptions</b> can raise practically anywhere.<sup id="cite_ref-48" class="reference"><a href="#cite_note-48">[48]</a></sup><sup id="cite_ref-49" class="reference"><a href="#cite_note-49">[49]</a></sup> It follows that asynchronous exception handling can't be required by the compiler. They are also difficult to program with. Examples of naturally asynchronous events include pressing <a href="/wiki/Control-C" title="Control-C">Ctrl-C</a> to interrupt a program, and receiving a <a href="/wiki/Signal_(computing)" class="mw-redirect" title="Signal (computing)">signal</a> such as "stop" or "suspend" from another <a href="/wiki/Thread_(computer_science)" class="mw-redirect" title="Thread (computer science)">thread of execution</a>.</p>/n<p>Programming languages typically deal with this by limiting asynchronicity, for example Java has deprecated the use of its ThreadDeath exception that was used to allow one thread to stop another one.<sup id="cite_ref-50" class="reference"><a href="#cite_note-50">[50]</a></sup> Instead, there can be semi-asynchronous exceptions that only raise in suitable locations of the program or synchronously.</p>/n<h3><span class="mw-headline" id="Condition_systems">Condition systems</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Exception_handling&amp;action=edit&amp;section=17" title="Edit section: Condition systems">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p><a href="/wiki/Common_Lisp" title="Common Lisp">Common Lisp</a>, <a href="/wiki/Dylan_programming_language" class="mw-redirect" title="Dylan programming language">Dylan</a> and <a href="/wiki/Smalltalk" title="Smalltalk">Smalltalk</a> have a <a href="/w/index.php?title=Condition_system&amp;action=edit&amp;redlink=1" class="new" title="Condition system (page does not exist)">condition system</a><sup id="cite_ref-51" class="reference"><a href="#cite_note-51">[51]</a></sup> (see <a href="/wiki/Common_Lisp#Condition_system" title="Common Lisp">Common Lisp Condition System</a>) that encompasses the aforementioned exception handling systems. In those languages or environments the advent of a condition (a "generalisation of an error" according to <a href="/wiki/Kent_Pitman" title="Kent Pitman">Kent Pitman</a>) implies a function call, and only late in the exception handler the decision to unwind the stack may be taken.</p>/n<p>Conditions are a generalization of exceptions. When a condition arises, an appropriate condition handler is searched for and selected, in stack order, to handle the condition. Conditions that do not represent errors may safely go unhandled entirely; their only purpose may be to propagate hints or warnings toward the user.<sup id="cite_ref-52" class="reference"><a href="#cite_note-52">[52]</a></sup></p>/n<h4><span class="mw-headline" id="Continuable_exceptions">Continuable exceptions</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Exception_handling&amp;action=edit&amp;section=18" title="Edit section: Continuable exceptions">edit</a><span class="mw-editsection-bracket">]</span></span></h4>/n<p>This is related to the so-called <i>resumption model</i> of exception handling, in which some exceptions are said to be <i>continuable</i>: it is permitted to return to the expression that signaled an exception, after having taken corrective action in the handler. The condition system is generalized thus: within the handler of a non-serious condition (a.k.a. <i>continuable exception</i>), it is possible to jump to predefined restart points (a.k.a. <i>restarts</i>) that lie between the signaling expression and the condition handler. Restarts are functions closed over some lexical environment, allowing the programmer to repair this environment before exiting the condition handler completely or unwinding the stack even partially.</p>/n<h4><span class="mw-headline" id="Restarts_separate_mechanism_from_policy">Restarts separate mechanism from policy</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Exception_handling&amp;action=edit&amp;section=19" title="Edit section: Restarts separate mechanism from policy">edit</a><span class="mw-editsection-bracket">]</span></span></h4>/n<p>Condition handling moreover provides a <a href="/wiki/Separation_of_mechanism_from_policy" class="mw-redirect" title="Separation of mechanism from policy">separation of mechanism from policy</a>. Restarts provide various possible mechanisms for recovering from error, but do not select which mechanism is appropriate in a given situation. That is the province of the condition handler, which (since it is located in higher-level code) has access to a broader view.</p>/n<p>An example: Suppose there is a library function whose purpose is to parse a single <a href="/wiki/Syslog" title="Syslog">syslog</a> file entry. What should this function do if the entry is malformed? There is no one right answer, because the same library could be deployed in programs for many different purposes. In an interactive log-file browser, the right thing to do might be to return the entry unparsed, so the user can see it—but in an automated log-summarizing program, the right thing to do might be to supply null values for the unreadable fields, but abort with an error, if too many entries have been malformed.</p>/n<p>That is to say, the question can only be answered in terms of the broader goals of the program, which are not known to the general-purpose library function. Nonetheless, exiting with an error message is only rarely the right answer. So instead of simply exiting with an error, the function may <i>establish restarts</i> offering various ways to continue—for instance, to skip the log entry, to supply default or null values for the unreadable fields, to ask the user for the missing values, <i>or</i> to unwind the stack and abort processing with an error message. The restarts offered constitute the <i>mechanisms</i> available for recovering from error; the selection of restart by the condition handler supplies the <i><a href="/wiki/Policy" title="Policy">policy</a></i>.</p>/n<h2><span class="mw-headline" id="See_also">See also</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a
