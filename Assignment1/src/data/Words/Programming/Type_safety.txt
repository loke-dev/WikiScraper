left;">/n<ul>/n<li><a href="/wiki/Strong_and_weak_typing" title="Strong and weak typing">Strong and weak typing</a></li>/n</ul>/n</td>/n</tr>/n</table>/n<p>In <a href="/wiki/Computer_science" title="Computer science">computer science</a>, <b>type safety</b> is the extent to which a <a href="/wiki/Programming_language" title="Programming language">programming language</a> discourages or prevents <b>type errors</b>. A type error is erroneous or undesirable program behaviour caused by a discrepancy between differing <a href="/wiki/Data_type" title="Data type">data types</a> for the program's constants, variables, and methods (functions), e.g., treating an integer (<b>int</b>) as a floating-point number (<b>float</b>). Type safety is sometimes alternatively considered to be a property of a computer program rather than the language in which that program is written; that is, some languages have type-safe facilities that can be circumvented by programmers who adopt practices that exhibit poor type safety. The formal <a href="/wiki/Type_theory" title="Type theory">type-theoretic</a> definition of type safety is considerably stronger than what is understood by most programmers.</p>/n<p><a href="/wiki/Type_enforcement" title="Type enforcement">Type enforcement</a> can be static, catching potential errors at <a href="/wiki/Compile_time" title="Compile time">compile time</a>, or dynamic, associating type information with values at <a href="/wiki/Run_time_(program_lifecycle_phase)" title="Run time (program lifecycle phase)">run-time</a> and consulting them as needed to detect imminent errors, or a combination of both.</p>/n<p>The behaviors classified as type errors by a given programming language are usually those that result from attempts to perform operations on <a href="/wiki/Value_(computer_science)" title="Value (computer science)">values</a> that are not of the appropriate <a href="/wiki/Data_type" title="Data type">data type</a>. This classification is partly based on opinion; it may imply that any operation not leading to program crashes, security flaws or other obvious failures is legitimate and need not be considered an error, or it may imply that any contravention of the programmer's explicit intent (as communicated via typing annotations) to be erroneous and not "type-safe".</p>/n<p>In the context of static (compile-time) type systems, type safety usually involves (among other things) a guarantee that the eventual value of any <a href="/wiki/Expression_(programming)" class="mw-redirect" title="Expression (programming)">expression</a> will be a legitimate member of that expression's static type. The precise requirement is more subtle than this — see, for example, <a href="/wiki/Subtype" class="mw-redirect" title="Subtype">subtype</a> and <a href="/wiki/Polymorphism_(computer_science)" title="Polymorphism (computer science)">polymorphism</a> for complications.</p>/n<p>Type safety is closely linked to <i><a href="/wiki/Memory_safety" title="Memory safety">memory safety</a></i>, a restriction on the ability to copy arbitrary bit patterns from one memory location to another. Conversely, if the language is type-unsafe to the extent of allowing an arbitrary integer to be used as a <a href="/wiki/Pointer_(computer_programming)" title="Pointer (computer programming)">pointer</a>, then it is not memory-safe.</p>/n<p>Most statically typed languages provide a degree of type safety that is strictly stronger than memory safety, because their type systems enforce the proper use of <a href="/wiki/Abstract_data_type" title="Abstract data type">abstract data types</a> defined by programmers even when this is not strictly necessary for memory safety or for the prevention of any kind of catastrophic failure.</p>/n<p></p>/n<div id="toc" class="toc">/n<div class="toctitle">/n<h2>Contents</h2>/n</div>/n<ul>/n<li class="toclevel-1 tocsection-1"><a class="toctext">References</span></a></li>/n</ul>/n</div>/n<p></p>/n<h2><span class="mw-headline" id="Definitions">Definitions</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Type_safety&amp;action=edit&amp;section=1" title="Edit section: Definitions">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Type-safe code accesses only the memory locations it is authorized to access. (For this discussion, type safety specifically refers to memory type safety and should not be confused with type safety in a broader respect.) For example, type-safe code cannot read values from another object's private fields.</p>/n<p><a href="/wiki/Robin_Milner" title="Robin Milner">Robin Milner</a> provided the following slogan to describe type safety:</p>/n<dl>/n<dd>Well-typed programs cannot "go wrong".<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">[1]</a></sup></dd>/n</dl>/n<p>The appropriate formalization of this slogan depends on the style of formal semantics used for a particular language. In the context of <a href="/wiki/Denotational_semantics" title="Denotational semantics">denotational semantics</a>, type safety means that the value of an expression that is well-typed, say with type τ, is a <i>bona fide</i> member of the set corresponding to τ.</p>/n<p>In 1994, Andrew Wright and <a href="/wiki/Matthias_Felleisen" title="Matthias Felleisen">Matthias Felleisen</a> formulated what is now the standard definition and proof technique for type safety in languages defined by <a href="/wiki/Operational_semantics" title="Operational semantics">operational semantics</a>. Under this approach, type safety is determined by two properties of the semantics of the programming language:</p>/n<dl>/n<dt>(Type-) preservation or <a href="/wiki/Subject_reduction" title="Subject reduction">subject reduction</a></dt>/n<dd>"Well typedness" ("typability") of programs remains invariant under the transition rules (i.e. evaluation rules or reduction rules) of the language.</dd>/n<dt>Progress</dt>/n<dd>A well typed (typable) program never gets "stuck", which means the <a href="/wiki/Expression_(computer_science)" title="Expression (computer science)">expressions</a> in the program will either be evaluated to a <a href="/wiki/Value_(computer_science)" title="Value (computer science)">value</a>, or there is a transition rule for it; in other words, the program never gets into an undefined state where no further transitions are possible.</dd>/n</dl>/n<p>These properties do not exist in a vacuum; they are linked to the semantics of the programming language they describe, and there is a large space of varied languages that can fit these criteria, since the notion of "well typed" program is part of the static semantics of the programming language and the notion of "getting stuck" (or "going wrong") is a property of its <a href="/wiki/Dynamic_semantics" class="mw-redirect" title="Dynamic semantics">dynamic semantics</a>.</p>/n<p>Vijay Saraswat provides the following definition:</p>/n<dl>/n<dd>"A language is type-safe if the only operations that can be performed on data in the language are those sanctioned by the type of the data." <sup id="cite_ref-saraswat-java_2-0" class="reference"><a href="#cite_note-saraswat-java-2">[2]</a></sup></dd>/n</dl>/n<h2><span class="mw-headline" id="Relation_to_other_forms_of_safety">Relation to other forms of safety</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Type_safety&amp;action=edit&amp;section=2" title="Edit section: Relation to other forms of safety">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Type safety is ultimately aimed at excluding other problems, e.g.:-</p>/n<ul>/n<li>Prevention of illegal operations. For example, we can identify an expression <code>3 / "Hello, World"</code> as invalid, because the rules of <a href="/wiki/Arithmetic" title="Arithmetic">arithmetic</a> do not specify how to divide an <a href="/wiki/Integer" title="Integer">integer</a> by a <a href="/wiki/String_(computer_science)" title="String (computer science)">string</a>.</li>/n<li><a href="/wiki/Memory_safety" title="Memory safety">Memory safety</a>/n<ul>/n<li><a href="/wiki/Wild_pointer" class="mw-redirect" title="Wild pointer">Wild pointers</a> can arise when a pointer to one type object is treated as a pointer to another type. For instance, the size of an object depends on the type, so if a pointer is incremented under the wrong credentials, it will end up pointing at some random area of memory.</li>/n<li><a href="/wiki/Buffer_overflow" title="Buffer overflow">Buffer overflow</a> - Out-of bound writes can corrupt the contents of objects already present on the heap. This can occur when a larger object of one type is crudely copied into smaller object of another type.</li>/n</ul>/n</li>/n<li><a href="/wiki/Logic_error" title="Logic error">Logic errors</a> originating in the <a href="/wiki/Semantics" title="Semantics">semantics</a> of different types. For instance, inches and millimeters may both be stored as integers, but should not be substituted for each other or added. A type system can enforce two different types of integer for them.</li>/n</ul>/n<h2><span class="mw-headline" id="Type-safe_and_type-unsafe_languages">Type-safe and type-unsafe languages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Type_safety&amp;action=edit&amp;section=3" title="Edit section: Type-safe and type-unsafe languages">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Type safety is usually a requirement for any <a href="/wiki/Toy_language" class="mw-redirect" title="Toy language">toy language</a> proposed in academic programming language research. Many languages, on the other hand, are too big for human-generated type safety proofs, as they often require checking thousands of cases. Nevertheless, some languages such as <a href="/wiki/Standard_ML" title="Standard ML">Standard ML</a>, which has rigorously defined semantics, have been proved to meet one definition of type safety.<sup id="cite_ref-3" class="reference"><a href="#cite_note-3">[3]</a></sup> Some other languages such as <a href="/wiki/Haskell_(programming_language)" title="Haskell (programming language)">Haskell</a> are <i>believed</i> to meet some definition of type safety, provided certain "escape" features are not used (for example Haskell's <tt>unsafePerformIO</tt>, used to "escape" from the usual restricted environment in which I/O is possible, circumvents the type system and so can be used to break type safety.<sup id="cite_ref-4" class="reference"><a href="#cite_note-4">[4]</a></sup>) <a href="/wiki/Type_punning" title="Type punning">Type punning</a> is another example of such an "escape" feature. Regardless of the properties of the language definition, certain errors may occur at <a href="/wiki/Run_time_(program_lifecycle_phase)" title="Run time (program lifecycle phase)">run-time</a> due to bugs in the implementation, or in linked <a href="/wiki/Library_(computer_science)" class="mw-redirect" title="Library (computer science)">libraries</a> written in other languages; such errors could render a given implementation type unsafe in certain circumstances. An early version of Sun's <a href="/wiki/Java_virtual_machine" title="Java virtual machine">Java virtual machine</a> was vulnerable to this sort of problem.<sup id="cite_ref-saraswat-java_2-1" class="reference"><a href="#cite_note-saraswat-java-2">[2]</a></sup></p>/n<h2><span class="mw-headline" id="Strong_and_weak_typing">Strong and weak typing</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Type_safety&amp;action=edit&amp;section=4" title="Edit section: Strong and weak typing">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Strong_and_weak_typing" title="Strong and weak typing">Strong and weak typing</a></div>/n<p>Programming languages are often colloquially classified as strongly typed or weakly typed (also loosely typed) to refer to certain aspects of type safety. In 1974, <a href="/wiki/Barbara_Liskov" title="Barbara Liskov">Liskov</a> and Zilles defined a strongly-typed language as one in which "whenever an object is passed from a calling function to a called function, its type must be compatible with the type declared in the called function."<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">[5]</a></sup> In 1977, Jackson wrote, "In a strongly typed language each data area will have a distinct type and each process will state its communication requirements in terms of these types."<sup id="cite_ref-6" class="reference"><a href="#cite_note-6">[6]</a></sup> In contrast, a weakly typed language may produce unpredictable results or may perform implicit type conversion.<sup id="cite_ref-7" class="reference"><a href="#cite_note-7">[7]</a></sup></p>/n<h2><span class="mw-headline" id="Type_safety_in_object_oriented_languages">Type safety in object oriented languages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Type_safety&amp;action=edit&amp;section=5" title="Edit section: Type safety in object oriented languages">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>In <a href="/wiki/Object_oriented" class="mw-redirect" title="Object oriented">object oriented</a> languages type safety is usually intrinsic in the fact that a <a href="/wiki/Type_system" title="Type system">type system</a> is in place. This is expressed in terms of class definitions.</p>/n<p>A <a href="/wiki/Class_(computer_science)" class="mw-redirect" title="Class (computer science)">class</a> essentially defines the structure of the objects derived from it and an <a href="/wiki/Api#API_in_object-oriented_languages" class="mw-redirect" title="Api">API</a> as a <i>contract</i> for handling these objects. Each time a new object is created it will <i>comply</i> with that contract.</p>/n<p>Each function that exchanges objects derived from a specific class, or implementing a specific <a href="/wiki/Interface_(computer_science)" class="mw-redirect" title="Interface (computer science)">interface</a>, will adhere to that contract: hence in that function the operations permitted on that object will be only those defined by the methods of the class the object implements. This will guarantee that the object integrity will be preserved.<sup id="cite_ref-8" class="reference"><a href="#cite_note-8">[8]</a></sup></p>/n<p>Exception to this are object oriented languages that allow dynamic modification of the object structure, or the use of reflection to modify the content of an object to overcome the constraints imposed by the class methods definitions.</p>/n<h2><span class="mw-headline" id="Type_safety_issues_in_specific_languages">Type safety issues in specific languages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Type_safety&amp;action=edit&amp;section=6" title="Edit section: Type safety issues in specific languages">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<h3><span class="mw-headline" id="Ada">Ada</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Type_safety&amp;action=edit&amp;section=7" title="Edit section: Ada">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<table role="presentation" class="mbox-small plainlinks sistersitebox" style="background-color:#f9f9f9;border:1px solid #aaa;color:#000">/n<tr>/n<td class="mbox-image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/d/df/Wikibooks-logo-en-noslogan.svg/40px-Wikibooks-logo-en-noslogan.svg.png" width="40" height="40" class="noviewer" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/d/df/Wikibooks-logo-en-noslogan.svg/60px-Wikibooks-logo-en-noslogan.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/d/df/Wikibooks-logo-en-noslogan.svg/80px-Wikibooks-logo-en-noslogan.svg.png 2x" data-file-width="400" data-file-height="400" /></td>/n<td class="mbox-text plainlist">The Wikibook <i><a href="https://en.wikibooks.org/wiki/Ada_Programming" class="extiw" title="wikibooks:Ada Programming">Ada Programming</a></i> has a page on the topic of: <i><b><a href="https://en.wikibooks.org/wiki/Ada_Programming/Type_System" class="extiw" title="wikibooks:Ada Programming/Type System">Type System</a></b></i></td>/n</tr>/n</table>/n<p><a href="/wiki/Ada_programming_language" class="mw-redirect" title="Ada programming language">Ada</a> was designed to be suitable for <a href="/wiki/Embedded_system" title="Embedded system">embedded systems</a>, <a href="/wiki/Device_driver" title="Device driver">device drivers</a> and other forms of <a href="/wiki/System_programming" title="System programming">system programming</a>, but also to encourage type-safe programming. To resolve these conflicting goals, Ada confines type-unsafety to a certain set of special constructs whose names usually begin with the string <tt>Unchecked_</tt>. Unchecked_Deallocation can be effectively banned from a unit of Ada text by applying <tt>pragma Pure</tt> to this unit. It is expected that programmers will use <tt>Unchecked_</tt> constructs very carefully and only when necessary; programs that do not use them are type-safe.</p>/n<p>The <a href="/wiki/SPARK_programming_language" class="mw-redirect" title="SPARK programming language">SPARK programming language</a> is a subset of Ada eliminating all its potential ambiguities and insecurities while at the same time adding <a href="/wiki/Design_by_contract" title="Design by contract">statically checked contracts</a> to the language features available. SPARK avoids the issues with <a href="/wiki/Dangling_pointer" title="Dangling pointer">dangling pointers</a> by disallowing allocation at run time entirely.</p>/n<p>Ada2012 adds <a href="/wiki/Design_by_contract" title="Design by contract">statically checked contracts</a> to the language itself (in form of pre-, and post-conditions, as well as type invariants).</p>/n<h3><span class="mw-headline" id="C">C</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Type_safety&amp;action=edit&amp;section=8" title="Edit section: C">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<table role="presentation" class="mbox-small plainlinks sistersitebox" style="background-color:#f9f9f9;border:1px solid #aaa;color:#000">/n<tr>/n<td class="mbox-image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/d/df/Wikibooks-logo-en-noslogan.svg/40px-Wikibooks-logo-en-noslogan.svg.png" width="40" height="40" class="noviewer" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/d/df/Wikibooks-logo-en-noslogan.svg/60px-Wikibooks-logo-en-noslogan.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/d/df/Wikibooks-logo-en-noslogan.svg/80px-Wikibooks-logo-en-noslogan.svg.png 2x" data-file-width="400" data-file-height="400" /></td>/n<td class="mbox-text plainlist">Wikibooks has a book on the topic of: <i><b><a href="https://en.wikibooks.org/wiki/C_Programming" class="extiw" title="wikibooks:C Programming">C Programming</a></b></i></td>/n</tr>/n</table>/n<p>The <a href="/wiki/C_(programming_language)" title="C (programming language)">C programming language</a> is typesafe in limited contexts; for example, a compile-time error is generated when an attempt is made to convert a pointer to one type of structure to a pointer to another type of structure, unless an explicit cast is used. However, a number of very common operations are non-typesafe; for example, the usual way to print an integer is something like <code>printf("%d", 12)</code>, where the <code>%d</code> tells <code>printf</code> at run-time to expect an integer argument. (Something like <code>printf("%s", 12)</code>, which erroneously tells the function to expect a pointer to a character-string, may be accepted by compilers, but will produce undefined results.) This is partially mitigated by some compilers (such as gcc) checking type correspondences between printf arguments and format strings.</p>/n<p>In addition, C, like Ada, provides unspecified or undefined explicit conversions; and unlike in Ada, idioms that use these conversions are very common, and have helped to give C a type-unsafe reputation. For example, the standard way to allocate memory on the heap is to invoke a memory allocation function, such as <code><a href="/wiki/Malloc" class="mw-redirect" title="Malloc">malloc</a></code>, with an argument indicating how many bytes are required. The function returns an untyped pointer (type <code>void *</code>), which the calling code must explicitly or implicitly cast to the appropriate pointer type. Pre-standardized implementations of C required an explicit cast to do so, therefore the code <code>(struct foo *) malloc(sizeof(struct foo))</code> became the accepted practice.<sup id="cite_ref-knr_9-0" class="reference"><a href="#cite_note-knr-9">[9]</a></sup> However, this practice is discouraged in ISO C as it can mask a failure to include the header file in which <code>malloc</code> is defined, resulting in downstream errors on machines where the int and pointer types are of different sizes, such as most common implementations of C for the now-ubiquitous <a href="/wiki/X86_64" class="mw-redirect" title="X86 64">x86 64</a> architecture.<sup id="cite_ref-stackdiscussion_10-0" class="reference"><a href="#cite_note-stackdiscussion-10">[10]</a></sup> A conflict arises in code that is required to compile as C++, since the cast is necessary in that language.</p>/n<h3><span id="C++"></span><span class="mw-headline" id="C.2B.2B">C++</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Type_safety&amp;action=edit&amp;section=9" title="Edit section: C++">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>Some features of C++ that promote more type-safe code:</p>/n<ul>/n<li>The <a href="/wiki/New_(C%2B%2B)" class="mw-redirect" title="New (C++)">new</a> operator returns a pointer of type based on operand, whereas <a href="/wiki/Malloc" class="mw-redirect" title="Malloc">malloc</a> returns a void pointer.</li>/n<li>C++ code can use virtual functions and <a href="/wiki/Template_(programming)" class="mw-redirect" title="Template (programming)">templates</a> to achieve polymorphism without void pointers.</li>/n<li><a href="/wiki/C_macro" class="mw-redirect" title="C macro">Preprocessor</a> constants (without type) can be rewritten as <a href="/wiki/Constant_(programming)" class="mw-redirect" title="Constant (programming)">const</a> variables (typed).</li>/n<li><a href="/wiki/C_macro" class="mw-redirect" title="C macro">Preprocessor</a> macro functions (without type) can be rewritten as <a href="/wiki/Inline_function" title="Inline function">inline</a> functions (typed). The flexibility of accepting and returning different types can still be obtained by function overloading.</li>/n<li>Safer casting operators, such as <a href="/wiki/Dynamic_cast" class="mw-redirect" title="Dynamic cast">dynamic cast</a> that performs run-time type checking.</li>/n</ul>/n<h3><span id="C#"></span><span class="mw-headline" id="C.23">C#</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Type_safety&amp;action=edit&amp;section=10" title="Edit section: C#">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p><a href="/wiki/C_Sharp_(programming_language)" title="C Sharp (programming language)">C#</a> is type-safe (but not statically type-safe). It has support for untyped pointers, but this must be accessed using the "unsafe" keyword which can be prohibited at the compiler level. It has inherent support for run-time cast validation. Casts can be validated by using the "as" keyword that will return a null reference if the cast is invalid, or by using a C-style cast that will throw an exception if the cast is invalid. See <a href="/wiki/C_Sharp_Syntax#Conversion_operators" class="mw-redirect" title="C Sharp Syntax">C Sharp conversion operators</a>.</p>/n<p>Undue reliance on the <a href="/wiki/C_Sharp_syntax#object_class" title="C Sharp syntax">object</a> type (from which all other types are derived) runs the risk of defeating the purpose of the C# type system. It is usually better practice to abandon object references in favour of <a href="/wiki/Generic_programming#Generic_programming_in_.NET" title="Generic programming">generics</a>, similar to templates in C++ and <a href="/wiki/Generics_in_Java" title="Generics in Java">generics in Java</a>.</p>/n<h3><span class="mw-headline" id="Java">Java</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Type_safety&amp;action=edit&amp;section=11" title="Edit section: Java">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<table role="presentation" class="mbox-small plainlinks sistersitebox" style="background-color:#f9f9f9;border:1px solid #aaa;color:#000">/n<tr>/n<td class="mbox-image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/d/df/Wikibooks-logo-en-noslogan.svg/40px-Wikibooks-logo-en-noslogan.svg.png" width="40" height="40" class="noviewer" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/d/df/Wikibooks-logo-en-noslogan.svg/60px-Wikibooks-logo-en-noslogan.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/d/df/Wikibooks-logo-en-noslogan.svg/80px-Wikibooks-logo-en-noslogan.svg.png 2x" data-file-width="400" data-file-height="400" /></td>/n<td class="mbox-text plainlist">Wikibooks has a book on the topic of: <i><b><a href="https://en.wikibooks.org/wiki/Java_Programming" class="extiw" title="wikibooks:Java Programming">Java Programming</a></b></i></td>/n</tr>/n</table>/n<p>The <a href="/wiki/Java_language" class="mw-redirect" title="Java language">Java language</a> is designed to enforce type safety. Anything in Java <i>happens</i> inside an <a href="/wiki/Object_(computer_science)" title="Object (computer science)">object</a> and each object is an instance of a <a href="/wiki/Class_(computer_programming)" title="Class (computer programming)">class</a>.</p>/n<p>To implement the <i>type safety</i> enforcement, each object, before usage, needs to be <a href="/wiki/Memory_allocation" class="mw-redirect" title="Memory allocation">allocated</a>. Java allows usage of <a href="/wiki/Primitive_data_type" title="Primitive data type">primitive types</a> but only inside properly allocated objects.</p>/n<p>Sometimes a part of the type safety is implemented indirectly: e.g. the class BigDecimal represents a floating point number of arbitrary precision, but handles only numbers that can be expressed with a finite representation. The operation BigDecimal.divide() calculates a new object as the division of two numbers expressed as BigDecimal.</p>/n<p>In this case if the division has no finite representation, as when one computes e.g. 1/3=0.33333..., the divide() method can raise an exception if no rounding mode is defined for the operation. Hence the library, rather than the language, guarantees that the object respects the contract implicit in the class definition.</p>/n<h3><span class="mw-headline" id="Standard_ML">Standard ML</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Type_safety&amp;action=edit&amp;section=12" title="Edit section: Standard ML">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<table role="presentation" class="mbox-small plainlinks sistersitebox" style="background-color:#f9f9f9;border:1px solid #aaa;color:#000">/n<tr>/n<td class="mbox-image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/d/df/Wikibooks-logo-en-noslogan.svg/40px-Wikibooks-logo-en-noslogan.svg.png" width="40" height="40" class="noviewer" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/d/df/Wikibooks-logo-en-noslogan.svg/60px-Wikibooks-logo-en-noslogan.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/d/df/Wikibooks-logo-en-noslogan.svg/80px-Wikibooks-logo-en-noslogan.svg.png 2x" data-file-width="400" data-file-height="400" /></td>/n<td class="mbox-text plainlist">Wikibooks has a book on the topic of: <i><b><a href="https://en.wikibooks.org/wiki/Standard_ML_Programming" class="extiw" title="wikibooks:Standard ML Programming">Standard ML Programming</a></b></i></td>/n</tr>/n</table>/n<p><a href="/wiki/Standard_ML" title="Standard ML">SML</a> has rigorously defined semantics and is known to be type-safe. However, some implementations of SML, including <a href="/wiki/Standard_ML_of_New_Jersey" title="Standard ML of New Jersey">Standard ML of New Jersey</a> (SML/NJ), its syntactic variant <a href="/w/index.php?title=Mythryl&amp;action=edit&amp;redlink=1" class="new" title="Mythryl (page does not exist)">Mythryl</a> and <a href="/wiki/Mlton" class="mw-redirect" title="Mlton">Mlton</a>, provide libraries that offer certain unsafe operations. These facilities are often used in conjunction with those implementations' <a href="/wiki/Foreign_function_interface" title="Foreign function interface">foreign function interfaces</a> to interact with non-ML code (such as C libraries) that may require data laid out in specific ways. Another example is the SML/NJ <a href="/wiki/Read-eval-print_loop" class="mw-redirect" title="Read-eval-print loop">interactive toplevel</a> itself, which must use unsafe operations to execute ML code entered by the user.</p>/n<h3><span class="mw-headline" id="Modula-2">Modula-2</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Type_safety&amp;action=edit&amp;section=13" title="Edit section: Modula-2">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>Modula-2 is a strongly typed language with a design philosophy to require any unsafe facilities to be explicitly marked as unsafe. This is achieved by "moving" such facilities into a built-in pseudo-library called SYSTEM from where they must be imported before they can be used. The import thus makes it visible when such facilities are used. class="mbox-small plainlinks sistersitebox" style="background-color:#f9f9f9;border:1px solid #aaa;color:#000">/n<tr>/n<td class="mbox-image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/d/df/Wikibooks-logo-en-noslogan.svg/40px-Wikibooks-logo-en-noslogan.svg.png" width="40" height="40" class="noviewer" class="mbox-text plainlist">Wikibooks has a book on the topic of: <i><b><a href="https://en.wikibooks.org/wiki/Pascal_Programming" class="extiw" title="wikibooks:Pascal Programming">Pascal Programming</a></b></i></td>/n</tr>/n</table>/n<p><a href="/wiki/Pascal_(programming_language)" title="Pascal (programming language)">Pascal</a> has had a number of type safety requirements, some of which are kept in some compilers. Where a Pascal compiler dictates "strict typing", two variables cannot be assigned to each other unless they are either compatible (such as conversion of integer to real) or assigned to the identical subtype. For example, if you have the following code fragment:</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n<span class="k">type</span>/n  <span class="n">TwoTypes</span> <span class="o">=</span> <span class="k">record</span>/n     <span class="n">I</span><span class="o">:</span> <span class="kt">Integer</span><span class="o">;</span>/n     <span class="n">Q</span><span class="o">:</span> <span class="kt">Real</span><span class="o">;</span>/n  <span class="k">end</span><span class="o">;</span>/n/n  <span class="n">DualTypes</span> <span class="o">=</span> <span class="k">record</span>/n    <span class="n">I</span><span class="o">:</span> <span class="kt">Integer</span><span class="o">;</span>/n    <span class="n">Q</span><span class="o">:</span> <span class="kt">Real</span><span class="o">;</span>/n  <span class="k">end</span><span class="o">;</span>/n/n<span class="k">var</span>/n  <span class="n">T1</span><span class="o">,</span> <span class="n">T2</span><span class="o">:</span>  <span class="n">TwoTypes</span><span class="o">;</span>/n  <span class="n">D1</span><span class="o">,</span> <span class="n">D2</span><span class="o">:</span>  <span class="n">DualTypes</span><span class="o">;</span>/n</pre></div>/n<p>Under strict typing, a variable defined as <tt>TwoTypes</tt> is <i>not compatible</i> with <tt>DualTypes</tt> (because they are not identical, even though the components of that user defined type are identical) and an assignment of <tt>T1&#160;:= D2;</tt> is illegal. An assignment of <tt>T1&#160;:= T2;</tt> would be legal because the subtypes they are defined to <i>are</i> identical. However, an assignment such as <tt>T1.Q&#160;:= D1.Q;</tt> would be legal.</p>/n<h3><span class="mw-headline" id="Common_Lisp">Common Lisp</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Type_safety&amp;action=edit&amp;section=15" title="Edit section: Common Lisp">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>In general, <a href="/wiki/Common_Lisp" title="Common Lisp">Common Lisp</a> is a type-safe language. A Common Lisp compiler is responsible for inserting dynamic checks for operations whose type safety cannot be proven statically. However, a programmer may indicate that a program should be compiled with a lower level of dynamic type-checking.<sup id="cite_ref-14" class="reference"><a href="#cite_note-14">[14]</a></sup> A program compiled in such a mode cannot be considered type-safe.</p>/n<h2><span id="C++_Examples"></span><span class="mw-headline" id="C.2B.2B_Examples">C++ Examples</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Type_safety&amp;action=edit&amp;section=16" title="Edit section: C++ Examples">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>The following examples illustrates how C++ cast operators can break type safety when used incorrectly. The first example shows how basic data types can be incorrectly casted:</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n<span class="cp">#include</span> <span class="cpf">&lt;iostream&gt;</span>/n<span class="k">using</span> <span class="k">namespace</span> <span class="n">std</span><span class="p">;</span>/n/n<span class="kt">int</span> <span class="nf">main</span> <span class="p">()</span> <span class="p">{</span>/n    <span class="kt">int</span>   <span class="n">ival</span> <span class="o">=</span> <span class="mi">5</span><span class="p">;</span>                              <span class="c1">// integer value</span>/n    <span class="kt">float</span> <span class="n">fval</span> <span class="o">=</span> <span class="k">reinterpret_cast</span><span class="o">&lt;</span><span class="kt">float</span><span class="o">&amp;&gt;</span><span class="p">(</span><span class="n">ival</span><span class="p">);</span> <span class="c1">// reinterpret bit pattern</span>/n    <span class="n">cout</span> <span class="o">&lt;&lt;</span> <span class="n">fval</span> <span class="o">&lt;&lt;</span> <span class="n">endl</span><span class="p">;</span>                        <span class="c1">// output integer as float</span>/n    <span class="k">return</span> <span class="mi">0</span><span class="p">;</span>/n<span class="p">}</span>/n</pre></div>/n<p>In this example, <code>reinterpret_cast</code> explicitly prevents the compiler from performing a safe conversion from integer to floating-point value.<sup id="cite_ref-15" class="reference"><a href="#cite_note-15">[15]</a></sup> When the program runs it will output a garbage floating-point value. The problem could have been avoided by instead writing <code>float fval = ival;</code></p>/n<p>The next example shows how object references can be incorrectly downcasted:</p>/n<div class="mw-highlight mw-content-ltr" dir="ltr">/n<pre>/n<span class="cp">#include</span> <span class="cpf">&lt;iostream&gt;</span>/n<span class="k">using</span> <span class="k">namespace</span> <span class="n">std</span><span class="p">;</span>/n/n<span class="k">class</span> <span class="nc">Parent</span> <span class="p">{</span>/n<span class="k">public</span><span class="o">:</span>/n    <span class="k">virtual</span> <span class="o">~</span><span class="n">Parent</span><span class="p">()</span> <span class="p">{}</span> <span class="c1">// virtual destructor for RTTI</span>/n<span class="p">};</span>/n/n<span class="k">class</span> <span class="nc">Child1</span> <span class="o">:</span> <span class="k">public</span> <span class="n">Parent</span> <span class="p">{</span>/n<span class="k">public</span><span class="o">:</span>/n    <span class="kt">int</span> <span class="n">a</span><span class="p">;</span>/n<span class="p">};</span>/n/n<span class="k">class</span> <span class="nc">Child2</span> <span class="o">:</span> <span class="k">public</span> <span class="n">Parent</span> <span class="p">{</span>/n<span class="k">public</span><span class="o">:</span>/n    <span class="kt">float</span> <span class="n">b</span><span class="p">;</span>/n<span class="p">};</span>/n/n<span class="kt">int</span> <span class="nf">main</span> <span class="p">()</span> <span class="p">{</span>/n    <span class="n">Child1</span> <span class="n">c1</span><span class="p">;</span>/n    <span class="n">c1</span><span class="p">.</span><span class="n">a</span> <span class="o">=</span> <span class="mi">5</span><span class="p">;</span>/n    <span class="n">Parent</span> <span class="o">&amp;</span> <span class="n">p</span> <span class="o">=</span> <span class="n">c1</span><span class="p">;</span>                     <span class="c1">// upcast always safe</span>/n    <span class="n">Child2</span> <span class="o">&amp;</span> <span class="n">c2</span> <span class="o">=</span> <span class="k">static_cast</span><span class="o">&lt;</span><span class="n">Child2</span><span class="o">&amp;&gt;</span><span class="p">(</span><span class="n">p</span><span class="p">);</span> <span class="c1">// invalid downcast</span>/n    <span class="n">cout</span> <span class="o">&lt;&lt;</span> <span class="n">c2</span><span class="p">.</span><span class="n">b</span> <span class="o">&lt;&lt;</span> <span class="n">endl</span><span class="p">;</span>          <span class="c1">// will output garbage data</span>/n    <span class="k">return</span> <span class="mi">0</span><span class="p">;</span>/n<span class="p">}</span>/n</pre></div>/n<p>The two child classes have members of different types. When downcasting a parent class pointer to a child class pointer, then the resulting pointer may not point to a valid object of correct type. In the example, this leads to garbage value being printed. The problem could have been avoided by replacing <code>static_cast</code> with <code>dynamic_cast</code> that throws an exception on invalid casts.<sup id="cite_ref-16" class="reference"><a
