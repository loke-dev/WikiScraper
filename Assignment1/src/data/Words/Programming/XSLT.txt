style="text-align:center"><a href="/wiki/DSSSL" class="mw-redirect" title="DSSSL">DSSSL</a></td>/n</tr>/n</table>/n<p><b>XSLT</b> (<b>Extensible Stylesheet Language Transformations</b>) is a language for <a href="/wiki/XML_transformation_language" title="XML transformation language">transforming</a> <a href="/wiki/XML" title="XML">XML</a> documents into other XML documents,<sup id="cite_ref-Transformation_1-0" class="reference"><a href="#cite_note-Transformation-1">[1]</a></sup> or other formats such as <a href="/wiki/HTML" title="HTML">HTML</a> for <a href="/wiki/Web_page" title="Web page">web pages</a>, <a href="/wiki/Plain_text" title="Plain text">plain text</a> or <a href="/wiki/XSL_Formatting_Objects" title="XSL Formatting Objects">XSL Formatting Objects</a>, which may subsequently be converted to other formats, such as <a href="/wiki/Portable_Document_Format" title="Portable Document Format">PDF</a>, <a href="/wiki/PostScript" title="PostScript">PostScript</a> and <a href="/wiki/Portable_Network_Graphic" class="mw-redirect" title="Portable Network Graphic">PNG</a>.<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">[2]</a></sup> XSLT 1.0 is widely supported in modern web browsers.</p>/n<p>The original document is not changed; rather, a new document is created based on the content of an existing one.<sup id="cite_ref-3" class="reference"><a href="#cite_note-3">[3]</a></sup> Typically, input documents are XML files, but anything from which the processor can build an <a href="/wiki/XQuery_and_XPath_Data_Model" title="XQuery and XPath Data Model">XQuery and XPath Data Model</a> can be used, such as <a href="/wiki/Relational_database" title="Relational database">relational database</a> tables or <a href="/wiki/Geographical_information_systems" class="mw-redirect" title="Geographical information systems">geographical information systems</a>.<sup id="cite_ref-Transformation_1-1" class="reference"><a href="#cite_note-Transformation-1">[1]</a></sup></p>/n<p>Although XSLT is designed as a special-purpose language for XML transformation, the language is <a href="/wiki/Turing-complete" class="mw-redirect" title="Turing-complete">Turing-complete</a>, making it theoretically capable of arbitrary computations.</p>/n<p></p>/n<div id="toc" class="toc">/n<div links</span></a></li>/n</ul>/n</div>/n<p></p>/n<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=XSLT&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>XSLT is influenced by <a href="/wiki/Functional_programming" title="Functional programming">functional languages</a>,<sup id="cite_ref-4" class="reference"><a href="#cite_note-4">[4]</a></sup> and by text-based pattern matching languages like <a href="/wiki/SNOBOL" title="SNOBOL">SNOBOL</a> and <a href="/wiki/AWK_(programming_language)" class="mw-redirect" title="AWK (programming language)">AWK</a>. Its most direct predecessor is <a href="/wiki/DSSSL" class="mw-redirect" title="DSSSL">DSSSL</a>, which did for <a href="/wiki/SGML" class="mw-redirect" title="SGML">SGML</a> what XSLT does for XML.<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">[5]</a></sup></p>/n<ul>/n<li>XSLT 1.0: XSLT was part of the <a href="/wiki/World_Wide_Web_Consortium" title="World Wide Web Consortium">World Wide Web Consortium</a> (W3C)'s <a href="/wiki/Extensible_Stylesheet_Language" class="mw-redirect" title="Extensible Stylesheet Language">Extensible Stylesheet Language</a> (XSL) development effort of 1998–1999, a project that also produced <a href="/wiki/XSL-FO" class="mw-redirect" title="XSL-FO">XSL-FO</a> and <a href="/wiki/XPath" title="XPath">XPath</a>. Some members of the standards committee that developed XSLT, including <a href="/wiki/James_Clark_(XML_expert)" class="mw-redirect" title="James Clark (XML expert)">James Clark</a>, the editor, had previously worked on DSSSL. XSLT 1.0 was published as a <a href="/wiki/W3C_recommendation" class="mw-redirect" title="W3C recommendation">W3C recommendation</a> in November 1999.<sup id="cite_ref-timelinehistory_6-0" class="reference"><a href="#cite_note-timelinehistory-6">[6]</a></sup></li>/n<li>XSLT 2.0: after an abortive attempt to create a version 1.1 in 2001,<sup id="cite_ref-7" class="reference"><a href="#cite_note-7">[7]</a></sup> the XSL working group joined forces with the <a href="/wiki/XQuery" title="XQuery">XQuery</a> working group to create <a href="/wiki/XPath_2.0" title="XPath 2.0">XPath 2.0</a>,<sup id="cite_ref-8" class="reference"><a href="#cite_note-8">[8]</a></sup> with a richer data model and type system based on <a href="/wiki/XML_Schema_(W3C)" title="XML Schema (W3C)">XML Schema</a>. Building on this is XSLT 2.0,<sup id="cite_ref-9" class="reference"><a href="#cite_note-9">[9]</a></sup> developed under the editorship of <a href="/wiki/Michael_Howard_Kay" title="Michael Howard Kay">Michael Kay</a>, which reached recommendation status in January 2007.<sup id="cite_ref-10" class="reference"><a href="#cite_note-10">[10]</a></sup> As of 2010, however, XSLT 1.0<sup id="cite_ref-11" class="reference"><a href="#cite_note-11">[11]</a></sup> is still widely used, since 2.0 is not supported natively in <a href="/wiki/Web_browser" title="Web browser">web browsers</a> or for environments like <a href="/wiki/LAMP_(software_bundle)" title="LAMP (software bundle)">LAMP</a>.</li>/n<li>XSLT 3.0: became a W3C Recommendation on 8 June 2017. The main new features are:<sup id="cite_ref-new-in-3_12-0" class="reference"><a href="#cite_note-new-in-3-12">[12]</a></sup>/n<ul>/n<li><a href="/wiki/Streaming_XML" title="Streaming XML">Streaming transformations</a>: in previous versions the entire input document had to be read into memory before it could be processed,<sup id="cite_ref-13" class="reference"><a href="#cite_note-13">[13]</a></sup> and output could not be written until processing had finished (although Saxon does have a streaming extension). The working draft allows <a href="/wiki/Streaming_XML" title="Streaming XML">XML streaming</a> which will be useful for processing documents too large to fit in memory, or when transformations are chained in <a href="/wiki/XML_Pipelines" class="mw-redirect" title="XML Pipelines">XML Pipelines</a>.</li>/n<li>Improvements to the modularity of large stylesheets.</li>/n<li>Improved handling of dynamic errors with, for example, an xsl:try instruction.</li>/n<li>Functions can now be arguments to other (higher-order) functions.</li>/n</ul>/n</li>/n</ul>/n<h2><span class="mw-headline" id="Design_and_processing_model">Design and processing model</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=XSLT&amp;action=edit&amp;section=2" title="Edit section: Design and processing model">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<div class="thumb tright">/n<div class="thumbinner" style="width:222px;"><a href="/wiki/File:XSLT_en.svg" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/e/e6/XSLT_en.svg/220px-XSLT_en.svg.png" width="220" height="417" class="thumbimage" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/e/e6/XSLT_en.svg/330px-XSLT_en.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/e/e6/XSLT_en.svg/440px-XSLT_en.svg.png 2x" data-file-width="231" data-file-height="438" /></a>/n<div class="thumbcaption">/n<div class="magnify"><a href="/wiki/File:XSLT_en.svg" class="internal" title="Enlarge"></a></div>/nDiagram of the basic elements and process flow of Extensible Stylesheet Language Transformations.</div>/n</div>/n</div>/n<p>The XSLT processor takes one or more XML source documents, plus one or more XSLT stylesheets, and processes them to produce an output document. In contrast to widely implemented <a href="/wiki/Imperative_programming" title="Imperative programming">imperative</a> programming languages like <a href="/wiki/C_(programming_language)" title="C (programming language)">C</a>, XSLT is <a href="/wiki/Declarative_programming" title="Declarative programming">declarative</a>.<sup id="cite_ref-14" class="reference"><a href="#cite_note-14">[14]</a></sup> The basic processing paradigm is pattern matching.<sup id="cite_ref-IBM_15-0" class="reference"><a href="#cite_note-IBM-15">[15]</a></sup> Rather than listing an imperative sequence of actions to perform in a stateful environment, template rules only define how to handle a node matching a particular XPath-like pattern, if the processor should happen to encounter one, and the contents of the templates effectively comprise <a href="/wiki/Functional_programming" title="Functional programming">functional</a> <a href="/wiki/Expression_(programming)" class="mw-redirect" title="Expression (programming)">expressions</a> that directly represent their evaluated form: the result tree, which is the basis of the processor's output.</p>/n<p>The processor follows a fixed algorithm.<sup id="cite_ref-16" class="reference"><a href="#cite_note-16">[16]</a></sup> First, assuming a stylesheet has already been read and prepared, the processor builds a source <a href="/wiki/Tree_data_structure" class="mw-redirect" title="Tree data structure">tree</a> from the input XML document. It then processes the source tree's root node, finds the best-matching template for that node in the stylesheet, and evaluates the template's contents. Instructions in each template generally direct the processor to either create nodes in the result tree, or to process more nodes in the source tree in the same way as the root node. Output derives from the result tree.</p>/n<h2><span class="mw-headline" id="Processor_implementations">Processor implementations</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=XSLT&amp;action=edit&amp;section=3" title="Edit section: Processor implementations">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<ul>/n<li>Altova RaptorXML Server: cross-platform engine that supports XSLT 1.0 and 2.0, most of XPath 3.0, and some features from the XSLT 3.0 working draft; also XQuery. Allows command line operations and interfaces to COM, <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>, and <a href="/wiki/.NET_Framework" title=".NET Framework">.NET</a><sup id="cite_ref-17" class="reference"><a href="#cite_note-17">[17]</a></sup> and also includes a built-in HTTP server.</li>/n<li>Exselt:<sup id="cite_ref-18" class="reference"><a href="#cite_note-18">[18]</a></sup> A streaming XSLT 3.0 processor which runs on the <a href="/wiki/.NET_Framework" title=".NET Framework">.NET framework</a> written in <a href="/wiki/F_Sharp_(programming_language)" title="F Sharp (programming language)">F#</a>. Fully supports the XSLT 3.0 Draft, XPath 3.0 Recommendation and XDM 3.0 Recommendation standards.</li>/n<li><a href="/wiki/Libxslt" title="Libxslt">libxslt</a> is a <a href="/wiki/Free_software" title="Free software">free</a> <a href="/wiki/Library_(computing)" title="Library (computing)">library</a> released under the <a href="/wiki/MIT_License" title="MIT License">MIT License</a> that can be reused in commercial applications. It is based on <a href="/wiki/Libxml" class="mw-redirect" title="Libxml">libxml</a> and implemented in <a href="/wiki/C_(programming_language)" title="C (programming language)">C</a> for speed and portability. It supports XSLT 1.0 and <a href="/wiki/EXSLT" title="EXSLT">EXSLT</a> extensions.<sup id="cite_ref-19" class="reference"><a href="#cite_note-19">[19]</a></sup>/n<ul>/n<li>It can be used at the command line via xsltproc<sup id="cite_ref-20" class="reference"><a href="#cite_note-20">[20]</a></sup> which is included in <a href="/wiki/MacOS" title="MacOS">macOS</a><sup id="cite_ref-21" class="reference"><a href="#cite_note-21">[21]</a></sup> and many <a href="/wiki/Linux_distributions" class="mw-redirect" title="Linux distributions">Linux distributions</a>, and can be used on <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Windows</a> via <a href="/wiki/Cygwin" title="Cygwin">Cygwin</a>.<sup id="cite_ref-22" class="reference"><a href="#cite_note-22">[22]</a></sup></li>/n<li>The <a href="/wiki/WebKit" title="WebKit">WebKit</a> and <a href="/wiki/Blink_(layout_engine)" class="mw-redirect" title="Blink (layout engine)">Blink</a> layout engines, used for example in the <a href="/wiki/Apple_Safari" class="mw-redirect" title="Apple Safari">Safari</a> and <a href="/wiki/Google_Chrome" title="Google Chrome">Chrome</a> web browsers respectively, uses the libxslt library to do XSL transformations.<sup id="cite_ref-23" class="reference"><a href="#cite_note-23">[23]</a></sup></li>/n<li><a href="/wiki/Language_binding" title="Language binding">Bindings</a> exist for <a href="/wiki/Python_(programming_language)" title="Python (programming language)">Python</a>,<sup id="cite_ref-24" class="reference"><a More recently, code generation is increasingly common, using portable intermediate languages (such as <a href="/wiki/Java_bytecode" title="Java bytecode">Java bytecode</a> or .NET <a href="/wiki/Common_Intermediate_Language" title="Common Intermediate Language">Common Intermediate Language</a>) as the target. However, even the interpretive products generally offer separate analysis and execution phases, allowing an optimized expression tree to be created in memory and reused to perform multiple transformations. This gives substantial performance benefits in online publishing applications, where the same transformation is applied many times per second to different source documents.<sup id="cite_ref-41" class="reference"><a href="#cite_note-41">[41]</a></sup> This separation is reflected in the design of XSLT processing APIs (such as <a href="/wiki/Java_API_for_XML_Processing" title="Java API for XML Processing">JAXP</a>).</p>/n<p>Early XSLT processors had very few optimizations. Stylesheet documents were read into <a href="/wiki/Document_Object_Model" title="Document Object Model">Document Object Models</a> and the processor would act on them directly. <a href="/wiki/XPath" title="XPath">XPath</a> engines were also not optimized. Increasingly, however, XSLT processors use optimization techniques found in functional programming languages and database query languages, such as static rewriting of an expression tree (e.g., to move calculations out of loops), and lazy pipelined evaluation to reduce the memory footprint of intermediate results (and allow "early exit" when the processor can evaluate an expression such as <code>following-sibling::*[1]</code> without a complete evaluation of all subexpressions). Many processors also use tree representations that are significantly more efficient (in both space and time)<sup id="cite_ref-42" class="reference"><a href="#cite_note-42">[42]</a></sup> than general-purpose DOM implementations.</p>/n<p>In June 2014, Debbie Lockett and <a href="/wiki/Michael_Howard_Kay" title="Michael Howard Kay">Michael Kay</a> introduced an open-source benchmarking framework for XSLT processors called XT-Speedo.<sup id="cite_ref-43" class="reference"><a href="#cite_note-43">[43]</a></sup></p>/n<h2><span class="mw-headline" id="XPath">XPath</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=XSLT&amp;action=edit&amp;section=5" title="Edit section: XPath">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<div role="note" class="hatnote navigation-not-searchable">For more details on this topic, see <a href="/wiki/XPath" title="XPath">XPath</a>.</div>/n<p>XSLT uses <a href="/wiki/XPath" title="XPath">XPath</a> to identify subsets of the source document tree and perform calculations. XPath also provides a range of <a href="/wiki/Subroutine" title="Subroutine">functions</a>, which XSLT itself further augments.</p>/n<p>XSLT 1.0 uses XPath 1.0, while XSLT 2.0 uses XPath 2.0. XSLT 3.0 will work with either XPath 3.0 or 3.1. In the case of 1.0 and 2.0, the XSLT and XPath specifications were published on the same date. With 3.0, however, they were no longer synchronized; XPath 3.0 became a Recommendation in April 2014, followed by XPath 3.1 in February 2017; XSLT 3.0 followed in June 2017.</p>/n<h2><span class="mw-headline" id="XQuery_compared">XQuery compared</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=XSLT&amp;action=edit&amp;section=6" title="Edit section: XQuery compared">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<div role="note" class="hatnote navigation-not-searchable">For more details on this topic, see <a href="/wiki/XQuery#XQuery_and_XSLT_compared" title="XQuery">XQuery §&#160;XQuery and XSLT compared</a>.</div>/n<p>XSLT functionalities overlap with those of <a href="/wiki/XQuery" title="XQuery">XQuery</a>, which was initially conceived as a query language for large collections of XML documents.</p>/n<p>The XSLT 2.0 and XQuery 1.0 standards were developed by separate working groups within <a href="/wiki/W3C" class="mw-redirect" title="W3C">W3C</a>, working together to ensure a common approach where appropriate. They share the same data model, type system, and function library, and both include <a href="/wiki/XPath" title="XPath">XPath</a> 2.0 as a sublanguage.</p>/n<p>The two languages, however, are rooted in different traditions and serve the needs of different communities. XSLT was primarily conceived as a stylesheet language whose primary goal was to render XML for the human reader on screen, on <a href="/wiki/The_web" class="mw-redirect" title="The web">the web</a> (as <a href="/wiki/Web_template" class="mw-redirect" title="Web template">web template language</a>), or on paper. XQuery was primarily conceived as a <a href="/wiki/Database_query_language" class="mw-redirect" title="Database query language">database query language</a> in the tradition of <a href="/wiki/SQL" title="SQL">SQL</a>.</p>/n<p>Because the two languages originate in different communities, XSLT is stronger in its handling of narrative documents with more flexible structure, while XQuery is stronger in its data handling, for example when performing relational joins.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (July 2009)">citation needed</span></a></i>]</sup></p>/n<h2><span class="mw-headline" id="Media_types">Media types</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=XSLT&amp;action=edit&amp;section=7" title="Edit section: Media types">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>The <code>&lt;output&gt;</code> element can optionally take the attribute <code>media-type</code>, which allows one to set the <a href="/wiki/Internet_media_type" class="mw-redirect" title="Internet media type">media type</a> (or MIME type) for the resulting output, for example: <code>&lt;xsl:output output="xml" media-type="application/xml"/&gt;</code>. The XSLT 1.0 recommendation recommends the more general attribute types <code>text/xml</code> and <code>application/xml</code> since for a long time there was no registered media type for XSLT. During this time <code>text/xsl</code> became the de facto standard. In XSLT 1.0 it was not specified how the <code>media-type</code> values should be used.</p>/n<p>With the release of the XSLT 2.0, the W3C recommended the registration of the MIME media type <code>application/xslt+xml</code><sup id="cite_ref-44" class="reference"><a href="#cite_note-44">[44]</a></sup> and it was later registered with the <a href="/wiki/Internet_Assigned_Numbers_Authority" title="Internet Assigned Numbers Authority">Internet Assigned Numbers Authority</a>.<sup id="cite_ref-45" class="reference"><a href="#cite_note-45">[45]</a></sup></p>/n<p>Pre-1.0 working drafts of XSLT used <code>text/xsl</code> in their embedding examples, and this type was implemented and continues to be promoted by Microsoft in Internet Explorer<sup id="cite_ref-46" class="reference"><a href="#cite_note-46">[46]</a></sup> and MSXML. It is also widely recognized in the <code>xml-stylesheet</code> processing instruction by other browsers. In practice, therefore, users wanting to control transformation in the browser using this processing instruction are obliged to use this unregistered media type.<sup id="cite_ref-47"
