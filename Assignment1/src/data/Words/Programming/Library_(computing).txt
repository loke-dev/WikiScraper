uses libvorbisfile to play an <a href="/wiki/Ogg_Vorbis" class="mw-redirect" title="Ogg Vorbis">Ogg Vorbis</a> file</div>/n</div>/n</div>/n<p>In <a href="/wiki/Computer_science" title="Computer science">computer science</a>, a <b>library</b> is a collection of <a href="/wiki/Non-volatile_memory" title="Non-volatile memory">non-volatile resources</a> used by <a href="/wiki/Computer_program" title="Computer program">computer programs</a>, often to <a href="/wiki/Software_development" title="Software development">develop software</a>. These may include configuration data, documentation, help data, message templates, <a href="/wiki/Code_reuse" title="Code reuse">pre-written code</a> and <a href="/wiki/Subroutine" title="Subroutine">subroutines</a>, <a href="/wiki/Class_(computer_science)" class="mw-redirect" title="Class (computer science)">classes</a>, <a href="/wiki/Value_(computer_science)" title="Value (computer science)">values</a> or <a href="/wiki/Data_type" title="Data type">type</a> specifications. In <a href="/wiki/OS/360_and_successors" title="OS/360 and successors">IBM's OS/360 and its successors</a> they are referred to as <a href="/wiki/Data_set_(IBM_mainframe)#Partitioned_datasets" title="Data set (IBM mainframe)">partitioned data sets</a>.</p>/n<p>A <b>library</b> is also a collection of implementations of behavior, written in terms of a language, that has a well-defined interface by which the behavior is invoked. For instance, people who want to write a higher level program can use a library to make <a href="/wiki/System_call" title="System call">system calls</a> instead of implementing those system calls over and over again. In addition, the behavior is provided for reuse by multiple independent programs. A program invokes the library-provided behavior via a mechanism of the language. For example, in a simple <a href="/wiki/Imperative_language" class="mw-redirect" title="Imperative language">imperative language</a> such as C, the behavior in a library is invoked by using C's normal function-call. What distinguishes the call as being to a library function, versus being to another function in the same program, is the way that the code is organized in the system.</p>/n<p>Library code is organized in such a way that it can be used by multiple programs that have no connection to each other, while code that is part of a program is organized to be used only within that one program. This distinction can gain a hierarchical notion when a program grows large, such as a multi-million-line program. In that case, there may be internal libraries that are reused by independent sub-portions of the large program. The distinguishing feature is that a library is organized for the purposes of being reused by independent programs or sub-programs, and the user only needs to know the interface, and not the internal details of the library.</p>/n<p>The value of a library lies in the reuse of the behavior. When a program invokes a library, it gains the behavior implemented inside that library without having to implement that behavior itself. Libraries encourage the sharing of code in a <a href="/wiki/Modular_programming" title="Modular programming">modular</a> fashion, and ease the distribution of the code.</p>/n<p>The behavior implemented by a library can be connected to the invoking program at different <a href="/wiki/Program_lifecycle_phase" title="Program lifecycle phase">program lifecycle phases</a>. If the code of the library is accessed during the build of the invoking program, then the library is called a <a href="/wiki/Static_library" title="Static library">static library</a>.<sup id="cite_ref-Static_Libraries_1-0" class="reference"><a href="#cite_note-Static_Libraries-1">[1]</a></sup> An alternative is to build the executable of the invoking program and distribute that, independently of the library implementation. The library behavior is connected after the executable has been invoked to be executed, either as part of the process of starting the execution, or in the middle of execution. In this case the library is called a <a href="/wiki/Dynamic_library" class="mw-redirect" title="Dynamic library">dynamic library</a> (loaded at <a href="/wiki/Run_time_(program_lifecycle_phase)" title="Run time (program lifecycle phase)">run time</a>). A dynamic library can be loaded and linked when preparing a program for execution, by the <a href="/wiki/Linker_(computing)" title="Linker (computing)">linker</a>. Alternatively, in the middle of execution, an application may explicitly request that a module be <a href="/wiki/Dynamic_loading" title="Dynamic loading">loaded</a>.</p>/n<p>Most <a href="/wiki/Compiled_language" title="Compiled language">compiled languages</a> have a <a href="/wiki/Standard_library" title="Standard library">standard library</a> although programmers can also create their own custom libraries. Most modern <a href="/wiki/Software_system" title="Software system">software systems</a> provide libraries that implement the majority of the system services. Such libraries have <a href="/wiki/Commoditization" title="Commoditization">commoditized</a> the services which a modern application requires. As such, most code used by modern applications is provided in these system libraries.</p>/n<p></p>/n<div id="toc" class="toc">/n<div class="toctitle">/n<h2>Contents</h2>/n</div>/n<ul>/n<li id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Library_(computing)&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>The earliest programming concepts analogous to libraries were intended to separate <a href="/wiki/Data" title="Data">data</a> definitions from the program <a href="/wiki/Implementation" title="Implementation">implementation</a>. <a href="/wiki/JOVIAL" title="JOVIAL">JOVIAL</a> brought the "COMPOOL" (Communication Pool) concept to popular attention in 1959, although it adopted the idea from the large-system <a href="/wiki/Semi_Automatic_Ground_Environment" class="mw-redirect" title="Semi Automatic Ground Environment">SAGE</a> software. Following the computer science principles of <a href="/wiki/Separation_of_concerns" title="Separation of concerns">separation of concerns</a> and <a href="/wiki/Information_hiding" title="Information hiding">information hiding</a>, "Comm Pool's purpose was to permit the sharing of System Data among many programs by providing a centralized data description."<sup id="cite_ref-Wexelblat_1981_369_2-0" class="reference"><a href="#cite_note-Wexelblat_1981_369-2">[2]</a></sup></p>/n<p><a href="/wiki/COBOL" title="COBOL">COBOL</a> also included "primitive capabilities for a library system" in 1959,<sup id="cite_ref-Wexelblat_1981_274_3-0" class="reference"><a href="#cite_note-Wexelblat_1981_274-3">[3]</a></sup> but <a href="/wiki/Jean_E._Sammet" title="Jean E. Sammet">Jean Sammet</a> described them as "inadequate library facilities" in retrospect.<sup id="cite_ref-Wexelblat_1981_258_4-0" class="reference"><a href="#cite_note-Wexelblat_1981_258-4">[4]</a></sup></p>/n<p>Another major contributor to the modern library concept came in the form of the <a href="/wiki/Subprogram" class="mw-redirect" title="Subprogram">subprogram</a> innovation of <a href="/wiki/FORTRAN" class="mw-redirect" title="FORTRAN">FORTRAN</a>. FORTRAN subprograms can be compiled independently of each other, but the compiler lacked a <a href="/wiki/Linker_(computing)" title="Linker (computing)">linker</a>. So prior to the introduction of modules in Fortran-90, <a href="/wiki/Type_checking" class="mw-redirect" title="Type checking">type checking</a> between FORTRAN<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">[NB 1]</a></sup> subprograms was impossible.<sup id="cite_ref-Wilson_Clark_1988_126_6-0" class="reference"><a href="#cite_note-Wilson_Clark_1988_126-6">[5]</a></sup></p>/n<p>Finally, historians of the concept should remember the influential <a href="/wiki/Simula_67" class="mw-redirect" title="Simula 67">Simula 67</a>. Simula was the first <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a> language, and its <a href="/wiki/Class_(computer_science)" class="mw-redirect" title="Class (computer science)">classes</a> were nearly identical to the modern concept as used in <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>, <a href="/wiki/C%2B%2B" title="C++">C++</a>, and <a href="/wiki/C_Sharp_(programming_language)" title="C Sharp (programming language)">C#</a>. The <i>class</i> concept of Simula was also a progenitor of the <i>package</i> in <a href="/wiki/Ada_(programming_language)" title="Ada (programming language)">Ada</a> and the <i>module</i> of <a href="/wiki/Modula-2" title="Modula-2">Modula-2</a>.<sup id="cite_ref-Wilson_Clark_1988_52_7-0" class="reference"><a href="#cite_note-Wilson_Clark_1988_52-7">[6]</a></sup> Even when developed originally in 1965, Simula classes could be included in library files and added at compile time.<sup id="cite_ref-Wexelblat_1981_716_8-0" class="reference"><a href="#cite_note-Wexelblat_1981_716-8">[7]</a></sup></p>/n<h2><span class="mw-headline" id="Linking">Linking</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Library_(computing)&amp;action=edit&amp;section=2" title="Edit section: Linking">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<div role="note" class="hatnote navigation-not-searchable">Main articles: <a href="/wiki/Link_time" title="Link time">Link time</a> and <a href="/wiki/Linker_(computing)" title="Linker (computing)">Linker (computing)</a></div>/n<p>Libraries are important in the program <i>linking</i> or <i>binding</i> process, which resolves references known as <i>links</i> or <i>symbols</i> to library modules. The linking process is usually automatically done by a <i><a href="/wiki/Linker_(computing)" title="Linker (computing)">linker</a></i> or <i>binder</i> program that searches a set of libraries and other modules in a given order. Usually it is not considered an error if a link target can be found multiple times in a given set of libraries. Linking may be done when an executable file is created, or whenever the program is used at <a href="/wiki/Run_time_(program_lifecycle_phase)" title="Run time (program lifecycle phase)">run time</a>.</p>/n<p>The references being resolved may be addresses for jumps and other routine calls. They may be in the main program, or in one module depending upon another. They are resolved into fixed or relocatable addresses (from a common base) by allocating runtime memory for the <a href="/wiki/Memory_segment" class="mw-redirect" title="Memory segment">memory segments</a> of each module referenced.</p>/n<p>Some programming languages may use a feature called <i>smart linking</i> whereby the linker is aware of or integrated with the compiler, such that the linker knows how external references are used, and code in a library that is never actually <i>used</i>, even though internally referenced, can be discarded from the compiled application. For example, a program that only uses integers for arithmetic, or does no arithmetic operations at all, can exclude floating-point library routines. This smart-linking feature can lead to smaller application file sizes and reduced memory usage.</p>/n<h2><span class="mw-headline" id="Relocation">Relocation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Library_(computing)&amp;action=edit&amp;section=3" title="Edit section: Relocation">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Relocation_(computer_science)" class="mw-redirect" title="Relocation (computer science)">Relocation (computer science)</a></div>/n<p>Some references in a program or library module are stored in a relative or symbolic form which cannot be resolved until all code and libraries are assigned final static addresses. <i>Relocation</i> is the process of adjusting these references, and is done either by the linker or the <a href="/wiki/Loader_(computing)" title="Loader (computing)">loader</a>. In general, relocation cannot be done to individual libraries themselves because the addresses in memory may vary depending on the program using them and other libraries they are combined with. <a href="/wiki/Position-independent_code" title="Position-independent code">Position-independent code</a> avoids references to absolute addresses and therefore does not require relocation.</p>/n<h2><span class="mw-headline" id="Static_libraries">Static libraries</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Library_(computing)&amp;action=edit&amp;section=4" title="Edit section: Static libraries">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Static_library" title="Static library">Static library</a></div>/n<p>When linking is performed during the creation of an executable or another object file, it is known as <i>static linking</i> or <i>early binding</i>. In this case, the linking is usually done by a <a href="/wiki/Linker_(computing)" title="Linker (computing)">linker</a>, but may also be done by the <a href="/wiki/Compiler" title="Compiler">compiler</a>. A <i>static library</i>, also known as an <i>archive</i>, is one intended to be statically linked. Originally, only static libraries existed. Static linking must be performed when any modules are recompiled.</p>/n<p>All of the modules required by a program are sometimes statically linked and copied into the executable file. This process, and the resulting stand-alone file, is known as a <a href="/wiki/Static_build" title="Static build">static build</a> of the program. A static build may not need any further <a href="/wiki/Relocation_(computer_science)" class="mw-redirect" title="Relocation (computer science)">relocation</a> if <a href="/wiki/Virtual_memory" title="Virtual memory">virtual memory</a> is used and no <a href="/wiki/Address_space_layout_randomization" title="Address space layout randomization">address space layout randomization</a> is desired.<sup id="cite_ref-9" class="reference"><a href="#cite_note-9">[8]</a></sup></p>/n<h2><span class="mw-headline" id="Shared_libraries">Shared libraries</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Library_(computing)&amp;action=edit&amp;section=5" title="Edit section: Shared libraries">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<div role="note" class="hatnote navigation-not-searchable">"Shared object" redirects here. For the synchronization mechanism, see <a href="/wiki/Monitor_(synchronization)" title="Monitor (synchronization)">Monitor (synchronization)</a>.</div>/n<p>A <b>shared library</b> or <b>shared object</b> is a file that is intended to be shared by <a href="/wiki/Executable_files" class="mw-redirect" title="Executable files">executable files</a> and further shared object files. Modules used by a program are loaded from individual shared objects into memory at <a href="/wiki/Load_time" class="mw-redirect" title="Load time">load time</a> or <a href="/wiki/Run_time_(program_lifecycle_phase)" title="Run time (program lifecycle phase)">run time</a>, rather than being copied by a linker when it creates a single monolithic executable file for the program.</p>/n<p>Shared libraries can be statically linked, meaning that references to the library modules are resolved and the modules are allocated memory when the executable file is created. But often linking of shared libraries is postponed until they are loaded.<sup class="noprint Inline-Template" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Accuracy_dispute#Disputed_statement" title="Wikipedia:Accuracy dispute"><span title="The material near this tag is possibly inaccurate or nonfactual. (May 2014)">dubious</span></a> <span class="metadata">– <a href="/wiki/Talk:Library_(computing)#Dubious" title="Talk:Library (computing)">discuss</a></span></i>]</sup></p>/n<p>Most modern <a href="/wiki/Operating_system" title="Operating system">operating systems</a><sup id="cite_ref-10" class="reference"><a href="#cite_note-10">[NB 2]</a></sup> can have shared library files of the same format as the executable files. This offers two main advantages: first, it requires making only one loader for both of them, rather than two (having the single loader is considered well worth its added complexity). Secondly, it allows the executables also to be used as shared libraries, if they have a <a href="/wiki/Symbol_table" title="Symbol table">symbol table</a>. Typical combined executable and shared library formats are <a href="/wiki/Executable_and_Linkable_Format" title="Executable and Linkable Format">ELF</a> and <a href="/wiki/Mach-O" title="Mach-O">Mach-O</a> (both in Unix) and <a href="/wiki/Portable_Executable" title="Portable Executable">PE</a> (Windows).</p>/n<p>In some older environments such as <a href="/wiki/16-bit_Windows" class="mw-redirect" title="16-bit Windows">16-bit Windows</a> or <a href="/wiki/HP_Multi-Programming_Executive" title="HP Multi-Programming Executive">MPE</a> for the <a href="/wiki/HP_3000" title="HP 3000">HP 3000</a> only stack based data (local) was allowed in shared library code, or other significant restrictions were placed on shared library code.</p>/n<h3><span class="mw-headline" id="Memory_sharing">Memory sharing</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Library_(computing)&amp;action=edit&amp;section=6" title="Edit section: Memory sharing">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Shared_memory_(interprocess_communication)" class="mw-redirect" title="Shared memory (interprocess communication)">Shared memory (interprocess communication)</a></div>/n<p>Library code may be shared in memory by multiple <a href="/wiki/Process_(computing)" title="Process (computing)">processes</a> as well as on disk. If virtual memory is used, processes would execute the same physical page of RAM that is mapped into the different address spaces of the processes. This has advantages. For instance on the <a href="/wiki/OpenStep" title="OpenStep">OpenStep</a> system, applications were often only a few hundred kilobytes in size and loaded quickly; the majority of their code was located in libraries that had already been loaded for other purposes by the operating system.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (December 2008)">citation needed</span></a></i>]</sup></p>/n<p>Programs can accomplish RAM sharing by using <a href="/wiki/Position_independent_code" class="mw-redirect" title="Position independent code">position independent code</a> as in <a href="/wiki/Unix" title="Unix">Unix</a>, which leads to a complex but flexible architecture, or by using common virtual addresses as in Windows and <a href="/wiki/OS/2" title="OS/2">OS/2</a>. These systems make sure, by various tricks like pre-mapping the address space and reserving slots for each shared library, that code has a great probability of being shared. A third alternative is <a href="/wiki/Single-level_store" title="Single-level store">single-level store</a>, as used by the <a href="/wiki/IBM_System/38" title="IBM System/38">IBM System/38</a> and its successors. This allows position-dependent code but places no significant restrictions on where code can be placed or how it can be shared.</p>/n<p>In some cases different versions of shared libraries can cause problems, especially when libraries of different versions have the same file name, and different applications installed on a system each require a specific version. Such a scenario is known as <a href="/wiki/DLL_hell" class="mw-redirect" title="DLL hell">DLL hell</a>, named after the Windows and OS/2 <a href="/wiki/DLL_file" class="mw-redirect" title="DLL file">DLL file</a>. Most modern operating systems after 2001 have clean-up methods to eliminate such situations or use application specific "private" libraries.<sup id="cite_ref-endofdllhell_11-0" class="reference"><a href="#cite_note-endofdllhell-11">[9]</a></sup></p>/n<h3><span class="mw-headline" id="Dynamic_linking">Dynamic linking</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Library_(computing)&amp;action=edit&amp;section=7" title="Edit section: Dynamic linking">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Dynamic_linker" title="Dynamic linker">Dynamic linker</a></div>/n<p>Dynamic linking or <a href="/wiki/Late_binding" title="Late binding">late binding</a> is linking performed while a program is being loaded (<a href="/wiki/Load_time" class="mw-redirect" title="Load time">load time</a>) or executed (<a href="/wiki/Run_time_(program_lifecycle_phase)" title="Run time (program lifecycle phase)">run time</a>), rather than when the executable file is created. A dynamically linked library (<a href="/wiki/Dynamic-link_library" title="Dynamic-link library">dynamic-link library</a> or DLL under <a href="/wiki/Windows_(OS)" class="mw-redirect" title="Windows (OS)">Windows</a> and <a href="/wiki/OS/2" title="OS/2">OS/2</a>; dynamic shared object or DSO under <a href="/wiki/Unix-like" title="Unix-like">Unix-like</a> systems) is a library intended for dynamic linking. Only a minimum amount of work is done by the <a href="/wiki/Linker_(computing)" title="Linker (computing)">linker</a> when the executable file is created; it only records what library routines the program needs and the index names or numbers of the routines in the library. The majority of the work of linking is done at the time the application is loaded (load time) or during execution (run time). Usually, the necessary linking program, called a "dynamic linker" or "linking loader", is actually part of the underlying <a href="/wiki/Operating_system" title="Operating system">operating system</a>. (However, it is possible, and not exceedingly difficult, to write a program that uses dynamic linking and includes its own dynamic linker, even for an operating system that itself provides no support for dynamic linking.)</p>/n<p>Programmers originally developed dynamic linking in the <a href="/wiki/Multics" title="Multics">Multics</a> operating system, starting in 1964, and the MTS (<a href="/wiki/Michigan_Terminal_System" title="Michigan Terminal System">Michigan Terminal System</a>), built in the late 1960s.<sup id="cite_ref-12" class="reference"><a href="#cite_note-12">[10]</a></sup></p>/n<h3><span class="mw-headline" id="Optimizations">Optimizations</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Library_(computing)&amp;action=edit&amp;section=8" title="Edit section: Optimizations">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>Since shared libraries on most systems do not change often, systems can compute a likely load address for each shared library on the system before it is needed, and store that information in the libraries and executables. If every shared library that is loaded has undergone this process, then each will load at its predetermined address, which speeds up the process of dynamic linking. This optimization is known as <a href="/wiki/Prebinding" class="mw-redirect" title="Prebinding">prebinding</a> in macOS and <a href="/wiki/Prelinking" class="mw-redirect" title="Prelinking">prelinking</a> in Linux. Disadvantages of this technique include the time required to precompute these addresses every time the shared libraries change, the inability to use <a href="/wiki/Address_space_layout_randomization" title="Address space layout randomization">address space layout randomization</a>, and the requirement of sufficient virtual address space for use (a problem that will be alleviated by the adoption of <a href="/wiki/64-bit" class="mw-redirect" title="64-bit">64-bit</a> architectures, at least for the time being).</p>/n<h3><span class="mw-headline" id="Locating_libraries_at_run_time">Locating libraries at run time</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Library_(computing)&amp;action=edit&amp;section=9" title="Edit section: Locating libraries at run time">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>Loaders for shared libraries vary widely in functionality. Some depend on the executable storing explicit paths to the libraries. Any change to the library naming or layout of the file system will cause these systems to fail. More commonly, only the name of the library (and not the path) is stored in the executable, with the operating system supplying a method to find the library on-disk based on some algorithm.</p>/n<p>If a shared library that an executable depends on is deleted, moved, or renamed, or if an incompatible version of the library is copied to a place that is earlier in the search, the executable would fail to load. This is called <i><a href="/wiki/Dependency_hell" title="Dependency hell">Dependency hell</a></i> existing on many platforms. The (infamous) Windows variant is commonly known as <a href="/wiki/DLL_hell" class="mw-redirect" title="DLL hell">DLL hell</a>. This problem cannot occur if each version of each library is uniquely identified and each program references libraries only by their full unique identifiers. The "DLL hell" problems with earlier Windows versions arose from using only the names of libraries, which were not guaranteed to be unique, to resolve dynamic links in programs. (To avoid "DLL hell", later versions of Windows rely largely on options for programs to install private DLLs—essentially a partial retreat from the use of shared libraries—along with mechanisms to prevent replacement of shared system DLLs with earlier versions of them.)</p>/n<h4><span class="mw-headline" id="Microsoft_Windows">Microsoft Windows</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Library_(computing)&amp;action=edit&amp;section=10" title="Edit section: Microsoft Windows">edit</a><span class="mw-editsection-bracket">]</span></span></h4>/n<p><a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Microsoft Windows</a> checks the <a href="/wiki/Windows_registry" class="mw-redirect" title="Windows registry">registry</a> to determine the proper place to load DLLs that implement <a href="/wiki/Component_Object_Model" title="Component Object Model">COM objects</a>, but for other DLLs it will check the directories in a defined order. First, Windows checks the directory where it loaded the program (<i>private DLL</i><sup id="cite_ref-endofdllhell_11-1" class="reference"><a href="#cite_note-endofdllhell-11">[9]</a></sup>); any directories set by calling the <code>SetDllDirectory()</code> function; the System32, System, and Windows directories; then the current working directory; and finally the directories specified by the PATH <a href="/wiki/Environment_variable" title="Environment variable">environment variable</a>.<sup id="cite_ref-13" class="reference"><a href="#cite_note-13">[11]</a></sup> Applications written for the <a href="/wiki/.NET_Framework" title=".NET Framework">.NET Framework</a> framework (since 2002), also check the <a href="/wiki/Global_Assembly_Cache" title="Global Assembly Cache">Global Assembly Cache</a> as the primary store of shared dll files to remove the issue of <a href="/wiki/DLL_hell" class="mw-redirect" title="DLL hell">DLL hell</a>.</p>/n<h4><span class="mw-headline" id="OpenStep">OpenStep</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Library_(computing)&amp;action=edit&amp;section=11" title="Edit section: OpenStep">edit</a><span class="mw-editsection-bracket">]</span></span></h4>/n<p><a href="/wiki/OpenStep" title="OpenStep">OpenStep</a> used a more flexible system, collecting a list of libraries from a number of known locations (similar to the PATH concept) when the system first starts. Moving libraries around causes no problems at all, although users incur a time cost when first starting the system.</p>/n<h4><span class="mw-headline" id="Unix-like_systems">Unix-like systems</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Library_(computing)&amp;action=edit&amp;section=12" title="Edit section: Unix-like systems">edit</a><span class="mw-editsection-bracket">]</span></span></h4>/n<p>Most <a href="/wiki/Unix-like" title="Unix-like">Unix-like</a> systems have a "search path" specifying file system <a href="/wiki/Directory_(file_systems)" class="mw-redirect" title="Directory (file systems)">directories</a> in which to look for dynamic libraries. Some systems specify the default path in a <a href="/wiki/Configuration_file" title="Configuration file">configuration file</a>; others hard-code it into the dynamic loader. Some <a href="/wiki/Executable" title="Executable">executable file</a> formats can specify additional directories in which to search for libraries for a particular program. This can usually be overridden with an <a href="/wiki/Environment_variable" title="Environment variable">environment variable</a>, although it is disabled for <a href="/wiki/Setuid" title="Setuid">setuid</a> and setgid programs, so that a user can't force such a program to run arbitrary code with root permissions. Developers of libraries are encouraged to place their dynamic libraries in places in the default search path. On the downside, this can make installation of new libraries problematic, and these "known" locations quickly become home to an increasing number of library files, making management more complex.</p>/n<h3><span class="mw-headline" id="Dynamic_loading">Dynamic loading</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Library_(computing)&amp;action=edit&amp;section=13" title="Edit section: Dynamic loading">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Dynamic_loading" title="Dynamic loading">Dynamic loading</a></div>/n<p>Dynamic loading, a subset of dynamic linking, involves a dynamically linked library loading and unloading at <a href="/wiki/Run_time_(program_lifecycle_phase)" title="Run time (program lifecycle phase)">run time</a> on request. Such a request may be made implicitly at <a href="/wiki/Compile_time" title="Compile time">compile time</a> or explicitly at run time. Implicit requests are made at compile time when a linker adds library references that include file paths or simply file names. Explicit requests are made when applications make direct calls to an operating system's API at run time.</p>/n<p>Most operating systems that support dynamically linked libraries also support dynamically loading such libraries via a <a href="/wiki/Run_time_(program_lifecycle_phase)" title="Run time (program lifecycle phase)">run-time</a> linker <a href="/wiki/Application_programming_interface" title="Application programming interface">API</a>. For instance, <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Microsoft Windows</a> uses the API functions <tt>LoadLibrary</tt>, <tt>LoadLibraryEx</tt>, <tt>FreeLibrary</tt> and <tt>GetProcAddress</tt> with <a href="/wiki/Microsoft_Dynamic_Link_Library" class="mw-redirect" title="Microsoft Dynamic Link Library">Microsoft Dynamic Link Libraries</a>; <a href="/wiki/POSIX" title="POSIX">POSIX</a> based systems, including most UNIX and UNIX-like systems, use <tt>dlopen</tt>, <tt>dlclose</tt> and <tt>dlsym</tt>. Some development systems automate this process.</p>/n<h2><span class="mw-headline" id="Object_and_class_libraries">Object and class libraries</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Library_(computing)&amp;action=edit&amp;section=14" title="Edit section: Object and class libraries">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Although originally pioneered in the 1960s, dynamic linking did not reach <a href="/wiki/Operating_system" title="Operating system">operating systems</a> used by consumers until the late 1980s. It was generally available in some form in most operating systems by the early 1990s. During this same period, <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a> (OOP) was becoming a significant part of the programming landscape. OOP with runtime binding requires additional information that traditional libraries don't supply. In addition to the names and entry points of the code located within, they also require a list of the objects they depend on. This is a side-effect of one of OOP's main advantages, inheritance, which means that parts of the complete definition of any method may be in different places. This is more than simply listing that one library requires the services of another: in a true OOP system, the libraries themselves may not be known at <a href="/wiki/Compile_time" title="Compile time">compile time</a>, and vary from system to system.</p>/n<p>At the same time many developers worked on the idea of multi-tier programs, in which a "display" running on a desktop computer would use the services of a <a href="/wiki/Mainframe_computer" title="Mainframe computer">mainframe</a> or <a href="/wiki/Minicomputer" title="Minicomputer">minicomputer</a> for data storage or processing. For instance, a program on a GUI-based computer would send messages to a minicomputer to return small samples of a huge dataset for display. <a href="/wiki/Remote_procedure_call" title="Remote procedure call">Remote procedure calls</a> (RPC) already handled these tasks, but there was no standard RPC system.</p>/n<p>Soon the majority of the minicomputer and mainframe vendors instigated projects to combine the two, producing an OOP library format that could be used anywhere. Such systems were known as <b>object libraries</b>, or <b>distributed objects</b>, if they supported remote access (not all did). Microsoft's COM is an example of such a system for local use, DCOM a modified version that supports remote access.</p>/n<p>For some time object libraries held the status of the "next big thing" in the programming world. There were a number of efforts to create systems that would run across platforms, and companies competed to try to get developers locked into their own system. Examples include <a href="/wiki/IBM" title="IBM">IBM</a>'s <a href="/wiki/System_Object_Model" class="mw-redirect" title="System Object Model">System Object Model</a> (SOM/DSOM), <a href="/wiki/Sun_Microsystems" title="Sun Microsystems">Sun Microsystems</a>' <a href="/wiki/Distributed_Objects_Everywhere" title="Distributed Objects Everywhere">Distributed Objects Everywhere</a> (DOE), <a href="/wiki/NeXT" title="NeXT">NeXT</a>'s <a href="/wiki/Portable_Distributed_Objects" title="Portable Distributed Objects">Portable Distributed Objects</a> (PDO), <a href="/wiki/Digital_Equipment_Corporation" title="Digital Equipment Corporation">Digital</a>'s <a href="/w/index.php?title=ObjectBroker&amp;action=edit&amp;redlink=1" class="new" title="ObjectBroker (page does not exist)">ObjectBroker</a>, Microsoft's <a href="/wiki/Component_Object_Model" title="Component Object Model">Component Object Model</a> (COM/DCOM), and any number of <a href="/wiki/CORBA" class="mw-redirect" title="CORBA">CORBA</a>-based systems.</p>/n<p>After the inevitable cooling of marketing hype, object libraries continue to be used in both object-oriented programming and distributed information systems. <b>Class libraries</b> are the rough OOP equivalent of older types of code libraries. They contain <a href="/wiki/Class_(computer_science)" class="mw-redirect" title="Class (computer science)">classes</a>, which describe characteristics and define actions (<a href="/wiki/Method_(computer_science)" class="mw-redirect" title="Method (computer science)">methods</a>) that involve objects. Class libraries are used to create <a href="https://en.wiktionary.org/wiki/instance" class="extiw" title="wikt:instance">instances</a>, or objects with their characteristics set to specific values. In some OOP languages, like <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>, the distinction is clear, with the classes often contained in library files (like Java's <a href="/wiki/Jar_(file_format)" class="mw-redirect" title="Jar (file format)">JAR file format</a>) and the instantiated objects residing only in memory (although potentially able to be made <a href="/wiki/Persistence_(computer_science)" title="Persistence (computer science)">persistent</a> in separate files). In others, like <a href="/wiki/Smalltalk" title="Smalltalk">Smalltalk</a>, the class libraries are merely the starting point for a <a href="/wiki/System_image" title="System image">system image</a> that includes the entire state of the environment, classes and all instantiated objects.</p>/n<h2><span class="mw-headline" id="Remote_libraries">Remote libraries</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Library_(computing)&amp;action=edit&amp;section=15" title="Edit section: Remote libraries">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Another solution to the library issue comes from using completely separate executables (often in some lightweight form) and calling them using a <a href="/wiki/Remote_procedure_call" title="Remote procedure call">remote procedure call</a> (RPC) over a network to another computer. This approach maximizes operating system re-use: the code needed to support the library is the same code being used to provide application support and security for every other program. Additionally, such systems do not require the library to exist on the same machine, but can forward the requests over the network.</p>/n<p>However, such an approach means that every library call requires a considerable amount of overhead. RPC calls are much more expensive than calling a shared library that has already been loaded on the same machine. This approach is commonly used in a <a href="/wiki/Distributed_computing" title="Distributed computing">distributed architecture</a> that makes heavy use of such remote calls, notably client-server systems and <a href="/wiki/Application_server" title="Application server">application servers</a> such as <a href="/wiki/Enterprise_JavaBean" class="mw-redirect" title="Enterprise JavaBean">Enterprise JavaBeans</a>.</p>/n<h2><span class="mw-headline" id="Code_generation_libraries">Code generation libraries</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Library_(computing)&amp;action=edit&amp;section=16" title="Edit section: Code generation libraries">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Code generation libraries are high-level <a href="/wiki/Application_Programming_Interface" class="mw-redirect" title="Application Programming Interface">APIs</a> that can generate or transform <a href="/wiki/Byte_code" class="mw-redirect" title="Byte code">byte code</a> for <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>. They are used by <a href="/wiki/Aspect-oriented_programming" title="Aspect-oriented programming">aspect-oriented programming</a>, some data access frameworks, and for testing to generate dynamic proxy objects. They also are used to intercept field access.<sup id="cite_ref-14" class="reference"><a href="#cite_note-14">[12]</a></sup></p>/n<h2><span class="mw-headline" id="File_naming">File naming</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Library_(computing)&amp;action=edit&amp;section=17" title="Edit section: File naming">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<h3><span class="mw-headline" id="Most_modern_Unix-like_systems">Most modern <a href="/wiki/Unix-like" title="Unix-like">Unix-like</a> systems</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Library_(computing)&amp;action=edit&amp;section=18" title="Edit section: Most modern Unix-like systems">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>The system stores <code>libfoo.a</code> and <code>libfoo.so</code> files in directories such as <code>/lib</code>, <code>/usr/lib</code> or <code>/usr/local/lib</code>. The filenames always start with <code>lib</code>, and end with a suffix of <code>.a</code> (<a href="/wiki/Ar_(file_format)" class="mw-redirect" title="Ar (file format)">archive</a>, static library) or of <code>.so</code> (shared object, dynamically linked library). Some systems might have multiple names for the dynamically linked library, with most of the names being names for <a href="/wiki/Symbolic_links" class="mw-redirect" title="Symbolic links">symbolic links</a> to the remaining name; those names might include the major version of the library, or the full version number; for example, on some systems <code>libfoo.so.2</code> would be the filename for the second major interface revision of the dynamically linked library <code>libfoo</code>. The <code>.la</code> files sometimes found in the library directories are <a href="/wiki/Libtool" class="mw-redirect" title="Libtool">libtool</a> archives, not usable by the system as such.</p>/n<h3><span class="mw-headline" id="macOS"><a href="/wiki/MacOS" title="MacOS">macOS</a></span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Library_(computing)&amp;action=edit&amp;section=19" title="Edit section: macOS">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>The system inherits static library conventions from <a href="/wiki/BSD" class="mw-redirect" title="BSD">BSD</a>, with the library stored in a <code>.a</code> file, and can use <code>.so</code>-style dynamically linked libraries (with the <code>.dylib</code> suffix instead). Most libraries in macOS, however, consist of "frameworks", placed inside special directories called "<a href="/wiki/Bundle_(macOS)" title="Bundle (macOS)">bundles</a>" which wrap the library's required files and metadata. For example, a framework called <code>MyFramework</code> would be implemented in a bundle called <code>MyFramework.framework</code>, with <code>MyFramework.framework/MyFramework</code> being either the dynamically linked library file or being a symlink to the dynamically linked library file in <code>MyFramework.framework/Versions/Current/MyFramework</code>.</p>/n<h3><span class="mw-headline" id="Microsoft_Windows_2"><a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Microsoft Windows</a></span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Library_(computing)&amp;action=edit&amp;section=20" title="Edit section: Microsoft Windows">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p><a href="/wiki/Dynamic-link_library" title="Dynamic-link library">Dynamic-link libraries</a> usually have the suffix <code>*.DLL</code>,<sup id="cite_ref-15" class="reference"><a href="#cite_note-15">[13]</a></sup> although other file name extensions may identify specific-purpose dynamically linked libraries, e.g. <code>*.OCX</code> for <a href="/wiki/Object_Linking_and_Embedding" title="Object Linking and Embedding">OLE</a> libraries. The interface revisions are either encoded in the file names, or abstracted away using <a href="/wiki/Component_object_model" class="mw-redirect" title="Component object model">COM-object</a> interfaces. Depending on how they are compiled, <code>*.LIB</code> files can be either static libraries or representations of dynamically linkable libraries needed only during compilation, known as "<a href="/wiki/Dynamic-link_library#Import_libraries" title="Dynamic-link library">import libraries</a>". Unlike in the <a href="/wiki/UNIX" class="mw-redirect" title="UNIX">UNIX</a> world, which uses different file extensions, when linking against <code>.LIB</code> file in <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Windows</a> one must first know if it is a regular static library or an import library. In the latter case, a <code>.DLL</code> file must be present at run time.</p>/n<h2><span class="mw-headline" id="See_also">See also</span><span class="mw-editsection"><span
