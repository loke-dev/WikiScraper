class="mw-body-content">/n									<div id="siteSub" class="noprint">From Wikipedia, the free encyclopedia</div>/n								<div id="contentSub"><span class="mw-redirectedfrom">  (Redirected from <a to:					<a href="#mw-head">navigation</a>, 					<a href="#p-search">search</a>/n				</div>/n				<div id="mw-content-text" lang="en" dir="ltr" class="mw-content-ltr"><div class="mw-parser-output"><p>A <b>data model</b> (or <b>datamodel</b><sup id="cite_ref-1" class="reference"><a href="#cite_note-1">[1]</a></sup><sup id="cite_ref-w3cxpath_2-0" class="reference"><a href="#cite_note-w3cxpath-2">[2]</a></sup><sup id="cite_ref-npmdatamodel_3-0" class="reference"><a href="#cite_note-npmdatamodel-3">[3]</a></sup><sup id="cite_ref-4" class="reference"><a href="#cite_note-4">[4]</a></sup><sup id="cite_ref-5" class="reference"><a href="#cite_note-5">[5]</a></sup>) is an abstract model that organizes elements of <a href="/wiki/Data" title="Data">data</a> and standardizes how they relate to one another and to properties of the real world entities. For instance, a data model may specify that the data element representing a car be composed of a number of other elements which, in turn, represent the color and size of the car and define its owner.</p>/n<p>The term <b>data model</b> is used in two distinct but closely related senses. Sometimes it refers to an abstract formalization of the objects and relationships found in a particular application domain, for example the customers, products, and orders found in a manufacturing organization. At other times it refers to a set of concepts used in defining such formalizations: for example concepts such as entities, attributes, relations, or tables. So the "data model" of a banking application may be defined using the entity-relationship "data model". This article uses the term in both senses.</p>/n<div class="thumb tright">/n<div class="thumbinner" style="width:362px;"><a href="/wiki/File:Data_modeling_context.svg" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/2/2b/Data_modeling_context.svg/360px-Data_modeling_context.svg.png" width="360" height="286" class="thumbimage" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/2/2b/Data_modeling_context.svg/540px-Data_modeling_context.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/2/2b/Data_modeling_context.svg/720px-Data_modeling_context.svg.png 2x" data-file-width="638" data-file-height="507" /></a>/n<div class="thumbcaption">/n<div class="magnify"><a href="/wiki/File:Data_modeling_context.svg" class="internal" title="Enlarge"></a></div>/nOverview of data modeling context: Data model is based on Data, Data relationship, Data semantic and Data constraint. A data model provides the details of <a href="/wiki/Information" title="Information">information</a> to be stored, and is of primary use when the final product is the generation of computer <a href="/wiki/Software_code" class="mw-redirect" title="Software code">software code</a> for an application or the preparation of a <a href="/wiki/Functional_specification" title="Functional specification">functional specification</a> to aid a <a href="/wiki/Computer_software" class="mw-redirect" title="Computer software">computer software</a> make-or-buy decision. The figure is an example of the interaction between <a href="/wiki/Business_process_modeling" title="Business process modeling">process</a> and data models.<sup id="cite_ref-SS93_6-0" class="reference"><a href="#cite_note-SS93-6">[6]</a></sup></div>/n</div>/n</div>/n<p>A data model explicitly determines the structure of data. Data models are specified in a <a href="/wiki/Data_modeling" title="Data modeling">data modeling</a> notation, which is often graphical in form.<sup id="cite_ref-MRM99_7-0" class="reference"><a href="#cite_note-MRM99-7">[7]</a></sup></p>/n<p>A data model can sometimes be referred to as a <a href="/wiki/Data_structure" title="Data structure">data structure</a>, especially in the context of <a href="/wiki/Programming_language" title="Programming language">programming languages</a>. Data models are often complemented by <a href="/wiki/Function_model" title="Function model">function models</a>, especially in the context of <a href="/wiki/Enterprise_model" class="mw-redirect" title="Enterprise model">enterprise models</a>.</p>/n<p></p>/n<div id="toc" class="toc">/n<div class="toctitle">/n<h2>Contents</h2>/n</div>/n<ul>/n<li class="toclevel-1 tocsection-1"><a class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=1" title="Edit section: Overview">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>Managing large quantities of structured and <a href="/wiki/Unstructured_data" title="Unstructured data">unstructured data</a> is a primary function of <a href="/wiki/Information_system" title="Information system">information systems</a>. Data models describe the structure, manipulation and integrity aspects of the data stored in data management systems such as relational databases. They typically do not describe unstructured data, such as <a href="/wiki/Word_processor" title="Word processor">word processing</a> documents, <a href="/wiki/Email" title="Email">email messages</a>, pictures, digital audio, and video.</p>/n<h3><span class="mw-headline" id="The_role_of_data_models">The role of data models</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=2" title="Edit section: The role of data models">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<div class="thumb tright">/n<div class="thumbinner" style="width:322px;"><a href="/wiki/File:3-4_Data_model_roles.svg" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/9/93/3-4_Data_model_roles.svg/320px-3-4_Data_model_roles.svg.png" width="320" height="206" class="thumbimage" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/9/93/3-4_Data_model_roles.svg/480px-3-4_Data_model_roles.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/9/93/3-4_Data_model_roles.svg/640px-3-4_Data_model_roles.svg.png 2x" data-file-width="741" data-file-height="477" /></a>/n<div class="thumbcaption">/n<div class="magnify"><a href="/wiki/File:3-4_Data_model_roles.svg" class="internal" title="Enlarge"></a></div>/nHow data models deliver benefit<sup id="cite_ref-MW99_8-0" class="reference"><a href="#cite_note-MW99-8">[8]</a></sup></div>/n</div>/n</div>/n<p>The main aim of data models is to support the development of <a href="/wiki/Information_system" title="Information system">information systems</a> by providing the definition and format of data. According to West and Fowler (1999) "if this is done consistently across systems then compatibility of data can be achieved. If the same data structures are used to store and access data then different applications can share data. The results of this are indicated above. However, systems and interfaces often cost more than they should, to build, operate, and maintain. They may also constrain the business rather than support it. A major cause is that the quality of the data models implemented in systems and interfaces is poor".<sup id="cite_ref-MW99_8-1" class="reference"><a href="#cite_note-MW99-8">[8]</a></sup></p>/n<ul>/n<li>"Business rules, specific to how things are done in a particular place, are often fixed in the structure of a data model. This means that small changes in the way business is conducted lead to large changes in computer systems and interfaces".<sup id="cite_ref-MW99_8-2" class="reference"><a href="#cite_note-MW99-8">[8]</a></sup></li>/n<li>"Entity types are often not identified, or incorrectly identified. This can lead to replication of data, data structure, and functionality, together with the attendant costs of that duplication in development and maintenance".<sup id="cite_ref-MW99_8-3" class="reference"><a href="#cite_note-MW99-8">[8]</a></sup></li>/n<li>"Data models for different systems are arbitrarily different. The result of this is that complex interfaces are required between systems that share data. These interfaces can account for between 25-70% of the cost of current systems".<sup id="cite_ref-MW99_8-4" class="reference"><a href="#cite_note-MW99-8">[8]</a></sup></li>/n<li>"Data cannot be shared electronically with customers and suppliers, because the structure and meaning of data has not been standardized. For example, engineering design data and drawings for process plant are still sometimes exchanged on paper".<sup id="cite_ref-MW99_8-5" class="reference"><a href="#cite_note-MW99-8">[8]</a></sup></li>/n</ul>/n<p>The reason for these problems is a lack of standards that will ensure that data models will both meet business needs and be consistent.<sup id="cite_ref-MW99_8-6" class="reference"><a href="#cite_note-MW99-8">[8]</a></sup></p>/n<p>A data model explicitly determines the structure of data. Typical applications of data models include database models, design of information systems, and enabling exchange of data. Usually data models are specified in a data modeling language.[3]</p>/n<h3><span class="mw-headline" id="Three_perspectives">Three perspectives</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=3" title="Edit section: Three perspectives">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<div class="thumb tright">/n<div class="thumbinner" style="width:322px;"><a href="/wiki/File:4-2_ANSI-SPARC_three_level_architecture.svg" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/8/84/4-2_ANSI-SPARC_three_level_architecture.svg/320px-4-2_ANSI-SPARC_three_level_architecture.svg.png" width="320" height="211" class="thumbimage" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/8/84/4-2_ANSI-SPARC_three_level_architecture.svg/480px-4-2_ANSI-SPARC_three_level_architecture.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/8/84/4-2_ANSI-SPARC_three_level_architecture.svg/640px-4-2_ANSI-SPARC_three_level_architecture.svg.png 2x" data-file-width="653" data-file-height="431" /></a>/n<div class="thumbcaption">/n<div class="magnify"><a href="/wiki/File:4-2_ANSI-SPARC_three_level_architecture.svg" class="internal" title="Enlarge"></a></div>/nThe ANSI/SPARC <a href="/wiki/Three_schema_approach" class="mw-redirect" title="Three schema approach">three level architecture</a>. This shows that a data model can be an external model (or view), a conceptual model, or a physical model. This is not the only way to look at data models, but it is a useful way, particularly when comparing models.<sup id="cite_ref-MW99_8-7" class="reference"><a href="#cite_note-MW99-8">[8]</a></sup></div>/n</div>/n</div>/n<p>A data model <i>instance</i> may be one of three kinds according to <a href="/wiki/ANSI" class="mw-redirect" title="ANSI">ANSI</a> in 1975:<sup id="cite_ref-9" class="reference"><a href="#cite_note-9">[9]</a></sup></p>/n<ul>/n<li><a href="/wiki/Conceptual_data_model" class="mw-redirect" title="Conceptual data model">Conceptual data model</a>&#160;: describes the semantics of a domain, being the scope of the model. For example, it may be a model of the interest area of an organization or industry. This consists of entity classes, representing kinds of things of significance in the domain, and relationship assertions about associations between pairs of entity classes. A conceptual schema specifies the kinds of facts or propositions that can be expressed using the model. In that sense, it defines the allowed expressions in an artificial 'language' with a scope that is limited by the scope of the model.</li>/n<li><a href="/wiki/Logical_data_model" title="Logical data model">Logical data model</a>&#160;: describes the semantics, as represented by a particular data manipulation technology. This consists of descriptions of tables and columns, object oriented classes, and XML tags, among other things.</li>/n<li><a href="/wiki/Physical_data_model" title="Physical data model">Physical data model</a>&#160;: describes the physical means by which data are stored. This is concerned with partitions, CPUs, tablespaces, and the like.</li>/n</ul>/n<p>The significance of this approach, according to ANSI, is that it allows the three perspectives to be relatively independent of each other. Storage technology can change without affecting either the logical or the conceptual model. The table/column structure can change without (necessarily) affecting the conceptual model. In each case, of course, the structures must remain consistent with the other model. The table/column structure may be different from a direct translation of the entity classes and attributes, but it must ultimately carry out the objectives of the conceptual entity class structure. Early phases of many software development projects emphasize the design of a <a href="/wiki/Conceptual_schema" title="Conceptual schema">conceptual data model</a>. Such a design can be detailed into a <a href="/wiki/Logical_data_model" title="Logical data model">logical data model</a>. In later stages, this model may be translated into <a href="/wiki/Physical_data_model" title="Physical data model">physical data model</a>. However, it is also possible to implement a conceptual model directly.</p>/n<h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=4" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>One of the earliest pioneering works in modelling information systems was done by Young and Kent (1958),<sup id="cite_ref-10" class="reference"><a href="#cite_note-10">[10]</a></sup><sup id="cite_ref-JAB07_11-0" class="reference"><a href="#cite_note-JAB07-11">[11]</a></sup> who argued for "a precise and abstract way of specifying the informational and time characteristics of a <a href="/wiki/Data_processing" title="Data processing">data processing</a> problem". They wanted to create "a notation that should enable the <a href="/wiki/Systems_analyst" title="Systems analyst">analyst</a> to organize the problem around any piece of <a href="/wiki/Computer_hardware" title="Computer hardware">hardware</a>". Their work was a first effort to create an abstract specification and invariant basis for designing different alternative implementations using different hardware components. A next step in IS modelling was taken by <a href="/wiki/CODASYL" title="CODASYL">CODASYL</a>, an IT industry consortium formed in 1959, who essentially aimed at the same thing as Young and Kent: the development of "a proper structure for machine independent problem definition language, at the system level of data processing". This led to the development of a specific IS <a href="/wiki/Information_algebra" title="Information algebra">information algebra</a>.<sup id="cite_ref-JAB07_11-1" class="reference"><a href="#cite_note-JAB07-11">[11]</a></sup></p>/n<p>In the 1960s data modeling gained more significance with the initiation of the <a href="/wiki/Management_information_system" title="Management information system">management information system</a> (MIS) concept. According to Leondes (2002), "during that time, the information system provided the data and information for management purposes. The first generation <a href="/wiki/Database_system" class="mw-redirect" title="Database system">database system</a>, called <a href="/wiki/Integrated_Data_Store" title="Integrated Data Store">Integrated Data Store</a> (IDS), was designed by <a href="/wiki/Charles_Bachman" title="Charles Bachman">Charles Bachman</a> at General Electric. Two famous database models, the <a href="/wiki/Network_data_model" class="mw-redirect" title="Network data model">network data model</a> and the <a href="/wiki/Hierarchical_data_model" class="mw-redirect" title="Hierarchical data model">hierarchical data model</a>, were proposed during this period of time".<sup id="cite_ref-12" class="reference"><a href="#cite_note-12">[12]</a></sup> Towards the end of the 1960s, <a href="/wiki/Edgar_F._Codd" title="Edgar F. Codd">Edgar F. Codd</a> worked out his theories of data arrangement, and proposed the <a href="/wiki/Relational_model" title="Relational model">relational model</a> for database management based on <a href="/wiki/First-order_logic" title="First-order logic">first-order predicate logic</a>.<sup id="cite_ref-13" class="reference"><a href="#cite_note-13">[13]</a></sup></p>/n<p>In the 1970s <a href="/wiki/Entity_relationship_model" class="mw-redirect" title="Entity relationship model">entity relationship modeling</a> emerged as a new type of conceptual data modeling, originally proposed in 1976 by <a href="/wiki/Peter_Chen" title="Peter Chen">Peter Chen</a>. Entity relationship models were being used in the first stage of <a href="/wiki/Information_system" title="Information system">information system</a> design during the <a href="/wiki/Requirements_analysis" title="Requirements analysis">requirements analysis</a> to describe information needs or the type of <a href="/wiki/Information" title="Information">information</a> that is to be stored in a <a href="/wiki/Database" title="Database">database</a>. This technique can describe any <a href="/wiki/Ontology_(computer_science)" class="mw-redirect" title="Ontology (computer science)">ontology</a>, i.e., an overview and classification of concepts and their relationships, for a certain <a href="/wiki/Universe_of_discourse" class="mw-redirect" title="Universe of discourse">area of interest</a>.</p>/n<p>In the 1970s <a href="/wiki/G.M._Nijssen" title="G.M. Nijssen">G.M. Nijssen</a> developed "Natural Language Information Analysis Method" (NIAM) method, and developed this in the 1980s in cooperation with <a href="/wiki/Terry_Halpin" title="Terry Halpin">Terry Halpin</a> into <a href="/wiki/Object-Role_Modeling" class="mw-redirect" title="Object-Role Modeling">Object-Role Modeling</a> (ORM). However, it was Terry Halpin's 1989 PhD thesis that created the formal foundation on which Object-Role Modeling is based.</p>/n<p>Bill Kent, in his 1978 book <i>Data and Reality,</i><sup id="cite_ref-14" class="reference"><a href="#cite_note-14">[14]</a></sup> compared a data model to a map of a territory, emphasizing that in the real world, "highways are not painted red, rivers don't have county lines running down the middle, and you can't see contour lines on a mountain". In contrast to other researchers who tried to create models that were mathematically clean and elegant, Kent emphasized the essential messiness of the real world, and the task of the data modeller to create order out of chaos without excessively distorting the truth.</p>/n<p>In the 1980s, according to Jan L. Harrington (2000), "the development of the <a href="/wiki/Object-oriented" class="mw-redirect" title="Object-oriented">object-oriented</a> paradigm brought about a fundamental change in the way we look at data and the procedures that operate on data. Traditionally, data and procedures have been stored separately: the data and their relationship in a database, the procedures in an application program. Object orientation, however, combined an entity's procedure with its data."<sup id="cite_ref-JLH00_15-0" class="reference"><a href="#cite_note-JLH00-15">[15]</a></sup></p>/n<h2><span class="mw-headline" id="Types_of_data_models">Types of data models</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=5" title="Edit section: Types of data models">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<p>123</p>/n<h3><span class="mw-headline" id="Database_model">Database model</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=6" title="Edit section: Database model">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Database_model" title="Database model">Database model</a></div>/n<p>A database model is a specification describing how a database is structured and used.</p>/n<p>Several such models have been suggested. Common models include:</p>/n<dl>/n<dt><a href="/wiki/Flat_file_database" title="Flat file database">Flat model</a></dt>/n<dd>This may not strictly qualify as a data model. The flat (or table) model consists of a single, two-dimensional array of data elements, where all members of a given column are assumed to be similar values, and all members of a row are assumed to be related to one another.</dd>/n<dt><a href="/wiki/Hierarchical_model" class="mw-redirect" title="Hierarchical model">Hierarchical model</a></dt>/n<dd>The hierarchical model is similar to the network model except that links in the hierarchical model form a tree structure,while the network model allows arbitrary graph.</dd>/n<dt><a href="/wiki/Network_model" title="Network model">Network model</a></dt>/n<dd>This model organizes data using two fundamental constructs, called records and sets. Records contain fields, and sets define one-to-many relationships between records: one owner, many members.The network data model is an abstraction of the design concept used in the implementation of databases.</dd>/n<dt><a href="/wiki/Relational_model" title="Relational model">Relational model</a></dt>/n<dd>is a database model based on first-order predicate logic. Its core idea is to describe a database as a collection of predicates over a finite set of predicate variables, describing constraints on the possible values and combinations of values.The power of the relational data model lies in its mathematical foundations and a simple user-level paradigm.</dd>/n<dt><a href="/wiki/Object-relational_model" class="mw-redirect" title="Object-relational model">Object-relational model</a></dt>/n<dd>Similar to a relational database model, but objects, classes and inheritance are directly supported in <a href="/wiki/Database_schema" title="Database schema">database schemas</a> and in the query language.</dd>/n<dt><a href="/wiki/Star_schema" title="Star schema">Star schema</a></dt>/n<dd>The simplest style of data warehouse schema. The star schema consists of a few "fact tables" (possibly only one, justifying the name) referencing any number of "dimension tables". title="Conceptual schema">conceptual data models</a> by providing graphical notations which document <a href="/wiki/Entity_class" class="mw-redirect" title="Entity class">entities</a> and their <a href="/wiki/Relational_model" title="Relational model">relationships</a>, and the <a href="/wiki/Integrity_constraints" class="mw-redirect" title="Integrity constraints">constraints</a> that bind them. The basic graphic elements of DSDs are <a href="/wiki/Box" title="Box">boxes</a>, representing entities, and <a href="/wiki/Arrow" title="Arrow">arrows</a>, representing relationships. Data structure diagrams are most useful for documenting complex data entities.</p>/n<p>Data structure diagrams are an extension of the <a href="/wiki/Entity-relationship_model" class="mw-redirect" title="Entity-relationship model">entity-relationship model</a> (ER model). In DSDs, <a href="/wiki/Attribute_(computing)" title="Attribute (computing)">attributes</a> are specified inside the entity boxes rather than outside of them, while relationships are drawn as boxes composed of attributes which specify the constraints that bind entities together. The E-R model, while robust, doesn't provide a way to specify the constraints between relationships, and becomes visually cumbersome when representing entities with several attributes. DSDs differ from the ER model in that the ER model focuses on the relationships between different entities, whereas DSDs focus on the relationships of the elements within an entity and enable users to fully see the links and relationships between each entity.</p>/n<p>There are several styles for representing data structure diagrams, with the notable difference in the manner of defining <a href="/wiki/Cardinality_(data_modeling)" title="Cardinality (data modeling)">cardinality</a>. The choices are between arrow heads, inverted arrow heads (<a href="/wiki/Entity-relationship_model" class="mw-redirect" title="Entity-relationship model">crow's feet</a>), or numerical representation class="mw-editsection-bracket">]</span></span></h3>/n<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Generic_data_model" title="Generic data model">Generic data model</a></div>/n<p>Generic data models are generalizations of conventional data models. They define standardised general relation types, together with the kinds of things that may be related by such a relation type. Generic data models are developed as an approach to solve some shortcomings of conventional data models. For example, different modelers usually produce different conventional data models of the same domain. This can lead to difficulty in bringing the models of different people together and is an obstacle for data exchange and data integration. Invariably, however, this difference is attributable to different levels of abstraction in the models and differences in the kinds of facts that can be instantiated (the semantic expression capabilities of the models). The modelers need to communicate and agree on certain elements which are to be rendered more concretely, in order to make the differences less significant.</p>/n<h3><span class="mw-headline" id="Semantic_data_model">Semantic data model</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=11" title="Edit section: Semantic data model">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Semantic_data_model" title="Semantic data model">Semantic data model</a></div>/n<div class="thumb tright">/n<div class="thumbinner" style="width:322px;"><a href="/wiki/File:A2_4_Semantic_Data_Models.svg" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/5/58/A2_4_Semantic_Data_Models.svg/320px-A2_4_Semantic_Data_Models.svg.png" width="320" height="175" class="thumbimage" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/5/58/A2_4_Semantic_Data_Models.svg/480px-A2_4_Semantic_Data_Models.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/5/58/A2_4_Semantic_Data_Models.svg/640px-A2_4_Semantic_Data_Models.svg.png 2x" data-file-width="608" data-file-height="333" /></a>/n<div class="thumbcaption">/n<div class="magnify"><a href="/wiki/File:A2_4_Semantic_Data_Models.svg" class="internal" title="Enlarge"></a></div>/nSemantic data models<sup id="cite_ref-FIPS184_16-1" class="reference"><a href="#cite_note-FIPS184-16">[16]</a></sup></div>/n</div>/n</div>/n<p>A semantic data model in software engineering is a technique to define the meaning of data within the context of its interrelationships with other data. A semantic data model is an abstraction which defines how the stored symbols relate to the real world.<sup id="cite_ref-FIPS184_16-2" class="reference"><a href="#cite_note-FIPS184-16">[16]</a></sup> A semantic data model is sometimes called a <a href="/wiki/Conceptual_data_model" class="mw-redirect" title="Conceptual data model">conceptual data model</a>.</p>/n<p>The logical data structure of a <a href="/wiki/Database_management_system" class="mw-redirect" title="Database management system">database management system</a> (DBMS), whether <a href="/wiki/Hierarchical_model" class="mw-redirect" title="Hierarchical model">hierarchical</a>, <a href="/wiki/Network_model" title="Network model">network</a>, or <a href="/wiki/Relational_model" title="Relational model">relational</a>, cannot totally satisfy the <a href="/wiki/Requirements_analysis" title="Requirements analysis">requirements</a> for a conceptual definition of data because it is limited in scope and biased toward the implementation strategy employed by the DBMS. Therefore, the need to define data from a <a href="/wiki/Three_schema_approach" class="mw-redirect" title="Three schema approach">conceptual view</a> has led to the development of semantic data modeling techniques. That is, techniques to define the meaning of data within the context of its interrelationships with other data. As illustrated in the figure. The real world, in terms of resources, ideas, events, etc., are symbolically defined within physical data stores. A semantic data model is an abstraction which defines how the stored symbols relate to the real world. Thus, the model must be a true representation of the real world.<sup id="cite_ref-FIPS184_16-3" class="reference"><a href="#cite_note-FIPS184-16">[16]</a></sup></p>/n<h2><span class="mw-headline" id="Data_model_topics">Data model topics</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=12" title="Edit section: Data model topics">edit</a><span class="mw-editsection-bracket">]</span></span></h2>/n<h3><span class="mw-headline" id="Data_architecture">Data architecture</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=13" title="Edit section: Data architecture">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Data_architecture" title="Data architecture">Data architecture</a></div>/n<p>Data architecture is the design of data for use in defining the target state and the subsequent planning needed to hit the target state. It is usually one of several <a href="/wiki/Architecture_domain" title="Architecture domain">architecture domains</a> that form the pillars of an <a href="/wiki/Enterprise_architecture" title="Enterprise architecture">enterprise architecture</a> or <a href="/wiki/Solution_architecture" title="Solution architecture">solution architecture</a>.</p>/n<p>A data architecture describes the data structures used by a business and/or its applications. There are descriptions of data in storage and data in motion; descriptions of data stores, data groups and data items; and mappings of those data artifacts to data qualities, applications, locations etc.</p>/n<p>Essential to realizing the target state, Data architecture describes how data is processed, stored, and utilized in a given system. class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=18" title="Edit section: Data model theory">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<p>The term data model can have two meanings:<sup id="cite_ref-Beynon-Davies_P._2004_20-0" class="reference"><a href="#cite_note-Beynon-Davies_P._2004-20">[20]</a></sup></p>/n<ol>/n<li>A data model <i>theory</i>, i.e. a formal description of how data may be structured and accessed.</li>/n<li>A data model <i>instance</i>, i.e. applying a data model <i>theory</i> to create a practical data model <i>instance</i> for some particular application.</li>/n</ol>/n<p>A data model theory has three main components:<sup id="cite_ref-Beynon-Davies_P._2004_20-1" class="reference"><a href="#cite_note-Beynon-Davies_P._2004-20">[20]</a></sup></p>/n<ul>/n<li>The structural part: a collection of data structures which are used to create databases representing the entities or objects modeled by the database.</li>/n<li>The integrity part: a collection of rules governing the constraints placed on these data structures to ensure structural integrity.</li>/n<li>The manipulation part: a collection of operators which can be applied to the data structures, to update and query the data contained in the database.</li>/n</ul>/n<p>For example, in the <a href="/wiki/Relational_model" title="Relational model">relational model</a>, the structural part is based on a modified concept of the <a href="/wiki/Relation_(mathematics)" class="mw-redirect" title="Relation (mathematics)">mathematical relation</a>; the integrity part is expressed in <a href="/wiki/First-order_logic" title="First-order logic">first-order logic</a> and the manipulation part is expressed using the <a href="/wiki/Relational_algebra" title="Relational algebra">relational algebra</a>, <a href="/wiki/Tuple_calculus" class="mw-redirect" title="Tuple calculus">tuple calculus</a> and <a href="/wiki/Domain_calculus" class="mw-redirect" title="Domain calculus">domain calculus</a>.</p>/n<p>A data model instance is created by applying a data model theory. This is typically done to solve some business enterprise requirement. Business requirements are normally captured by a semantic <a href="/wiki/Logical_data_model" title="Logical data model">logical data model</a>. This is transformed into a physical data model instance from which is generated a physical database. For example, a data modeler may use a data modeling tool to create an <a href="/wiki/Entity-relationship_model" class="mw-redirect" title="Entity-relationship model">entity-relationship model</a> of the corporate data repository of some business enterprise. This model is transformed into a <a href="/wiki/Relational_model" title="Relational model">relational model</a>, which in turn generates a <a href="/wiki/Relational_database" title="Relational database">relational database</a>.</p>/n<h3><span class="mw-headline" id="Patterns">Patterns</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a Data flow diagrams were invented by <a href="/wiki/Larry_Constantine" title="Larry Constantine">Larry Constantine</a>, the original developer of structured design,<sup id="cite_ref-23" class="reference"><a href="#cite_note-23">[23]</a></sup> based on Martin and Estrin's "data flow graph" model of computation.</p>/n<p>It is common practice to draw a <a href="/wiki/System_Context_Diagram" class="mw-redirect" title="System Context Diagram">context-level Data flow diagram</a> first which shows the interaction between the system and outside entities. The <b>DFD</b> is designed to show how a system is divided into smaller portions and to highlight the flow of data between those parts. This context-level Data flow diagram is then "exploded" to show more detail of the system being modeled</p>/n<h3><span class="mw-headline" id="Information_model">Information model</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=22" title="Edit section: Information model">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Information_model" title="Information model">Information model</a></div>/n<div class="thumb tright">/n<div class="thumbinner" style="width:322px;"><a href="/wiki/File:A_01_Audio_compact_disc_collection.svg" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/1/10/A_01_Audio_compact_disc_collection.svg/320px-A_01_Audio_compact_disc_collection.svg.png" width="320" height="266" class="thumbimage" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/1/10/A_01_Audio_compact_disc_collection.svg/480px-A_01_Audio_compact_disc_collection.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/1/10/A_01_Audio_compact_disc_collection.svg/640px-A_01_Audio_compact_disc_collection.svg.png 2x" data-file-width="813" data-file-height="677" /></a>/n<div class="thumbcaption">/n<div class="magnify"><a href="/wiki/File:A_01_Audio_compact_disc_collection.svg" class="internal" title="Enlarge"></a></div>/nExample of an <a href="/wiki/EXPRESS_(data_modeling_language)" title="EXPRESS (data modeling language)">EXPRESS G</a> <a href="/wiki/Information_model" title="Information model">Information model</a></div>/n</div>/n</div>/n<p>An Information model is not a type of data model, but more or less an alternative model. Within the field of software engineering both a data model and an information model can be abstract, formal representations of entity types that includes their properties, relationships and the operations that can be performed on them. The entity types in the model may be kinds of real-world objects, such as devices in a network, or they may themselves be abstract, such as for the entities used in a billing system. Typically, they are used to model a constrained domain that can be described by a closed set of entity types, properties, relationships and operations.</p>/n<p>According to Lee (1999)<sup id="cite_ref-Lee99_24-0" class="reference"><a href="#cite_note-Lee99-24">[24]</a></sup> an information model is a representation of concepts, relationships, constraints, rules, and <a href="/wiki/Operation_(mathematics)" title="Operation (mathematics)">operations</a> to specify <a href="/wiki/Semantic_data_model" title="Semantic data model">data semantics</a> for a chosen domain of discourse. It can provide sharable, stable, and organized structure of information requirements for the domain context.<sup id="cite_ref-Lee99_24-1" class="reference"><a href="#cite_note-Lee99-24">[24]</a></sup> More in general the term <i>information model</i> is used for models of individual things, such as facilities, buildings, process plants, etc. In those cases the concept is specialised to <a href="/wiki/Facility_Information_Model" class="mw-redirect" title="Facility Information Model">Facility Information Model</a>, <a href="/wiki/Building_information_modeling" title="Building information modeling">Building Information Model</a>, Plant Information Model, etc. Such an information model is an integration of a model of the facility with the data and documents about the facility.</p>/n<p>An information model provides formalism to the description of a problem domain without constraining how that description is mapped to an actual implementation in software. There may be many mappings of the information model. Such mappings are called data models, irrespective of whether they are <a href="/wiki/Object_model" title="Object model">object models</a> (e.g. using <a href="/wiki/Unified_Modeling_Language" title="Unified Modeling Language">UML</a>), <a href="/wiki/Entity-relationship_model" class="mw-redirect" title="Entity-relationship model">entity relationship models</a> or <a href="/wiki/XML_schema" title="XML schema">XML schemas</a>.</p>/n<div class="thumb tleft">/n<div class="thumbinner" style="width:182px;"><a href="/wiki/File:JKDOM.SVG" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/e/e4/JKDOM.SVG/180px-JKDOM.SVG.png" width="180" height="240" class="thumbimage" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/e/e4/JKDOM.SVG/270px-JKDOM.SVG.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/e/e4/JKDOM.SVG/360px-JKDOM.SVG.png 2x" data-file-width="602" data-file-height="802" /></a>/n<div class="thumbcaption">/n<div class="magnify"><a href="/wiki/File:JKDOM.SVG" class="internal" title="Enlarge"></a></div>/n<a href="/wiki/Document_Object_Model" title="Document Object Model">Document Object Model</a>, a standard <a href="/wiki/Object_model" title="Object model">object model</a> for representing <a href="/wiki/HTML" title="HTML">HTML</a> or <a href="/wiki/XML" title="XML">XML</a></div>/n</div>/n</div>/n<h3><span class="mw-headline" id="Object_model">Object model</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=23" title="Edit section: Object model">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Object_model" title="Object model">Object model</a></div>/n<p>An object model in computer science is a collection of objects or classes through which a program can examine and manipulate some specific parts of its world. In other words, the object-oriented interface to some service or system. Such an interface is said to be the <i>object model of</i> the represented service or system. For example, the <a href="/wiki/Document_Object_Model" title="Document Object Model">Document Object Model (DOM)</a> <a rel="nofollow" class="external autonumber" href="http://www.w3.org/DOM/">[1]</a> is a collection of objects that represent a <a href="/wiki/Web_page" title="Web page">page</a> in a <a href="/wiki/Web_browser" title="Web browser">web browser</a>, used by <a href="/wiki/Scripting_language" title="Scripting language">script</a> programs to examine and dynamically change the page. There is a <a href="/wiki/Microsoft_Excel" title="Microsoft Excel">Microsoft Excel</a> object model<sup id="cite_ref-25" class="reference"><a href="#cite_note-25">[25]</a></sup> for controlling Microsoft Excel from another program, and the <a href="/wiki/ASCOM_(standard)" title="ASCOM (standard)">ASCOM</a> Telescope Driver<sup id="cite_ref-26" class="reference"><a href="#cite_note-26">[26]</a></sup> is an object model for controlling an astronomical telescope.</p>/n<p>In <a href="/wiki/Computing" title="Computing">computing</a> the term <i>object model</i> has a distinct second meaning of the general properties of <a href="/wiki/Object_(computer_science)" title="Object (computer science)">objects</a> in a specific computer <a href="/wiki/Programming_language" title="Programming language">programming language</a>, technology, notation or <a href="/wiki/Methodology" title="Methodology">methodology</a> that uses them. For example, the <i><a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a> object model</i>, the <i><a href="/wiki/Component_Object_Model" title="Component Object Model">COM</a> object model</i>, or <i>the object model of <a href="/wiki/Object-modeling_technique" title="Object-modeling technique">OMT</a></i>. Such object models are usually defined using concepts such as <a href="/wiki/Class_(computer_science)" class="mw-redirect" title="Class (computer science)">class</a>, <a href="/wiki/Message_(computer_science)" class="mw-redirect" title="Message (computer science)">message</a>, <a href="/wiki/Inheritance_(computer_science)" class="mw-redirect" title="Inheritance (computer science)">inheritance</a>, <a href="/wiki/Polymorphism_in_object-oriented_programming" class="mw-redirect" title="Polymorphism in object-oriented programming">polymorphism</a>, and <a href="/wiki/Information_hiding" title="Information hiding">encapsulation</a>. There is an extensive literature on formalized object models as a subset of the <a href="/wiki/Formal_semantics_of_programming_languages" class="mw-redirect" title="Formal semantics of programming languages">formal semantics of programming languages</a>.</p>/n<h3><span class="mw-headline" id="Object-Role_Model">Object-Role Model</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=24" title="Edit section: Object-Role Model">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Object-Role_Modeling" class="mw-redirect" title="Object-Role Modeling">Object-Role Modeling</a></div>/n<div class="thumb tright">/n<div class="thumbinner" style="width:322px;"><a href="/wiki/File:Schema_for_Geologic_Surface.svg" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/commons/thumb/c/c4/Schema_for_Geologic_Surface.svg/320px-Schema_for_Geologic_Surface.svg.png" width="320" height="233" class="thumbimage" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/c/c4/Schema_for_Geologic_Surface.svg/480px-Schema_for_Geologic_Surface.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/c/c4/Schema_for_Geologic_Surface.svg/640px-Schema_for_Geologic_Surface.svg.png 2x" data-file-width="535" data-file-height="390" /></a>/n<div class="thumbcaption">/n<div class="magnify"><a href="/wiki/File:Schema_for_Geologic_Surface.svg" class="internal" title="Enlarge"></a></div>/nExample of the application of Object-Role Modeling in a "Schema for Geologic Surface", Stephen M. Richard (1999)<sup id="cite_ref-SMR99_27-0" class="reference"><a href="#cite_note-SMR99-27">[27]</a></sup></div>/n</div>/n</div>/n<p>Object-Role Modeling (ORM) is a method for <a href="/wiki/Conceptual_modeling" class="mw-redirect" title="Conceptual modeling">conceptual modeling</a>, and can be used as a tool for information and rules analysis.<sup id="cite_ref-28" class="reference"><a href="#cite_note-28">[28]</a></sup></p>/n<p>Object-Role Modeling is a fact-oriented method for performing <a href="/wiki/Systems_analysis" title="Systems analysis">systems analysis</a> at the conceptual level. The quality of a database application depends critically on its design. To help ensure correctness, clarity, adaptability and productivity, information systems are best specified first at the conceptual level, using concepts and language that people can readily understand.</p>/n<p>The conceptual design may include data, process and behavioral perspectives, and the actual DBMS used to implement the design might be based on one of many logical data models (relational, hierarchic, network, object-oriented etc.).<sup id="cite_ref-msd_29-0" class="reference"><a href="#cite_note-msd-29">[29]</a></sup></p>/n<h3><span class="mw-headline" id="Unified_Modeling_Language_models">Unified Modeling Language models</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Data_model&amp;action=edit&amp;section=25" title="Edit section: Unified Modeling Language models">edit</a><span class="mw-editsection-bracket">]</span></span></h3>/n<div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Unified_Modeling_Language" title="Unified Modeling Language">Unified Modeling Language</a></div>/n<p>The Unified Modeling Language (UML) is a standardized general-purpose <a href="/wiki/Modeling_language" title="Modeling language">modeling language</a> in the field of <a href="/wiki/Software_engineering" title="Software engineering">software engineering</a>. It is a <a href="/wiki/Visual_language" title="Visual language">graphical language</a> for visualizing, specifying, constructing, and documenting the <a href="/wiki/Artifact_(software_development)" title="Artifact (software development)">artifacts</a> of a software-intensive system. The Unified Modeling Language offers a standard way to write a system's blueprints, including:<sup id="cite_ref-OMG00_30-0" class="reference"><a href="#cite_note-OMG00-30">[30]</a></sup></p>/n<ul>/n<li>Conceptual things such as <a href="/wiki/Business_process" title="Business process">business processes</a> and system functions</li>/n<li>Concrete things such as <a href="/wiki/Programming_language" title="Programming language">programming language</a> statements, database schemas, and</li>/n<li>Reusable <a href="/wiki/Component-based_software_engineering" title="Component-based software engineering">software components</a>.</li>/n</ul>/n<p>UML offers a mix of <a href="/wiki/Functional_model" class="mw-redirect" title="Functional model">functional models</a>, data models, and <a href="/wiki/Database_model" title="Database model">database models</a>.</p>/n<h2><span class="mw-headline"
